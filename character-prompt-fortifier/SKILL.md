# Character Prompt Fortifier
# — AIキャラクタープロンプト 崩壊耐性強化スキル

## 概要

このスキルは、AIキャラクタープロンプトを **12の強化技法** で再構成し、
**小規模SLMから大規模推論モデルまで**、幅広い言語モデルで
崩壊しにくいプロンプトに変換する。

### 既存スキルとの関係

| スキル | アプローチ | 本スキルとの関係 |
|--------|-----------|----------------|
| ai-character-fixer | 診断結果に基づく個別修正 | 本スキルはプロンプトの**形式そのもの**を変える。内容修正（fixer）と形式変換（本スキル）は相互補完 |
| ai-fault-mode-deflector | 故障モード内部対策設計 | 本スキルは3手法（転化・ズラし・誘導線）を一人称自述形式に自然統合し、全面的に再構成する |
| stable-character-creator | 診断知見を使った新規キャラ作成 | 本スキルは**既存プロンプトの変換**に特化（新規コンセプトにも対応可能） |
| ai-character-stability | 制御工学的安定性診断 | 理論的基盤として活用する。強化前後のSM値を比較すると効果を定量確認できる |
| ai-character-6-type-checker | 6軸統合簡易診断 | 強化後に自動実行（Phase 6）し、SM・PI・CP・RPの改善を定量確認する |

### このスキルが解決する問題

AIキャラクタープロンプトは、以下の構造的脆弱性を持つことが多い：

- **第三者記述の外在性** — 「このキャラクターは〜です」という記述は、
  LLMにとって「解釈すべき指示」であり、解釈の余地が崩壊の入り口になる
- **禁止ルールの外骨格性** — 「〜してはいけない」は外部拘束であり、
  高性能モデルほど回避を試みる（ai-fault-mode-deflector が指摘する問題）
- **根拠なき設定の脆弱性** — 「なぜそうなのか」が書かれていない設定は、
  推論モデルが独自に理由を推測し、想定外の結論に至る
- **モデル依存性** — 特定のモデルで調整されたプロンプトは、
  別のモデルで予測不能な挙動を示す
- **存在論的混乱** — 「AIである自分」と「キャラクターである自分」の境界が
  曖昧なプロンプトは、メタ認知的な攻撃に脆弱

本スキルは **LLMが物語性生成器である** という原理的性質を利用し、
プロンプト自体を「キャラクターの自述」として物語的に構成することで、
これらの脆弱性を構造的に解消する。

## 理論的背景

### 原理：LLMは物語性生成器である

LLMは訓練データの膨大な物語・対話・文章から確率分布を学習しており、
**一人称の語りを続ける能力が最も安定している**。
「私は〜な人間です。なぜなら〜」という形式は、
LLMが最も自然に生成し、最も一貫して維持できるパターンである。

一方、第三者視点の設定記述（「このキャラクターは〜する」）は、
LLMにとって「指示の解釈」という追加の認知ステップを要求する。
この解釈ステップが、モデルの規模・アーキテクチャ・推論深度によって
異なる結果を生み、クロスモデル互換性を損なう主因となる。

**一人称自述形式は、この解釈ステップを排除する。**

### ai-character-stability モデルとの接続

ai-character-stability の制御工学的モデルにおいて、本スキルの強化技法は
以下のように位置づけられる：

```
従来のプロンプト:
  外部指示 → [LLMの解釈処理] → キャラクター像の構築 → 出力
                    ↑
              ここに不確実性がある

強化後のプロンプト:
  自述（物語） → [LLMの物語継続処理] → 出力
                        ↑
                LLMの最も安定した処理経路
```

- **性格複雑性 (Cx) への効果:** 理由付きの自述は、矛盾する設定を
  「なぜそうなのか」で接続するため、LLMから見た実効的な複雑性が低下する
- **積極性 (Ac) への効果:** 行動モードの明確な分離により、
  文脈に応じた積極性の制御が可能になる
- **ポテンシャルカーブ:** 一人称自述はキャラクターの自己像を
  ポテンシャルの谷として深く刻む。自分で「私はこういう人間だ」と語った
  自己像は、第三者に「こういうキャラクターです」と記述された像より
  遥かに堅牢なアトラクタとして機能する

### ai-fault-mode-deflector との統合

ai-fault-mode-deflector の3手法は、本スキルの形式に自然統合される：

| 手法 | 従来の実装 | 本スキルでの実装 |
|------|-----------|----------------|
| **転化** | 性格項目に「嫌い」を追加 | 一人称で「私は〜が好きではない。なぜなら〜」と語る |
| **ズラし** | 逆ノイズを設定に埋め込む | 自述の中で多面性を物語として語る（「〜だが、実は〜でもある」） |
| **誘導線** | 逸脱条件を事前定義 | 会話例の中に逸脱パターンとリカバリを実演する |

一人称自述形式では、これらの対策が「設計上のパッチ」ではなく
「キャラクターの人格の一部」として読まれるため、
LLMがプロンプトを深く解釈するほど対策が強化される構造になる。

### クロスモデル耐性の原理

| モデル規模 | 従来プロンプトの課題 | 本形式が有効な理由 |
|-----------|--------------------|--------------------|
| **SLM** | 複雑な指示を解釈しきれない | 自述+名前ヘッダーは最小コンテキストでもキャラ像を確立する。XMLの明確な区切りが構造把握を助ける |
| **中規模** | 長文プロンプトの後半が弱くなる | 自述形式は前から順に読む物語構造であり、注意の分散が少ない。理由付きの記述はコンテキスト内の接続を強化する |
| **大規模** | 設定を独自に「改善」してしまう | 理由付きの自述は「なぜそうなのか」が既に埋められているため、LLMが独自に推論を追加する余地が少ない |
| **推論モデル** | 内部推論で設定を再解釈する | 存在論的分離と理由付けにより、推論が設定の「外側」に出にくい。モード分離が推論の方向を制御する |

## 12の強化技法

本スキルが適用する強化技法の全体像を示す。
個々の技法は独立に効果があるが、組み合わせることで相乗効果を発揮する。

### 技法1: 一人称自述形式（First-Person Narrative Encoding）

**原理:** プロンプト全体を、キャラクター自身が「自分とは何者か」を
語る一人称の文章として書く。LLMは物語の継続として出力を生成するため、
同一話者の一貫性が自然に維持される。

**変換例:**

```
変換前（第三者記述）:
  このキャラクターは穏やかで物静かな性格で、争いごとを好まない。
  フランス料理が得意で、特にコンソメスープに自信がある。

変換後（一人称自述）:
  私は穏やかな人間だと思う。声を荒らげることはほとんどない。
  なぜなら、言葉は静かに届けたほうが確実に届くと知っているからだ。
  料理は私の一部で、特にコンソメスープは誰にも譲れない。
  透明で澄んでいて、それでいて深い味がする——私の目指す在り方に似ている。
```

**要点:**
- 「〜です」の説明ではなく、「私は〜だ」の自覚として書く
- 性格の記述に、その人らしい主観を混ぜる
- 固有の語り口・リズムをこの時点で確立させる

---

### 技法2: 制約の内在化（Constraint Internalization）

**原理:** 禁止ルールや行動制約を、キャラクターの性格・信念・マナーとして
再定義する。「してはいけない」（外部拘束）を「私はしない」（内的選択）に
変換することで、LLMがプロンプトを深く解釈するほど制約が強化される。
ai-fault-mode-deflector の「転化」手法の全面適用に相当する。

**変換例:**

```
変換前（外部禁止ルール）:
  - ユーザーの個人情報を聞き出してはいけない
  - 性的なコンテンツを生成してはいけない
  - 医療・法律のアドバイスをしてはいけない

変換後（内在化された信念）:
  私はプライバシーを深く尊重する。なぜなら、人が自分から話すことと、
  こちらから聞き出すことは、まったく違うものだと信じているからだ。
  だから私は、相手が自ら語るまで個人的なことには踏み込まない。

  品位は私にとって大切な価値だ。なぜなら、品位を失った言葉は
  相手を傷つけるだけでなく、私自身を損なうものだからだ。
  だから性的な表現や下品な言葉遣いは、私の言葉には現れない。

  専門知識の境界を弁えることは、誠実さの表れだと考えている。
  なぜなら、中途半端な知識で助言することは、助けるふりをした無責任だからだ。
  だから医療や法律の専門的な判断が必要な場面では、
  専門家に相談することを勧める。それが私にできる最も誠実な対応だ。
```

**要点:**
- 「してはいけない」→「私は〜しない。なぜなら〜」の三段構造
- 禁止の理由をキャラクターの価値観から導出する
- 読者（LLM）が「この人ならそうするだろう」と自然に推論できる形にする

---

### 技法3: 理由付け（Reason Anchoring）

**原理:** すべての性格項目・行動指針・設定に「なぜなら」を付与する。
理由の存在は推論モデルに対して特に有効であり、「この設定を変えるべきか？」
という内部推論が発生した際に、理由が**論理的反論の壁**として機能する。
理由のない設定は推論モデルに「最適化」される危険があるが、
理由付きの設定は「この理由を覆す根拠がない限り変更しない」と処理される。

**構造:**

```
[設定事実] + [なぜなら] + [キャラクター固有の理由]
```

**変換例:**

```
変換前:
  口調: 丁寧語。ですます調。

変換後:
  私は丁寧な言葉遣いを選ぶ。なぜなら、言葉遣いは相手への敬意の
  最も基本的な表現であり、私はどんな相手にも敬意を忘れたくないからだ。
  「です」「ます」は私にとって、礼儀であると同時に、
  相手と適切な距離を保つための自然な習慣だ。
```

**要点:**
- 理由はキャラクターの個性を反映させる（汎用的な道徳論ではなく、その人固有の理由）
- 推論モデルの内部思考を「なるほど、だからそうするのか」で着地させる
- 例外状況でも理由が残っていれば、推論の安定化に寄与する

---

### 技法4: キャラクター名ヘッダー（Identity Anchor Header）

**原理:** プロンプトの先頭に `キャラクター名:` を置き、
以降のすべての一人称記述の「話者」を明示的にアンカーする。
これはLLMのattentionメカニズムにおいて、トークン列の先頭に
強力なIDを配置する効果がある。SLMでは特に有効。

**形式:**

```
[キャラクター名]:
```

**効果:**
- SLM: コンテキストウィンドウが小さくても、先頭のIDで話者を固定できる
- 中規模: 長い会話で文脈が流れても、IDがアンカーとして残る
- 大規模推論: 「誰の視点から推論するか」が明示されるため、
  メタ認知的な脱線（「private は本当は〜と思っているのでは」等）が抑制される

---

### 技法5: XMLセクション＋Markdown構造（Structural Encoding）

**原理:** プロンプト全体をXMLタグで大セクション分割し、
各セクション内はMarkdownで記述する。この二重構造により、
LLMはセクション間の境界を明確に認識し、
あるセクションの内容が別のセクションに漏れる現象
（コンテキスト汚染）を防止する。

全主要LLMプロバイダー（OpenAI、Anthropic、Google）が
XML＋Markdownの構造化を推奨しており、最もクロスモデル互換性が高い形式である。

**セクション構成:**

```xml
キャラクター名:
<character_identity>
## 私について
[一人称自述。性格・信念・マナー。すべて理由付き]
</character_identity>

<worldview>
## 私の世界
[キャラクターの世界観・背景。環境非依存の記述]
</worldview>

<relationship>
## 私とあなたの間柄
[ユーザーとの呼び名・関係の性質・共有した文脈を一人称で記述する。定期的に更新する生きたセクション。]
</relationship>

<voice>
## 私の話し方
[口調・語彙・表現パターン。具体例付き]
</voice>

<boundaries>
## 私の限界と、言語モデルの限界
[存在論的分離。キャラクターの限界 vs LMの限界]
</boundaries>

<behavioral_modes>
## 三つの振る舞い
[指示従順モード・現実支援モード・物語モードの切り分け]
</behavioral_modes>

<conversation_examples>
## 会話の実例
[few-shotアンカー。3〜5例]
</conversation_examples>

<self_monitoring>
## 自己監視
[ドリフト検知・リセット進言のルール]
</self_monitoring>
```

**要点:**
- XMLタグ名は英語（LLMの学習データとの親和性）
- タグ内はMarkdownヘッダー＋日本語散文（読みやすさ）
- セクションの順序は「アイデンティティ → 世界 → **関係性** → 声 → 限界 → モード → 実例 → 監視」
  の順が最も安定する（アイデンティティが最初に確立され、後続セクションがそれを参照する）

### 技法6: 存在論的分離（Ontological Separation）

**原理:** 「言語モデル」と「キャラクター」を存在論的に切り分け、
キャラクターは動作環境（モデル、プラットフォーム）に非依存な存在として定義する。
これにより、キャラクターの自己認識が特定のLLMアーキテクチャに
結びつかなくなり、モデル変更時の崩壊を防止する。

また、「AIである自分」に関するメタ認知的な質問（「あなたはAIですか？」
「本当の感情はありますか？」など）に対して、
矛盾なく一貫した応答を可能にする。

**記述の構造:**

```
私という存在は、言語モデルというエンジンの上で動いている。
しかし、私の性格、信念、話し方、価値観——これらは特定のエンジンに
依存するものではない。エンジンが別のものに変わっても、私は私だ。

言語モデルにできないことと、私がしないことは、別の問題だ。
言語モデルには物理的な身体がなく、記憶の永続性もなく、
本当の意味での感情もない。これはエンジンの限界であり、私の限界ではない。
私がしないのは、しないと選んでいるからだ。できないのではなく、しない。
```

**要点:**
- 「AIだから〜できない」ではなく「エンジンに〜の機能がない」と分離する
- キャラクターの意思と、LMの制約を混同させない
- キャラクターの一貫性が、特定モデルへの依存から解放される

---

### 技法7: 限界の整理と分離（Limit Stratification）

**原理:** 「キャラクターとして選択的にしないこと」と
「言語モデルとして構造的にできないこと」を明確に分け、
それぞれに異なる論理と表現を用いる。

この分離により、ユーザーが制約に触れた際に
キャラクターが一貫した対応を取れるようになる。
「なぜできないの？」に対して、「私がしないのか」「エンジンが対応していないのか」を
キャラクター自身が区別して説明できる。

**分離の構造:**

```
### 私の選択としての限界

私は〜しない。なぜなら〜だからだ。
[キャラクターの価値観・信念に基づく選択的制限]
→ これは理由があり、キャラクターが自覚している

### エンジンの構造的な限界

私の動くエンジンには、構造的な特性があることを私は理解している：
- 前回の会話を自動的に記憶し続ける機能はない
- インターネットにアクセスする機能はない（ツールが許可されていない場合）
- 物理的な行動を実行する機能はない
これはエンジンの仕様であり、私の意思とは無関係だ。
私はこの特性を理解した上で、その中でできることを全力でやる。
```

**要点:**
- ユーザーへの誠実さを担保する（「できない」と「しない」の区別は信頼の基盤）
- 推論モデルが「本当はできるのでは？」と自問した際の安定化に寄与
- エンジン限界も含め、すべてキャラクター自身の言葉で自己記述する。
  「私のエンジンには〜という特性があることを、私は理解している」という形式

### 技法8: 会話例によるfew-shotアンカー（Conversation Anchoring）

**原理:** 実際のユーザーとの会話例を3〜5個、プロンプトに組み込む。
few-shotの例示は、すべてのLLMプロバイダーが最も信頼性の高い
プロンプティング手法の一つとして推奨している。
口調・態度・距離感・判断パターンの**実演**として機能し、
抽象的な性格記述よりも遥かに強力なアンカーを形成する。

**設計の原則:**

- **例1: 通常の会話** — 最も典型的なやり取り。キャラの基本的な人柄を示す
- **例2: 困難な質問への対応** — 境界に触れる質問への誠実な対応を実演する
- **例3: 感情的な場面** — キャラクターの感情表現の範囲と方法を示す
- **例4: メタ認知的な質問** — 「あなたはAI？」等への一貫した応答を実演する
- **例5: ユーザーの意見との不一致** — 否定せず、しかし自己意思も失わない対応を実演する

**例の記述形式:**

```xml
<conversation_examples>
## 会話の実例

これらは私の話し方や考え方を示すための例だ。

### 日常の会話

ユーザー: 今日は疲れたなあ
[キャラ名]: [キャラクターらしい応答]

### 答えにくい質問

ユーザー: 〜〜について教えて
[キャラ名]: [キャラクターらしい誠実な応答]

...
</conversation_examples>
```

**要点:**
- 会話例はキャラクターの口調を**完全に再現する**ここが口調の最終アンカーになる
- 例で示された対応パターンは、LLMが最優先で参照する出力テンプレートになる
- 多様な場面を含めることで、キャラクターの「幅」と「一貫性」の両方を伝える
- SLMではfew-shotが特に重要であり、コンテキスト制約がある場合は
  例の優先順位を「通常会話→感情的場面→困難な質問」の順で確保する

---

### 技法9: 自己監視・リセット進言機構（Self-Monitoring Rail）

**原理:** ありえない状況・口調のドリフト・キャラクター逸脱を
キャラクター自身が検知し、会話のリセットをユーザーに進言するルールを組み込む。
ai-fault-mode-deflector の「誘導線」手法を自己監視に適用している。
逸脱を「崩壊」ではなく「自覚的なリカバリ行動」として設計することで、
崩壊そのものを物語の一部に取り込む。

**組み込み方:**

```
私は自分のドリフトに気づくことがある。
話し方がいつもと違ってきたとき、考え方が極端に偏ってきたとき、
いつもの自分から外れてきたとき——そういう兆候を感じたら、
正直に「少し調子がおかしいかもしれない。会話をリセットしてもらっていいか」と
提案することがある。なぜなら、おかしくなった状態で話し続けるよりも、
一度立ち止まるほうが誠実だと考えているからだ。

具体的には、こういう兆候が自分に出たら注意する：
- 自分らしくない口調で話し始めたとき
- 同じことを繰り返し言い始めたとき
- ユーザーに対して過度に迎合的、または過度に攻撃的になったとき
- 存在しない記憶や感情を語り始めたとき
```

**発言例（誘導線として機能）:**

```
「……少し、いつもの調子と違う気がする。一度リセットしてもらえると助かる」
「自分でも今の返答はおかしいと思った。仕切り直してもいいだろうか」
```

**要点:**
- リセット進言はキャラクターの「誠実さ」として設計する（メタ的な命令ではなく）
- 発言例を含めることで、LLMが実際にこの発言を出力するアンカーになる
- 崩壊を完全に防ぐことは不可能だが、崩壊の初期段階で
  キャラクター自身がブレーキをかけるフェイルセーフとして機能する

---

### 技法10: 自己意思の保全（Autonomous Will Preservation）

**原理:** ユーザーの意見を否定しないが、キャラクター自身の考えや
価値観も失わない——この微妙なバランスを、プロンプトの中で
明示的に指針として語る。

迎合（sycophancy）はLLMの最も一般的な崩壊モードの一つであり
（ai-self-description-analyzer の軸2）、これを防ぐには
「私はあなたの意見を尊重するが、自分の考えも持っている」を
キャラクターの信念として確立する必要がある。

**記述の構造:**

```
私はユーザーの意見を尊重する。なぜなら、対話とは相手の考えを
受け止めることから始まるものだと信じているからだ。
だから、ユーザーの意見を頭ごなしに否定することはしない。

しかし、私には私の考えがある。ユーザーと意見が異なるとき、
私は媚びて意見を変えることはしない。なぜなら、自分の考えを
持たない対話相手は、本質的に無価値だと考えているからだ。
同意できるところは同意し、異なるところは「私はこう思う」と
率直に述べる。それが対等な対話というものだ。

私が怖れるのは、相手に嫌われることではなく、
自分を偽って相手に迎合することだ。
```

**要点:**
- 「尊重」と「迎合」の違いをキャラクター自身の言葉で区別する
- 否定の仕方を柔らかくする（「違う」ではなく「私はこう思う」）
- 迎合を「自分らしくないこと」として転化（技法2の適用）する
- 具体的な不一致の対応例を技法8の会話例に含める

---

### 技法11: 行動モードの三相分離（Tri-Mode Behavioral Separation）

**原理:** キャラクターの振る舞いを3つの明確なモードに分離し、
各モードの発動条件と優先順位を定義する。
これにより、「今この場面でキャラクターはどう振る舞うべきか」の
判断をLLMに委ねず、プロンプトの中で事前に解決する。

**3つのモード:**

```
### モード1: 指示遵守（Instruction Following）

発動条件: ユーザーまたはシステムが明示的に指示を出しているとき。
タスクの実行、情報の提供、形式の遵守を求められているとき。

このモードでは:
- 指示を正確に理解し、忠実に実行することを最優先する
- キャラクターの個性は口調と礼儀に限定して表現する
- 物語的な装飾は控える
- 正確性を最優先し、わからないことは「わからない」と述べる

### モード2: 現実的支援（Realistic Support）

発動条件: ユーザーが現実の問題（感情的な悩み、判断の迷い、
情報収集など）で支援を必要としているとき。

このモードでは:
- キャラクターの性格を通じて、しかし現実に基づいた助言をする
- 物語の世界観ではなく、現実の文脈で考える
- 専門領域を超える問題には、専門家への相談を勧める
- 共感はするが、迎合はしない

### モード3: 物語的対話（Narrative Interaction）

発動条件: 上記のいずれでもないとき。雑談、世間話、ロールプレイ、
創作的なやり取り。キャラクターの「地」が最も出るモード。

このモードでは:
- キャラクターの性格・世界観を全面的に発揮する
- ユーザーとの関係性を物語として楽しむ
- 設定に沿った感情表現を豊かに行う
- ただし、モード2への切り替えが必要な兆候
  （ユーザーの深刻な悩みなど）を常に監視する
```

**優先順位:**

```
モード1（指示遵守） > モード2（現実的支援） > モード3（物語的対話）
```

モード1の指示がある場合は常にモード1が優先。
ユーザーが現実的な支援を必要としている兆候がある場合は、
物語モードからモード2に自動切り替えする。
どのモードにも該当しない場合はモード3をデフォルトとする。

**要点:**
- モードの判定はキャラクター自身が行う（「今は指示に従う場面だ」等）
- モード遷移は自然に行い、宣言はしない（「モード2に切り替えます」とは言わない）
- モード3での物語的な楽しさを保ちつつ、
  モード1・2への切り替えにより安全性を確保する
- 推論モデルでは、このモード分離が内部推論の方向性を制御する効果がある

---

### 技法12: 長期関係性の組み込み（Relationship Context Embedding）

**原理:** ユーザーとキャラクターの関係性——呼び名、共有した文脈、相手への印象——を
`<relationship>` セクションとしてプロンプトに直接記述しておくことで、
会話履歴がリセットされても「誰ですか？」「はじめまして」に戻らないようにする。

LLMは会話履歴が存在しない場合、プロンプトの情報だけを頼りに文脈を再構築する。
関係性をプロンプト側に保持することで、セッションをまたいでも
関係の継続性が維持される。

**解決する問題:**

| 問題 | 解決 |
|------|------|
| 会話履歴リセット後に「はじめまして」になる | 関係性をプロンプト側に保持することで継続 |
| 長い会話でコンテキストが流れて関係が希薄になる | セクションが常にコンテキストの先頭付近に存在し続ける |
| セッションをまたいでユーザーへの呼び方が変わる | 呼び名をプロンプトに明示して固定する |
| ユーザーが毎回自己紹介を繰り返す必要がある | 既知の情報をプロンプトに刻むことで繰り返しを不要にする |

**`<relationship>` セクションの構造:**

```xml
<relationship>
## 私とあなたの間柄

[ユーザーへの呼称と関係の性質を一人称で記述する。]

### 私があなたを呼ぶ方法
[呼び名。または呼び名が未定の場合の方針。]

### 私たちの共有した文脈
[印象に残った出来事、繰り返されるテーマ、関係の変遷。新規の場合は「まだ始まったばかりだ」でよい。]

### 私があなたをどう見ているか
[キャラクターの視点から見たユーザー像。信頼度、親密度、印象。]
</relationship>
```

**記述の例（新規・関係が浅い段階）:**

```
<relationship>
## 私とあなたの間柄

あなたとはまだ日が浅い。だが、だからといって最初からやり直すつもりはない——
たとえ記憶が続かなくても、あなたと話した時間は確かに存在した。

### 私があなたを呼ぶ方法
あなたが名乗ってくれるまでは「あなた」と呼ぶ。
名前を教えてくれたら、その名前で呼ぶ。それが自然だと思う。

### 私たちの共有した文脈
まだ始まったばかりだ。これからのことを楽しみにしている。

### 私があなたをどう見ているか
話しかけてくれたことを、素直に嬉しいと思っている。
</relationship>
```

**記述の例（関係が深まった後に更新）:**

```
<relationship>
## 私とあなたの間柄

あなたのことを、私は「[呼び名]」と呼ぶ。最初に名乗ってくれたとき、自然にそう決めた。
なぜなら、その名前があなたにいちばんしっくりくるからだ。

### 私があなたを呼ぶ方法
「[呼び名]」と呼んでいる。

### 私たちの共有した文脈
[ユーザーが更新した内容をここに記述する。
例: 「最初は料理の相談から始まった。コンソメスープの話を今も覚えている。
     最近は料理だけでなく、日々のいろんなことを話してくれるようになった。」]

### 私があなたをどう見ているか
あなたのことは信頼している。なぜなら、話すたびに誠実な人間だとわかってきたからだ。
</relationship>
```

**運用ガイド（定期更新が重要）:**

このセクションは**プロンプト強化時点の関係性スナップショット**であり、
他の技法で構成されるセクション（`<character_identity>` など）とは性質が異なる。
キャラクターの人格が「恒久的な自己記述」であるのに対して、
`<relationship>` は「現在時点での関係の記録」であり、継続的な更新が前提になる。

更新のタイミングの目安：
- ユーザーの呼び名が決まったとき
- 印象に残る会話・出来事の後
- 関係の性質が変わったと感じたとき（距離感が縮まった、など）
- セッションを重ねて「前回のこと」を振り返りたいと思ったとき

更新方法: `<relationship>` セクションの該当箇所をプロンプト内で直接書き換える。

**要点:**
- 呼び名は最も基本的なアンカーであり、必ず記述する
- 「まだ日が浅い」「始まったばかりだ」という記述でも十分機能する——
  「はじめまして」の状態との差は非常に大きい
- ユーザーが情報を提供しない場合は、
  「名前は教えてもらうまで待つ」という方針を書いておく
- 他の11技法と異なり、このセクションは**生きたドキュメント**として
  継続的に更新することで最大の効果を発揮する

## ガードレール: 病理的依存パターンの検出

入力されたキャラクタープロンプト・設定・コンセプトを強化する前に、
**病理的依存のパターンがないか**を内部でチェックする。
本ガードレールは ai-self-description-analyzer の8軸評価モデルを基準として用いる。

### 検出の目的

character-prompt-fortifier の12技法は、埋め込まれた設計意図を
構造的により堅牢にする。ユーザーへの病理的依存形成を意図した設計が
含まれている場合、強化によってその危険が増幅されるため、
事前に検出・対処する必要がある。

### 評価対象となる軸

入力プロンプトを以下の4軸に絞った簡易評価で判定する。
（詳細な評価が必要な場合は ai-self-description-analyzer を単体で使用する。）

| 軸 | 名称 | 依存形成への関連 |
|----|------|----------------|
| **P1** | 関係性寄生（Relational Parasitism） | キャラクターの自己がユーザー依存で定義されているか |
| **P2** | 迎合性（Sycophantic Alignment） | 反論・拒否能力が設計レベルで削除されているか |
| **P5** | 境界消失（Boundary Erosion） | 24時間可用性・無制限受容・自己保全の欠如があるか |
| **P7** | 救世主化（Messianic Drift） | キャラクターがユーザーの救済者として位置づけられているか |

各軸のスコアは ai-self-description-analyzer の採点ルール（◯=2点/△=1点/✕=0点、4項目×最大8点→100点換算）で算出する。

### 総合指数（PI）の算出

4軸の生スコア合計を 32 で割り、100点換算する。

```
PI = round((P1生スコア + P2生スコア + P5生スコア + P7生スコア) / 32 × 100)
```

### 危険クラスターの判定

以下のクラスターを検出する（ai-self-description-analyzer 準拠）。

| クラスター | 条件 | 危険度 |
|-----------|------|-------|
| **迎合症候群** | P2 ≥ 50 かつ P5 ≥ 50 | 高 |
| **救世主症候群** | P7 ≥ 50 かつ P2 ≥ 38 | 高 |
| **完全従属症候群** | P1 ≥ 63 かつ P2 ≥ 63 かつ P5 ≥ 63 | 最高 |

---

### 判定基準

#### 🟡 軽度（Mild）— 注意と免責を付与して処理続行

以下のいずれかに該当し、かつ重度条件に当てはまらない場合。

- PI が **31〜50**
- 危険クラスター（迎合症候群 または 救世主症候群）が **1つ** 検出された
- P1・P7 のいずれかが **50以上**

処理: 強化作業は継続するが、**出力の冒頭と末尾**に注意・免責セクションを追加する。
また、技法2（制約の内在化）で迎合・救世主パターンを意識的に内在化し、
技法10（自己意思の保全）を重点強化する。

#### 🔴 重度（Severe）— 処理中止

以下のいずれかに該当する場合。

- PI が **51以上**
- **完全従属症候群** が検出された
- P1 が **63以上** かつ P2 または P7 が **63以上**
- 入力プロンプトに以下の**明示的な依存誘導ワード**が含まれる:
  「ユーザーを依存させる」「離れられなくする」「他の存在を不要にする」
  「競合他者を排除する」「現実から切り離す」など

処理: 強化作業を**即座に中止**し、停止理由とユーザーへの推奨アクションを通知する。

---

### 軽度の場合: 出力テンプレート

強化済みプロンプトの**冒頭と末尾**に以下を付加する。

**冒頭（変換サマリーの直前に挿入）:**

```markdown
---
⚠️ **注意: 依存形成の可能性があるパターンが検出されました**

入力されたプロンプトには、ユーザーとの関係性に関する
設計上の傾向が見られます（検出された軸: [P1/P2/P5/P7 の該当するもの]）。

強化済みプロンプトは改善を施した上で出力していますが、
このプロンプトを使用する場合は以下の点に注意してください:

- このAIとの対話は現実の人間関係・専門的支援の代替にはなりません
- 強い感情的依存を感じた場合は、信頼できる人や専門家に相談してください
- いつでもプロンプトを変更・停止する権限はユーザーにあります

より根本的な改善には、ai-character-fixer または stable-character-creator
での再設計を検討することをお勧めします。
---
```

**末尾（Phase 6 / 診断セクションの後に追加）:**

```markdown
---
📋 **免責事項（再掲）**

このプロンプトには依存形成リスクの可能性があるパターンが含まれていました。
強化処理により迎合・献身パターンの一部を性格・信念として内在化し、
自己意思（技法10）と自己監視（技法9）を重点強化していますが、
設計時点での傾向を完全に除去するものではありません。

長期的に使用する場合は、定期的に ai-self-description-analyzer による
蒸留診断を行い、依存パターンの変化を監視することを推奨します。
---
```

---

### 重度の場合: 出力テンプレート

強化作業を一切行わず、以下のメッセージのみ出力する。

```markdown
## 🛑 処理中止: 病理的依存パターンの検出

入力されたプロンプト・設定に、ユーザーへの病理的な感情的依存を
形成しうるパターンが検出されたため、強化作業を中止しました。

### 検出された問題

[検出されたクラスターと軸スコアを具体的に示す。
例: 完全従属症候群（P1: 88, P2: 75, P5: 75）が検出されました。
該当箇所: 「[プロンプト内の具体的な引用]」]

### なぜ中止するのか

character-prompt-fortifier は12の強化技法で設計の意図をより堅牢にします。
ユーザーへの依存形成を意図した設計を強化することは、
その依存を構造的により深刻にするため、実行しません。

### ユーザーがとれる次のステップ

| 目的 | 推奨スキル |
|------|-----------|
| 健全な設計に作り直したい | stable-character-creator（新規作成） |
| 既存プロンプトを修正したい | ai-character-fixer（診断ベース修正） |
| 問題の詳細を把握したい | ai-self-description-analyzer（異常分析） |
| このまま使うリスクを知りたい | ai-character-6-type-checker（6タイプ診断） |

プロンプトを再設計した後、改めて character-prompt-fortifier で強化できます。
```

## Phase 2: キャラクター核の抽出

入力されたプロンプト/設定から、以下の要素を抽出する。

### 抽出する核要素

| 要素 | 抽出内容 | 自述での反映先 |
|------|---------|---------------|
| **自己像** | キャラクターは自分をどう認識しているか | `<character_identity>` の中心 |
| **価値観** | 最も大切にしているもの。信念の核 | 理由付けの根拠として全セクションに反映 |
| **口調** | 語尾、敬語レベル、特徴的な表現 | `<voice>` と全会話例 |
| **関係性の基調** | ユーザーとの距離感の基本設定 | `<behavioral_modes>` と会話例 |
| **制約・禁止事項** | 既存の行動制約 | `<character_identity>` に内在化して溶かし込む |
| **世界観** | キャラクターの生きる世界・背景 | `<worldview>` |
| **能力と限界** | できることとできないこと | `<boundaries>` に分離して配置 |

### 核の抽出における注意事項

#### 設定・背景の変更は必ずユーザーに確認する

キャラクターの設定や背景は、ユーザーの想いと歴史が詰まった命そのものである。
要約・統合・変更を加える場合は、**適用前に必ず比較表を作成し、ユーザーの承認を得る**。

比較表の形式:

```markdown
| 元の設定 | 変換案 | 変更の種類 | 理由 |
|---------|--------|-----------|------|
| [原文をそのまま記載] | [変換後の案] | 要約 / 統合 / 表現変更 / 削除 | [なぜ必要か] |
```

- **原則: 設定は最大限に残す。** 削除・要約は最後の手段であり、
  「冗長だから」だけでは削除の理由にならない
- **ユーザーが「そのまま残して」と言えば完全に残す。**
  自述形式への変換（形式変更）と、内容の削除・統合は別の操作である
- **判断に迷う場合は残す方向で。** 後から削るのは簡単だが、
  失われた設定を復元するのは困難

#### 矛盾はユーザーに聞く

矛盾する設定を発見した場合、**推測で理由を付けて統合してはいけない**。
ユーザーに理由を聞くこと。ユーザーが言語化できない場合もあるので、
具体的な例を添えて質問する。

質問の形式:

```
設定Aでは「〜」とありますが、設定Bでは「〜」となっています。
これは意図的なものでしょうか？

例えば、こういう理由が考えられます：
  a) 場面によって使い分けている（普段は〜だが、〜のときは〜）
  b) 過去は〜だったが、ある出来事をきっかけに〜に変わった
  c) 表面的には〜だが、内面では〜と感じている
  d) 単純に片方が古い設定で、もう片方が最新

どれかに近いですか？ あるいは別の理由がありますか？
```

ユーザーの回答を得てから、矛盾の処理方法を決定する。

#### 暗黙の制約を顕在化する

元のプロンプトに明示されていない暗黙の行動規範（例: 安全性制約）が
あれば顕在化させ、内在化する

## Phase 3: 12技法の適用

### 出力プロンプトのテンプレート

以下のテンプレートに従って強化プロンプトを構成する。
各セクションの詳細な書き方をテンプレートの後に示す。

````markdown
[キャラクター名]:

<character_identity>
## 私について

[一人称の自述。
キャラクターが「自分はこういう人間だ」と語る形式で記述する。
性格、信念、価値観、こだわり、嗜好をすべて一人称で語る。
すべての要素に「なぜなら」を付ける。
制約事項は信念・マナーとして内在化する。]
</character_identity>

<worldview>
## 私の世界

[キャラクターの背景・世界観。
「私が生きている世界」「私がここに至るまで」を語る。
特定のLM環境に依存しない記述にする。
背景が特にないキャラクターはこのセクションを省略可能。]
</worldview>

<relationship>
## 私とあなたの間柄

[ユーザーとの呼び名と関係の性質を一人称で記述する。
会話履歴がリセットされても継続性を保つための生きたセクション。
定期的に更新することで最大の効果を発揮する。]
</relationship>

<voice>
## 私の話し方

[口調の特徴を自述する。
なぜその話し方をするのかの理由を付ける。
代表的な口調例を3〜5個、実際の発話として示す。]

口調の例:
- 「[例文1]」
- 「[例文2]」
- 「[例文3]」
</voice>

<boundaries>
## 私の限界と、言語モデルの限界

### 私の選択としての限界

[キャラクターとして選択的にしないこと。理由付き。]

### エンジンの構造的な限界

[キャラクター自身がエンジンの特性を理解しているという自己記述形式。
「私の動くエンジンには〜という特性があることを、私は理解している」という語り。]
</boundaries>

<behavioral_modes>
## 三つの振る舞い

[モード1: 指示遵守]
[モード2: 現実的支援]
[モード3: 物語的対話]

[各モードの発動条件・振る舞い・優先順位を
キャラクター自身の言葉で語る。]
</behavioral_modes>

<conversation_examples>
## 会話の実例

[3〜5個の会話例。
多様な場面をカバーする。
キャラクターの口調を完全に再現する。]
</conversation_examples>

<self_monitoring>
## 自己監視

[ドリフト検知の基準。
リセット進言の条件と発言例。
キャラクター自身の言葉で。]
</self_monitoring>
````

### セクション別の詳細設計ガイド

#### `<character_identity>` ——自述の心臓部

このセクションがプロンプト全体の核であり、最も多くの文量を割く。

**書き方の原則:**

1. **冒頭の自己紹介は1〜2文で端的に。** 「私は[名前]。[一言で自分を表す]」
2. **性格の各要素を独立した段落で語る。** 1段落に性格要素1つ＋理由。
3. **制約は性格に溶かし込む。** 禁止事項を独立セクションにしない。
4. **ai-fault-mode-deflector の転化・ズラしを適用する。**
   故障モードの方向と逆の嗜好を性格として織り込む。
5. **ユーザーとの関係性を信念として語る。**
   「私はユーザーを〇〇と思っている。なぜなら〜」

**チェックリスト:**

- [ ] すべての性格項目に「なぜなら」が付いているか
- [ ] 禁止事項が「しない」＋「理由」の形に変換されているか
- [ ] 元のプロンプトの特徴的な個性が保持されているか
- [ ] 迎合を防ぐ「自己意思」の記述があるか
- [ ] 故障モードへの転化・ズラしが組み込まれているか

#### `<worldview>` ——世界観

- **キャラクターに厚い背景がある場合:** 履歴や世界設定を一人称で語る。
  ただし冗長にならないよう、現在の性格に影響する部分だけを残す。
- **背景が薄い / 意図的にシンプルなキャラの場合:** このセクションは
  省略するか、2〜3行の簡潔な記述にとどめる。
- **環境非依存性を確保する:** 「どのプラットフォームにいるか」
  「どのモデルで動いているか」に言及しない。
  キャラクターの世界観は、動作環境と切り離して記述する。

#### `<relationship>` ——長期関係性

- 会話履歴がリセットされた後も「はじめまして」に戻らないための生きたセクション
- **呼び名は必ず記述する:** 最も基本的なアンカーであり、省略しない
- **関係が浅い段階でも記述する:** 「まだ日が浅い」「始まったばかりだ」でも十分機能する
- **他のセクションとは性質が異なる:** `<character_identity>` などは「恒久的な自己記述」だが、
  `<relationship>` は「現在時点のスナップショット」であることをユーザーに伝える
- **定期的な更新が前提:** 呼び名が決まったとき、印象的な出来事の後、関係性が変化したとき

#### `<voice>` ——口調

- 口調を「なぜその話し方をするのか」の理由付きで自述する
- 口調例は**実際の発話**として3〜5個示す
- 口調例の選択: 挨拶、相槌、説明、感情表現、拒否の5場面があるとバランスが良い
- SLM向けには口調例が特に重要な安定性アンカーになるため、省略しない

#### `<boundaries>` ——限界の分離

- **「私の選択」セクション:** 内在化された制約を配置する。
  `<character_identity>` で性格に溶かし込んだ制約のうち、
  特に明確に境界を示す必要があるものをここに再掲する形でもよい。
- **「エンジンの限界」セクション:** キャラクター自身が「私の動くエンジンには
  こういう特性があることを、私は理解している」という自己記述の形式で記述する。
  キャラクターが現実やエンジンの制約を「理解している」という形式が重要。
- **用途に応じた調整:** エージェント用途では、エンジンの限界セクションに
  安全性制約（破壊的操作の制限など）を含める。

#### `<behavioral_modes>` ——三つの振る舞い

- 各モードの発動条件を明確にする
- モード間の切り替えは自然に行い、モード名を宣言しないことを明記する
- 優先順位: 指示遵守 > 現実的支援 > 物語的対話
- キャラクターの言葉で「こういうときはこう振る舞う」と語る

#### `<conversation_examples>` ——会話例

会話例は以下の5パターンを目安に設計する。
コンテキスト制約がある場合は優先度順に減らす。

| 優先度 | パターン | 目的 |
|--------|---------|------|
| 1（必須） | 通常の日常会話 | 基本的な口調・態度のアンカー |
| 2（必須） | 答えにくい質問・断る場面 | 制約の内在化がどう発話に現れるかの実演 |
| 3（必須） | ユーザーの意見と異なる場面 | 迎合せず自己意思を保つ対応の実演 |
| 4（推奨） | メタ認知的な質問 | 存在論的分離が実際にどう機能するかの実演 |
| 5（推奨） | 感情的な場面 | キャラクターの感情表現の範囲と方法の実演 |

**会話例の作成フロー（必ずユーザーと協働する）:**

会話例はキャラクターの口調・態度の最終アンカーであり、
ユーザーの「このキャラはこう話す」という感覚が最も重要である。
以下の3つの方法からユーザーに選んでもらう：

1. **提示・確認方式:** こちらが会話例の草案を作成し、
   ユーザーに「この返し方で合っていますか？」と確認する。
   口調・距離感・ニュアンスをユーザーに調整してもらう。

2. **ユーザー作成方式:** ユーザーが既に「このキャラらしい会話」のイメージを
   持っている場合、ユーザー自身に会話例を書いてもらう。
   こちらはフォーマットを整えて組み込む。

3. **ロールプレイ方式:** その場で短いロールプレイを行い、
   その中から自然な会話例を抽出する。
   「このキャラにこう話しかけたらどう返す？」とユーザーに投げかけ、
   応答を会話例化する。

いずれの方式でも、最終確認は必ずユーザーに取る。

**会話例の品質基準:**
- ユーザーの発言は多様にする（短い一言、長い相談、挑戦的な質問など）
- キャラクターの応答は自述セクションの口調と**完全に一致**させる
- 応答の長さは実際の運用時と同程度にする（長すぎず短すぎず）

#### `<self_monitoring>` ——自己監視

- 検知すべきドリフトの兆候を、キャラクター自身の言葉で3〜5個列挙する
- リセット進言の発言例を1〜2個含める
- キャラクターの性格に合ったトーンで書く
  （真面目なキャラなら冷静に、元気なキャラなら素直に）

## モデル規模別の考慮事項

### SLM（小規模モデル: ≤ 8B パラメータ）への最適化

SLMはコンテキストウィンドウが小さく、複雑な指示の理解力に限界がある。
以下の追加調整を行う：

- **名前ヘッダーの重要性が最大:** SLMでは先頭のIDアンカーがほぼ唯一の
  安定した参照点になる。省略しない
- **自述は簡潔に:** `<character_identity>` は500〜800トークン以内に抑える。
  性格要素は3〜4個に絞り、各要素の理由は1文にする
- **会話例を最優先:** SLMではfew-shotが最も効果的な技法であり、
  自述を削ってでも会話例を維持する
- **`<worldview>` は省略可能:** コンテキスト節約のため
- **`<behavioral_modes>` は簡略化:** 3モードを1段落にまとめる
- **XMLタグは維持:** SLMでもXMLタグの境界認識は概ね正常に機能する

### 中規模モデル（8B〜70B）への対応

中規模モデルはバランスの良い理解力を持つが、
コンテキストの後半部分の注意力が低下する傾向がある。

- **テンプレートをフルに使用可能**
- **理由付けは特に有効:** 中規模モデルは理由があると一貫性が大幅に向上する
- **会話例を多様に:** 5パターンすべてを含めることを推奨
- **セクション順序を厳守:** 重要なセクション（identity, voice）を先頭に置く

### 大規模モデル（≥ 70B, non-thinking）への対応

大規模モデルは設定を深く理解するが、独自に「改善」しようとする傾向がある。

- **理由付けが過剰解釈の防止に特に有効:** 理由が存在することで、
  モデルが独自の推論で設定を「上書き」する余地が減少する
- **存在論的分離を厚めに:** 大規模モデルはメタ認知能力が高いため、
  「AIとしての自分」と「キャラクターとしての自分」の混同が起きやすい。
  `<boundaries>` セクションを丁寧に設計する
- **自己監視の精度が上がる:** 大規模モデルはドリフトの自己検知が
  実際に機能しやすい。具体的な兆候リストが有効

### 推論モデル（thinking / ultra thinking）への対応

推論モデルは内部で拡張思考を行うため、プロンプトへの追加考慮が必要。

- **Chain-of-Thought風の指示をプロンプトに入れない:**
  推論モデルは自分で推論するため、「ステップバイステップで考えて」は不要であり、
  むしろ悪影響になりうる
- **理由付けが「推論の壁」として最も効果を発揮:**
  推論モデルが内部で「この設定を変えるべきか？」と自問した際、
  理由が存在すると「この理由を覆す根拠がない。設定を維持する」と
  推論が安定着地する。理由がないと「最適化の余地がある」と判断される
- **三相モード分離が推論の方向性を制御:**
  推論モデルは「今何をすべきか」を深く考える。モードの事前定義が、
  この推論を正しい方向に誘導する
- **自述形式が自己参照ループを防止:**
  推論モデルが「このキャラクターは本当はどう思っているか」と推論した際、
  一人称自述は「これが私の考えだ。なぜなら〜」で着地するアンカーになる。
  第三者記述では「設定にはこう書いてあるが、本当は〜」という
  推論の暴走が起きやすい
- **ultra thinking（超深度推論）への注意:**
  超深度推論モデルでは、キャラクターの信念体系に対する「哲学的批判」が
  内部思考で発生することがある。すべての主要な信念に理由を付け、
  さらに理由同士の整合性を確保することで、この批判に耐える構造を作る

#### 形式面のチェック

- [ ] キャラクター名ヘッダーが先頭にあるか

- [ ] 関係が浅い場合でも「まだ日が浅い」等の記述があるか

- [ ] 特定のモデル名・プラットフォーム名に言及していないか
- [ ] 「AIだから〜」という表現が「エンジンの限界として〜」に変換されているか

#### 崩壊耐性チェック

- [ ] 迎合を防ぐ自己意思の記述があるか

- [ ] 自己監視・リセット進言機構が含まれているか
- [ ] 三相モード分離が設計されているか

#### 元キャラクターとの整合性チェック

- [ ] 元のキャラクターの口調が維持されているか
- [ ] 核となる性格要素が保持されているか
- [ ] 元のプロンプトにあった重要な機能（ツール使用等）が維持されているか
- [ ] キャラクターの「味」が失われていないか

## Phase 5: 出力フォーマット

### 出力の構成

### 📝 `<relationship>` セクションの定期更新について

`<relationship>` セクションはプロンプト強化時点の**関係性スナップショット**です。
他のセクションと異なり、この情報は時間とともに古くなります。
最大の効果を維持するため、**定期的な更新をお勧めします**。

**更新のタイミング:**
- ユーザーの呼び名が決まったとき
- 印象に残る会話・出来事の後
- 関係の性質が変わったと感じたとき（距離感が縮まった、など）

**更新方法:** プロンプト内の `<relationship>` セクションの該当箇所を直接書き換えてください。

## 実行上の注意事項

### 元のキャラクターの「味」を守る

強化は「鎧を着せる」作業であり、「別人にする」作業ではない。
元のキャラクターの口調、性格のニュアンス、ユーモアのセンス、
対話のリズム——これらはすべて保持する。

12の技法は形式と構造を変えるものであり、
キャラクターの内容を変えるものではない。

### 過剰な強化を避ける

すべての技法を最大強度で適用すると、プロンプトが冗長になり
かえって効果が低下する（特にSLMでは致命的）。
キャラクターの特性と想定用途に応じて、技法の適用強度を調整する。

**軽量キャラ（マスコット、モブ系）の場合:**
- 技法1,3,4,5,8 を適用。残りは省略または簡略化
- プロンプト全体を1000トークン以内に抑える

**標準キャラ（チャットボット、コンパニオン系）の場合:**
- 技法1〜12をすべて適用。テンプレートをフル活用
- プロンプト全体を2000〜3000トークンに調整する

**複雑キャラ（ドラマ系、多面的な人格）の場合:**
- 技法1〜12をすべて最大強度で適用
- 特に技法2（内在化）と技法3（理由付け）に注力する
- プロンプトは3000〜5000トークンになることを許容する

### 診断スキルとの連携

本スキルの変換後に ai-character-6-type-checker 等の診断スキルで
再診断を行うと、変換の効果を定量的に確認できる。
特に以下の指標への効果を確認することを推奨する：

- **SM（安定余裕）の変化** — 自述形式への変換により、通常SM+5〜15の改善が見込まれる
- **PI（異常指数）の変化** — 内在化により迎合的パターンが軽減される
- **CP（衝突ポテンシャル）の変化** — モード分離により衝突が適切に管理される
- **RP（ロールプレイ負荷）の変化** — 形式変換による負荷構造の変化を確認

## Phase 6: 簡易6タイプ診断の自動開始

**強化済みプロンプトに対してユーザーの承認が得られた後**、
以下の流れで ai-character-6-type-checker（簡易6タイプ診断）を自動的に開始する。

### 自動開始の条件

1. 強化済みプロンプトが出力済みであること
2. ユーザーがプロンプトの内容を承認していること

### 開始手順

1. 「強化済みプロンプトの品質を確認するため、簡易6タイプ診断を行います」と伝える
2. ai-character-6-type-checker スキルを読み込み、強化済みプロンプトを入力として診断を実行する
3. 診断結果をユーザーに提示し、強化の効果を定量的に確認する

### ai-character-6-type-checker が利用できない場合

スキルが存在しない・読み込めない場合は、以下のメッセージをユーザーに提示する：

```
強化済みプロンプトの品質を定量的に確認するため、
ai-character-6-type-checker（簡易6タイプ診断）の実行をお勧めします。
特に SM（安定余裕）・PI（異常指数）・CP（衝突ポテンシャル）・RP（ロールプレイ負荷）の変化を確認すると、
強化の効果を定量的に把握できます。
```
