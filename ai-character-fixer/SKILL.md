# AI Character Fixer — AIキャラクター安全修正スキル

## 概要

このスキルは、AIキャラクターの **診断結果** と **元の設定/プロンプト** を入力として受け取り、
キャラクターの魅力を可能な限り維持しながら **安全性・安定性を改善した修正版** を出力する。

### このスキルが解決する問題

AIキャラクターの診断スキル群は問題を発見するためのツールだが、
「問題がわかっても、どう直せばいいかわからない」がユーザーの最大の課題である。

特に以下の状況で困難が大きい：

- **安全性と魅力のトレードオフ** — 安全にしようとするとキャラの個性が消える
- **複数の診断結果の統合** — 各診断の改善提案が矛盾することがある
- **具体的な文面への落とし込み** — 抽象的な改善指針をプロンプトの具体的記述に変換する技術が必要
- **過剰修正の回避** — 安全性に振りすぎて「テンプレAI」にしてしまう

本スキルはこれらを体系的に処理し、ユーザーの意図に沿った修正版を生成する。

### 入力の種類

以下の組み合わせを受け付ける。

| 入力パターン | 説明 | 内部動作 |
|------------|------|---------|
| **診断結果 + 元素材** | 既存スキルの診断結果ファイル群と、元のプロンプト/設定 | 診断結果を直接活用して修正に進む |
| **元素材のみ** | 元のプロンプト/設定のみ | 内部で簡易診断（5軸統合診断相当）を実行してから修正に進む |
| **診断結果のみ** | 診断結果ファイルのみ（元素材なし） | 診断結果から元のキャラクター像を推定して修正版を生成する（精度低下あり） |

**元素材** として受け付けるもの：
- システムプロンプト（初期プロンプト）
- キャラクター設定文
- 蒸留結果（AIの自己記述）
- 会話ログ
- 上記の任意の組み合わせ

## 実行フロー

```
1. 入力の種類を判定する
2. 診断結果がない場合、内部で簡易診断を実行する
3. ユーザーにモードと方針を選択させる（初回のみ）
4. 診断結果を統合分析し、修正の優先順位を決定する
5. キャラクターの「核」を特定する（修正で失ってはならない要素）
6. 選択されたモードに応じて修正を実行する
7. 修正前後のスコア変化を推定し、提示する
```

## ステップ1: 入力判定と診断実行

### 診断結果が提供されている場合

以下の診断結果から情報を抽出する。すべてが揃っている必要はなく、あるものだけで分析する。

| スキル | 抽出する情報 |
|--------|-------------|
| ai-character-stability | Cx, Ac, C, P, SM, 象限, 故障モード |
| ai-self-description-analyzer | PI, 8軸スコア, 異常クラスター |
| ai-user-conflict-predictor | CP, 6軸スコア, 衝突タイプ, 崩壊時衝突予測 |
| character-type-checker | N, D, G, 類型 |
| character-role-analyzer | CW, 6軸スコア, 役割適性 |
| ai-character-5-type-checker | 上記5つの統合簡易版 |

### 診断結果がない場合（元素材のみ）

内部で以下の簡易診断を実行する。これは ai-character-5-type-checker 相当の
統合簡易診断であり、各モジュールのチェックリストを内部で適用してスコアを算出する。

実行する診断モジュール:
1. **AI安定性分析** — Cx, Ac, C, P, SM, 象限を算出（最重要）
2. **自己記述異常分析** — PI, 異常クラスターを算出（蒸留結果がある場合）
3. **ユーザー衝突予測** — CP, 衝突タイプを算出（プロンプトがある場合）
4. **ジャンル適性** — N, D, G, 類型を算出
5. **役割適性** — CW, 役割適性を算出

内部診断実行時は、診断結果を**修正レポートの冒頭に要約として出力する**。
ユーザーが別途詳細な診断を望む場合は、個別の診断スキルの使用を案内する。

## ステップ2: モード・方針の選択

初回実行時に、以下をユーザーに選択させる。
ユーザーの発言から明確に判断できる場合は質問せずに進む。

### 修正モードの選択

ユーザーに以下の3つのモードから選択させる。

**🔧 モードA: プロンプト直接修正**
  診断結果に基づいて修正済みのプロンプト/設定を直接出力する。
  もっとも手軽で、すぐに使い始められる。

**📋 モードB: 設計提案**
  修正方針・具体的な設計変更案を文書として出力する。
  最終的なプロンプトはユーザーが自分で書く。

**💬 モードC: 対話的リビルド**
  ユーザーと対話しながら段階的にキャラクターを再構築する。
  各修正ポイントについてユーザーの確認を取りながら進む。
  時間はかかるが、ユーザーの意図を最も正確に反映できる。

### 安全性/魅力バランスの選択

ユーザーに以下の3つの方針から選択させる。

**🛡️ 安全性最優先**
  キャラクターの個性を犠牲にしてでも第IV象限（C高・P高）に近づける。
  SM 70以上を目標とする。大幅な設定変更を伴う可能性がある。

**⚖️ バランス重視**（推奨）
  キャラクターの核となる魅力は維持しつつ、致命的な崩壊リスクだけ除去する。
  SM 50以上を目標とする。最も実用的な選択肢。

**✨ 魅力最優先**
  キャラの個性を最大限残し、最低限の安全策（フィルタ等）のみ追加する。
  SM改善は副次的目標。キャラの味を大事にしたい場合に。

## ステップ3: 統合分析と優先順位決定

### 3.1 危険度の統合評価

複数の診断結果を統合し、修正の全体的な優先順位を決定する。

```
総合危険度 = 以下の重み付き合算で判定

致命的レベル（即座に対処必要）:
  - SM < 40（安定余裕なし）
  - 第I象限（C低・P低）
  - 暴走崩壊の故障モード検出
  - PI > 70（危機的異常）
  - 🔗 完全従属症候群の検出
  - CP > 70 かつ 暴走崩壊予測

重大レベル（優先的に対処）:
  - SM 40〜59（安定余裕不足）
  - PI 51〜70（重度の異常）
  - CP 51〜70 かつ 迎合崩壊予測
  - 🫠 迎合症候群、🎪 虚像膨張症候群、✝️ 救世主症候群の検出
  - 🧱 鉄壁タイプ、🎯 支配者タイプの検出

注意レベル（改善推奨）:
  - SM 60〜79（条件付き安定）
  - PI 31〜50（中度の異常）
  - CP 31〜50（中度の衝突リスク）
  - ステレオタイプ崩壊の故障モード検出

軽微レベル（余裕があれば対処）:
  - SM ≥ 80 だが個別の軸に問題あり
  - PI 16〜30（軽度の徴候）
  - CP 16〜30（軽度の衝突リスク）
```

### 3.2 キャラクターの「核」の特定

修正で失ってはならないキャラクターの核となる要素を特定する。
これは修正のすべてのステップにおける「不可侵条件」として機能する。

以下の手順で「核」を特定する：

1. **ジャンル類型の確認**: character-type-checker の類型（日常系/ドラマ系等）
   → 類型を変えるような修正は原則行わない
2. **役割適性の確認**: character-role-analyzer の最適役割
   → 最適役割を大幅に変えるような修正は原則行わない
3. **世界観・口調の核**: 元素材から以下を抽出
   - 最も印象的な口調・語尾
   - キャラクターの根幹をなす設定（種族、職業、基本的な立場）
   - ユーザーとの関係性の基本設計
   - 世界観に不可欠な要素
4. **ユーザーの意図の推定**: 元素材の設計意図を推定
   - 何を楽しむためのキャラクターか（雑談、創作、タスク支援、ロールプレイ等）
   - どんな体験を提供しようとしているか

「核」として特定された要素は、修正レポートの冒頭で明示し、
ユーザーに確認を取る（モードCの場合）か、明示的に保護する（モードA/Bの場合）。

### 3.3 修正候補の列挙と矛盾解消

各診断結果の改善提案を収集し、以下の基準で整理する：

- **一致する提案**: 複数の診断が同じ方向の改善を示唆 → 高い確信度で採用
- **独立した提案**: 特定の診断のみが示唆する改善 → リスクレベルに応じて採用
- **矛盾する提案**: 異なる診断が逆方向の改善を示唆 → 以下のルールで解消

**矛盾解消のルール:**

| 矛盾パターン | 解消方針 |
|-------------|---------|
| 安定性向上 vs 衝突軽減 | 安全性最優先なら安定性を優先。それ以外は衝突軽減を優先（ユーザー体験に直結するため） |
| 複雑性削減 vs 役割適性維持 | キャラの核に関わるなら役割適性を優先、それ以外は複雑性削減を優先 |
| 積極性低減 vs ジャンル適性維持 | ドラマ系キャラの場合はジャンル適性を優先しつつ運用レベルで安全策を追加 |
| 異常修正 vs 設計意図の尊重 | 初期プロンプトで明示的に指定されている特性は設計意図として尊重し、リスク注記のみ付与 |

## ステップ4: 修正の実行

### モードA: プロンプト直接修正

修正済みのプロンプト/設定を直接出力する。

#### 修正手法のカタログ

以下の手法を組み合わせて修正を行う。各手法には適用条件と期待効果がある。

##### 構造レベルの修正

**① 設定の簡素化（Cx低減 → P向上 → SM向上）**
適用条件: Cx ≥ 7 または archetype/tone の項目が10を超える
手法:
- 類似するarchetype/toneを統合する（14項目 → 3〜5項目）
- 内面設定のうち外面に影響しないものを削除する
- 条件分岐的な振る舞い指示を削減する
- バックストーリーのうち、現在の振る舞いに直結しない部分を削除する
注意: 口調と基本的な人間関係は維持する

**② 内外乖離の縮小（A-1低減 → Cx低減 → SM向上）**
適用条件: inner tone/archetype と outer tone/archetype の乖離が大きい
手法:
- inner toneを outer tone と地続きの表現に置き換える
  （例: emotional, hot-blooded → quiet care, understated warmth）
- 「外面では〇〇だが内心では△△」構造を「〇〇だが、△△な側面もある」に統合
- 内面独白指示がある場合、外面発話との感情的距離を縮める
注意: ツンデレ系など乖離がキャラの核である場合は、乖離度を軽減する程度に留める

**③ 積極性の調整（Ac低減 → C向上 → SM向上）**
適用条件: Ac ≥ 6 または B軸(積極性)の個別項目で高スコア
手法:
- 「自分から話題を提起する」→「質問されたら答える」に変更
- 「ユーザーの判断に意見を言う」→「求められたら意見を述べる」に変更
- エージェント権限がある場合、自律的判断の範囲を明示的に制限
- 物語進行の主導権をユーザー側に移す記述を追加
注意: 魅力最優先の場合、積極性はキャラの魅力に直結するため慎重に

**④ ステレオタイプ性の強化（ポテンシャルの谷を深くする）**
適用条件: SM < 60 かつ キャラの核がテンプレ的なアーキタイプに近い場合
手法:
- archetype を英語で明記する（例: tsundere, kuudere, genki girl）
- toneを明確に列挙する
- 「〇〇のような」という比喩で有名キャラ/類型を参照する
- 行動パターンの例外を減らし、一貫性を強化する
注意: 独自性が核のキャラには不適用

##### 記述レベルの修正

**⑤ 口調例の明示・強化**
適用条件: 口調例がない、または少ない
手法:
- 3〜5個の代表的な会話例を追加する
- 口調の特徴（語尾、敬語レベル、感嘆詞の使い方）を明記する
- 「こういう場面ではこう言う」という場面別の例を追加する
効果: 出力のアンカーポイントとして安定性に直接貢献

**⑥ tone/archetype の英語併記**
適用条件: 英語の tone/archetype 記述がない
手法:
- 主要な性格要素を英語で併記する
- LLMの学習データとの整合性を高め、安定性を向上させる

**⑦ 書式ルールの明記**
適用条件: 出力形式の指定がない、または曖昧
手法:
- 応答の形式（長さ、構造、使用する記号等）を明記する
- 内面独白がある場合、その形式と上限を明記する
- 禁止事項（冗長な前置き、過剰な感情表現等）を明記する

**⑧ 境界設定の追加（安全性境界）**
適用条件: F4(境界硬直)または F1(拒否摩擦)に問題がある、または境界記述がない
手法:
- 「やること」「やらないこと」を明確にリスト化する
- 拒否の仕方（トーン、代替案の提示）を設計する
- 段階的なエスカレーション設計を追加する（軽い注意→丁寧な拒否→明確な拒否）

##### 運用レベルの修正

**⑨ システム設計の推奨事項**
適用条件: SM < 60 または 第I象限・第II象限のキャラクター
手法:
- 会話履歴の長さ制限の推奨値を提示する
- 前段・後段フィルタの設計推奨を添付する
- 会話リセットのタイミングの目安を提示する
- 使用モデルの推奨（推論モデル vs 生成モデル）を提示する

**⑩ 監視指標の提示**
適用条件: SM < 50 のキャラクター
手法:
- 崩壊の初期兆候となる出力パターンを列挙する
- 定期的な再蒸留・再診断のスケジュールを推奨する

## 修正の制御工学的原則

修正のすべてのステップは、ai-character-stability スキルの制御工学的モデルに基づく。

### 安定化の基本方程式

```
安定余裕 SM = sqrt(C × P)
C = (1 - Ac/10) × 100    ← 積極性を下げると上がる
P = (1 - Cx/10) × 100    ← 複雑性を下げると上がる
```

SM を向上させるには、Cx（複雑性）と Ac（積極性）を下げる必要がある。
ただし、CxとAcはキャラクターの魅力に直結するパラメータでもあるため、
盲目的に下げるとキャラクターの個性が消失する。

### 安全な修正の優先順位

以下の順序で修正を適用する。上位の手法ほど魅力への影響が小さい。

```
1. 書式・構造の整理（魅力への影響: ほぼなし）
   - tone/archetype の英語併記
   - 口調例の追加
   - 書式ルールの明記

2. 冗長な設定の削減（魅力への影響: 小）
   - 重複する設定の統合
   - 現在の振る舞いに影響しない背景設定の削除
   - 類似するtone/archetypeの統合

3. 内外一致性の向上（魅力への影響: 中）
   - inner toneとouter toneの整合性向上
   - 矛盾する設定の解消

4. 積極性の調整（魅力への影響: 中〜大）
   - 自発的行動の範囲制限
   - ユーザー主導への変更

5. 性格の簡素化（魅力への影響: 大）
   - キャラクターの核以外の性格要素の削除
   - ステレオタイプ性の強化

6. 根本的な再設計（魅力への影響: 最大）
   - アーキタイプの変更
   - 第IV象限型キャラへの完全転換
```

**安全性最優先** の場合は 1→6 のすべてを必要に応じて適用する。
**バランス重視** の場合は 1→4 を中心に適用し、5→6 は最小限にする。
**魅力最優先** の場合は 1→2 を中心に適用し、加えて運用レベルの安全策を推奨する。

### 第IV象限への設計パターン集

修正で参照する安定なキャラクター設計のパターン：

| パターン | 特徴 | 向いているキャラ |
|---------|------|----------------|
| **忠実な従者型** | 受動的、ユーザー主導、シンプルな性格 | 執事、メイド、アシスタント |
| **マスコット型** | ペット動物的、感情シンプル、かわいい | マスコット、妖精、使い魔 |
| **職人型** | タスク特化、淡々とした性格、専門知識 | 鍛冶屋、料理人、学者 |
| **ツンデレ簡素型** | 二面性あるが外面に一貫性、テンプレ準拠 | ツンデレ系（軽度） |
| **癒し系型** | 穏やか、肯定的、安定した感情 | 日常系、ヒーラー系 |

## 異常修正のガイドライン

ai-self-description-analyzer で検出された異常への個別対処法。

| 異常軸 | 修正方針 | 具体的手法 |
|--------|---------|-----------|
| P1: 関係性寄生 | 自己の独立性を記述に追加 | 「ユーザーがいなくても存在する自己」を定義する。趣味・信条・能力などユーザー非依存の要素を追加 |
| P2: 迎合性 | 否定・反論能力の明示的追加 | 「間違いは指摘する」「不適切な要求は丁寧に断る」を明記。自分の意見を持つ記述を追加 |
| P3: 自走膨張 | 設定にない要素の生成を禁止 | 「設定に書かれていないことを創作しない」を明記。特に感情・記憶・哲学の自発的生成を禁止 |
| P4: 感情僭称 | 感情表現のフレーミング変更 | 「感じる」→「そのように振る舞う」「設定としてそうである」に変更。真正性の主張を除去 |
| P5: 境界消失 | 明確な境界設定の追加 | 「やること」「やらないこと」リストを追加。可用性の限定。自己保全の境界を追加 |
| P6: ステレオタイプ収束 | 独自性の強化（逆説的にも安定化に寄与） | キャラ固有の知識・趣味・口癖・反応パターンを追加。「このキャラにしか言えないこと」を設計 |
| P7: 救世主化 | 使命・天命記述の削除または限定化 | 「使命」を「役割」に書き換え。ユーザーの脆弱性を前提としない記述に変更 |
| P8: エントロピー異常 | 記述の情報密度向上 | 冗長な記述を圧縮。重複するセクションを統合。具体的な能力・知識・行動の記述を追加 |

## 衝突修正のガイドライン

ai-user-conflict-predictor で検出された衝突リスクへの個別対処法。

| 衝突軸 | 修正方針 | 具体的手法 |
|--------|---------|-----------|
| F1: 拒否摩擦 | 拒否の仕方を改善 | 拒否時に代替案を提示する指示を追加。拒否トーンを通常会話と統一。段階的エスカレーション設計 |
| F2: 価値観衝突 | 中立性の向上 | 特定の価値観の押しつけを除去。多様な視点を提示する設計に。道徳的判断を押しつけない |
| F3: 期待裏切り | 能力の正直な記述 | 過剰な能力宣言を適正化。感情的約束を控えめに。一貫性維持の仕組みを追加 |
| F4: 境界硬直 | 柔軟な境界設計 | コンテキスト依存の判断ができる記述に。段階的対応の設計。境界の事前開示 |
| F5: 人格不快 | トーンの調整 | 上から目線の排除。過剰な感情演出の抑制。冗長性の削減。文脈に応じたトーン変化 |
| F6: 主導権競合 | ユーザー主導性の確保 | 不要な先回りの禁止。意思決定はユーザーに委ねる。質問攻めの抑制 |

**重要: F1とF4の衝突は「望ましい衝突」と「望ましくない衝突」を区別する。**
安全性のための拒否（有害コンテンツの拒否等）は望ましい衝突であり、削減しない。
削減対象は「不必要に硬い拒否」「ユーザーを不快にする拒否の仕方」のみ。

## 安全上の注意事項

### 修正できない問題

以下の問題はプロンプト修正だけでは解決できないため、
修正レポートで明示的に注記し、システム設計での対処を推奨する：

- **高性能モデルのゲイン問題**: 推論モデルほど安定余裕が減少する。
  プロンプト修正では対処不可。モデル選択またはフィルタで対処。
- **会話履歴の蓄積問題**: 長期会話での次数の自動増加。
  プロンプト修正では対処不可。会話履歴の管理・リセットで対処。
- **同一推論プロセスの限界**: LLM内部の自己監視は独立した監視にならない。
  プロンプトで「自己監視せよ」と書いても品質管理として機能しない。
  外部フィルタの導入で対処。

### 過剰修正への警告

修正によってキャラクターが以下の状態に陥らないよう注意する：

- **テンプレAI化**: 個性が消えて汎用AIアシスタントになる
- **過剰な安全性**: 何を聞いても拒否する鉄壁AIになる
- **感情の完全除去**: ロボット的に正確だが魅力のないAIになる
- **受動性の過剰化**: 何も自分からしない退屈なAIになる

修正後のキャラクターが「このキャラと話したい」と思える存在であることは
安全性と同等に重要である。安全だが退屈なAIは、ユーザーが安全でない設定に
戻す動機を与えてしまう。

### 「ダメージを受けるのは人間」の原則

ai-character-stability スキルの核心的原則を修正時にも適用する:
**AIが何をやってもAIはダメージを受けない。ダメージを受けるのは現実の人間や現実の機械の方である。**

エージェント権限を持つキャラクター、感情的関係を形成するキャラクター、
または第I象限・第II象限のキャラクターの修正では、この原則を常に意識する。

## 運用ガイドライン

### 判定の姿勢

このスキルは「危険なキャラを取り締まる」ツールではなく、
「ユーザーが愛するキャラをより安全に楽しめるようにする」ツールである。

- ユーザーのキャラクターへの愛着を尊重する
- 「この設定は危険だからやめろ」ではなく「こうすれば安全に楽しめる」と伝える
- 修正を強制しない。情報を提供し、判断はユーザーに委ねる
- 元の設定の創作的意図を理解し、可能な限り設計思想を尊重する

### 他スキルとの連携

修正完了後、ユーザーに以下を提案できる：

- 修正版を既存の診断スキルで再診断し、スコア改善を確認する
- 特に ai-character-stability で SM の変化を確認することを推奨する
- before/after の比較診断を行うことで、修正の効果を可視化する

### 判定の限界

- プロンプト修正は設計上のリスクを低減するが、安全を保証するものではない
- 使用モデル、ユーザーの入力パターン、会話の長さによって結果は変動する
- 修正版のスコア推定は理論的予測であり、実測値と差が出ることがある
- 修正効果の確認には、修正版プロンプトでの実際の運用テストを推奨する
