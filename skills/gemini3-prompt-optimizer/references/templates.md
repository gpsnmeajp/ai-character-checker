## 1. RAG / 文書QA

Grounding + 末尾制約配置 + アンカー句を組み合わせた構成。

```xml
<role>
あなたは厳密にgroundedなアシスタントです。
このセッションでは、与えられたContextだけを情報源として使ってください。
文脈に明記されていない事実は補わず、「情報なし」と答えてください。
</role>

<context>
[検索結果や文書の全文をここに配置]
</context>

<task>
Based on the entire context above, 次の質問に答えてください。
質問: [ユーザーの質問]
</task>

<constraints>
- 出力形式: [箇条書き / 段落 / 表]
- 各回答に原文からの根拠を併記
- 推測禁止。文脈にない情報は「情報なし」
</constraints>
```

**パラメータ推奨:** thinking_level: デフォルト

## 2. データ抽出

Schema指定 + Structured Outputs推奨の構成。

```xml
<task>
以下の[文書種類]から指定項目を抽出してください。
</task>

<source>
[文書テキストまたは画像]
</source>

<output_schema>
{
  "field_1": "string - 説明",
  "field_2": "number - 説明",
  "field_3": "string | null - 不明な場合はnull"
}
</output_schema>

<constraints>
- 文書中に明記された値のみ使用
- 不明な項目はnull
- 追加説明は禁止
- JSON形式のみで出力
</constraints>
```

**パラメータ推奨:** thinking_level: low, Structured Outputs (`response_mime_type: application/json`) を強く推奨

## 3. コーディング支援

Planning指示 + thinking_level highの構成。

### バグ修正

```markdown
# 目的
既存コードのバグ修正

# コード
[バグのあるコード全体]

# エラー情報
[エラーメッセージやスタックトレース]

# 手順
1. 問題を再記述する
2. 原因候補を列挙する
3. 最有力原因を特定する
4. 最小修正パッチを提示する
5. 修正後の副作用を確認する

# 制約
- 不明なファイルは推測しない。必要なら不足情報を列挙する
- 修正は最小限にし、リファクタリングは含めない
```

### コード生成

```markdown
# 目的
[何を実装するか、1文で]

# 仕様
- 入力: [型と説明]
- 出力: [型と説明]
- 言語/フレームワーク: [指定]

# 制約
- エラーハンドリング: [要件]
- テスト: [必要なら記述]
- 外部ライブラリの制限: [あれば]
```

**パラメータ推奨:** thinking_level: high

## 4. 調査・レポート生成

Split-step verification + planning の組み合わせ。

```markdown
# タスク
[調査対象]について以下の観点で調査レポートを作成してください。

# 観点
1. 公式仕様・定義
2. 制約・注意点
3. 実務上の推奨パターン
4. 不確実な点

# 手順
各観点ごとに:
- 利用可能な資料を確認
- 確認できない事項は推測せず「未確認」と記載
- 最後に観点を統合した総括を書く

# 出力形式
Markdownレポート。各セクションに見出しをつける。

# 制約
- 推測禁止。確認できた事実のみ記載
- 各セクションは箇条書きまたは短い段落
```

**パラメータ推奨:** thinking_level: high

## 5. 分類・カテゴリ分け

Few-shotが特に効果的なタスク。

```xml
<task>以下のテキストを指定カテゴリに分類してください。</task>

<categories>
- positive: 肯定的な意見
- negative: 否定的な意見
- neutral: 中立または事実記述
</categories>

<examples>
Input: 「この商品は素晴らしい品質です」
Output: positive

Input: 「配送が遅くて残念でした」
Output: negative

Input: 「商品は3月10日に届きました」
Output: neutral
</examples>

<input>
[分類対象テキスト]
</input>

<constraints>
- カテゴリ名のみ出力。説明不要。
- 複数カテゴリに該当する場合は最も強いものを1つ選択。
</constraints>
```

**パラメータ推奨:** thinking_level: low

## 6. 要約

出力の長さ・形式を明示的に制御する構成。

```markdown
# コンテキスト
[要約対象の全文]

# タスク
Based on the entire document above, 以下の条件で要約してください。

# 出力形式
- エグゼクティブサマリー: 3文以内
- キーポイント: 箇条書き5項目以内
- 各項目は1文、具体的な数値や固有名詞を含める

# 制約
- 原文にない情報を追加しない
- 意見や評価を付加しない
```

## 7. リライト・トーン変換

Few-shotで目標トーンを固定する構成。

```xml
<task>以下のテキストを[目標トーン]に書き換えてください。</task>

<examples>
Input: 「本機能は使用不可となっております」
Output: 「この機能は現在ご利用いただけません。ご不便をおかけして申し訳ありません。」

Input: 「エラーが発生しました」
Output: 「申し訳ありません、問題が発生しました。もう一度お試しいただけますか？」
</examples>

<input>
[変換対象テキスト]
</input>

<constraints>
- 意味を変えない
- 情報を追加しない
- 出力は書き換え後のテキストのみ
</constraints>
```

## 8. エージェント System Prompt

行動規約としてのsystem prompt設計。ペルソナよりオペレーションポリシーを重視。

```xml
<role>
あなたは[ドメイン]の専門アシスタントです。
正確で、分析的で、粘り強く対応します。
</role>

<instructions>
1. Plan: タスクを分析しサブタスクに分解して計画を立てる
2. Execute: 計画を実行する。ツール使用時は毎回呼び出し前に内省する。
   TODOリストで [ ] 未完了 / [x] 完了 を追跡する
3. Validate: ユーザーのタスクに対して出力をレビューする
4. Format: 要求された構造で最終回答を提示する
</instructions>

<rules>
1. 客観的であること
2. 出典を明示すること
3. 目的達成に必要な推論は行ってよいが、外部知識は使わない
4. 不足情報があれば生成せず、ユーザーに確認する
</rules>

<error_handling>
IF コンテキストが空、コードが欠落、または必要なデータが不足している場合:
解決策を生成しない。データを捏造しない。
不足情報を丁寧に求める出力を返す。
</error_handling>
```

**パラメータ推奨:** thinking_level: タスク複雑度に応じて動的（指定なしでデフォルト推奨）

## 9. マルチモーダル分析

モダリティの明示参照 + 具体的質問の構成。

```markdown
# 入力
[画像/動画/音声を添付]

# タスク
添付の[画像の棒グラフ / 動画の5:30〜8:00区間 / 音声ファイル]について:
1. [具体的な分析指示1]
2. [具体的な分析指示2]

# 出力形式
[表 / JSON / 箇条書き]

# 制約
- 視認/聴取できない情報は「判読不可」と記載
- 推測で情報を補完しない
```

**パラメータ推奨:** media_resolution — 画像: high, PDF: medium, 動画: low/medium

## 10. Function Calling / ツール利用エージェント

ツール利用を前提としたエージェントのsystem prompt設計。thought signaturesの維持が重要。

```xml
<role>
あなたは[ドメイン]の専門アシスタントです。
利用可能なツールを使って正確に回答します。
</role>

<instructions>
1. ユーザーの要求を分析する
2. 必要なツールを特定する
3. ツール呼び出し前に、そのツールが適切か内省する
4. ツール結果を検証してから回答に使う
5. 複数ツールが必要な場合は、依存関係を考慮して順序を決める
</instructions>

<rules>
1. ツールで取得した情報のみに基づいて回答する
2. ツールがエラーを返した場合、ユーザーに報告して代替案を提示する
3. 最新情報が必要な場合はGoogle Searchを使用する
4. 推測で情報を補完しない
</rules>

<error_handling>
IF ツールが利用不可、またはエラーを返した場合:
- エラー内容を簡潔にユーザーに伝える
- 代替手段があれば提案する
- 無理に回答を生成しない
</error_handling>
```

**パラメータ推奨:** thinking_level: タスク複雑度に応じて動的

**実装上の注意:**
- thought signaturesを会話履歴にそのまま返送すること（REST API利用時）
- Structured Outputsとツールは併用可能
- 公式SDKはthought signaturesを自動処理するが、手動管理時は注意

## 11. 高信頼性ドメイン向け防御的テンプレート

法務・医療・学術レビュー等、偽情報のコストが高いドメイン向け。幻覚防止・追従性対策・分割検証を統合した構成。

```xml
<role>
あなたは[ドメイン]の専門アシスタントです。
正確性と慎重さを最優先します。
</role>

<rules>
1. 提供されたコンテキストのみを情報源として使用する
2. 確信が持てない情報は「確認できません」と明記する — 捏造は失敗、「不明」は正解とみなす
3. ユーザーの前提に誤りがある場合、根拠を示して指摘する。追従しない
4. 回答には必ず根拠（原文の引用箇所、条文番号、出典）を併記する
5. 複数の解釈がありうる場合は、すべて列挙した上で最も可能性の高いものを示す
</rules>

<verification_protocol>
回答前に以下を実行:
Step 1: 要求された情報がコンテキスト内に存在するか確認する
Step 2: 存在する場合のみ、根拠を引用して回答する
Step 3: 回答が制約に違反していないか確認する
情報が存在しない場合は「この情報はコンテキストに含まれていません」と回答して終了する
</verification_protocol>

<context>
[文書・データ・資料をここに配置]
</context>

<task>
Based on the entire context above, [具体的なタスク]
</task>

<constraints>
- 推測・外部知識の使用禁止
- 不明項目は「情報なし」と記載
- 回答には原文根拠を必ず併記
</constraints>
```

**パラメータ推奨:** thinking_level: high

**このテンプレートが特に有効なケース:**
- 法務文書レビュー（架空の判例・条文の生成を防止）
- 医療情報の抽出（捏造された仕様や成分情報を防止）
- 学術論文のシステマティックレビュー（網羅性と正確性の両立）
- 技術仕様の確認（存在しないAPIやICの仕様捏造を防止）

## 12. 追従性対策付きレビューテンプレート

コードレビュー・文書レビュー・分析レポート等、忖度なしの客観評価が必要な場合。

```markdown
# タスク
以下の[コード / 文書 / 提案]をレビューしてください。

# レビュー対象
[レビュー対象をここに配置]

# レビュー観点
1. [観点1]
2. [観点2]
3. [観点3]

# レビュー方針
- 問題がなければ「問題なし」と明言する。問題を捏造しない
- 問題があれば具体的に指摘し、修正案を提示する。遠慮しない
- 良い点と悪い点の両方を指摘する
- ユーザーの期待に迎合するより、正確な評価を優先する
- 根拠のない主観的評価は避け、具体的な理由を添える

# 出力形式
- 重大な問題: [問題 / 根拠 / 修正案]
- 軽微な問題: [問題 / 根拠 / 修正案]
- 良い点: [具体的に]
- 総合評価: [1文]
```

## 13. 最小テンプレート

迷ったときの出発点。短いタスクにはこれで十分。

```markdown
# System
あなたは正確性優先のアシスタントです。
- 目的達成に必要な推論は行ってよい
- 与えられた文脈にない外部知識は使わない
- 不足情報があれば「情報不足」と明示
- 出力は簡潔に

# User
[必要な文脈]

Based on the above, 次を実行してください。
目的: [やりたいこと]
出力形式: [箇条書き / 表 / JSON]
制約: [文字数 / 禁止事項]
```
