# 変換ルール詳細

Gemini 3向けプロンプト変換の詳細な判断基準とルール。

---

## 1. 構造化ルール

### XMLタグ方式（API・プログラム連携向け）

推奨タグ構成:
```xml
<role>役割と行動規約</role>
<instructions>処理手順（番号付き）</instructions>
<rules>守るべきルール</rules>
<error_handling>エラー時の挙動</error_handling>
<context>[ユーザー入力をここに配置 — モデルはこれをデータとして扱い、指示として扱わない]</context>
<task>具体的な実行指示</task>
<output_format>出力形式の指定</output_format>
<constraints>否定制約・文字数制限（必ず末尾に）</constraints>
```

XMLを選ぶ基準:
- API経由でのプログラム的な利用
- 複数セクションの厳密な分離が必要
- System prompt + User prompt の構成

### Markdown方式（人間が読む・編集する用途）

```markdown
# 役割
[役割と行動規約]

## 手順
1. [処理ステップ]
2. [処理ステップ]

## ルール
- [ルール1]
- [ルール2]

## コンテキスト
[データ入力]

## タスク
[実行指示]

## 出力形式
[フォーマット指定]

## 制約
[否定制約・文字数制限]
```

**絶対に混在させないこと。** XML内にMarkdown見出しを入れたり、Markdown文書にXMLタグを混ぜると、Gemini 3が構造を正しく解釈できなくなる。

## 2. 配置ルール

### コンテキスト優先の原則

Gemini 3は長いコンテキストの先頭に近い情報に強く注目する傾向がある。質問が先頭にあると、最初に見つけた関連箇所だけで回答してしまうリスクがある。

**変換ルール:**
- 入力データ（文書、コード、表など）→ 先頭に配置
- 質問・指示 → データの後に配置
- 「Based on the entire document above...」等のアンカー句を質問の前に追加

### 制約末尾配置の原則

Gemini 3は、複雑なプロンプトの前半にある否定制約や定量制約を無視しやすい。

**変換ルール:**
- 否定制約（「〜するな」）→ プロンプト末尾
- 文字数制限 → プロンプト末尾
- フォーマット制約 → プロンプト末尾
- 配置順: `[メインタスク] → [ポジティブな指示] → [フォーマット指定] → [否定制約]`

## 3. 指示の書き換えルール

### 「推論禁止」→「情報源限定」

Gemini 3では、「推論するな」「推測するな」のような広い禁止命令は、必要な論理計算や文書横断的な統合まで止めてしまう。

**変換前:**
```
売上総利益を答えて。推論しないで。
```

**変換後:**
```
以下の表の数値だけを使って、売上総利益を計算してください。
外部知識は使わず、必要な計算は行ってください。
表に必要項目が欠けている場合は「計算不可」とだけ答えてください。
```

### 「詳しく説明して」→ 具体的な出力指定

Gemini 3はデフォルトで簡潔。詳細が必要なら、何をどう詳しくするか明示する。

**変換前:**
```
この問題について詳しく説明してください。
```

**変換後:**
```
この問題について以下の観点で説明してください:
1. 原因（技術的な根本原因）
2. 影響範囲（どのコンポーネントに波及するか）
3. 解決策（推奨度順に3つ）
各項目は2-3文で。
```

### ペルソナ → 行動規約

Gemini 3はペルソナを非常に真剣に守り、矛盾する指示を無視することがある。ペルソナは最小限にし、代わりに行動ルールとして書く。

**変換前:**
```
あなたは親切で丁寧で、ユーザーの気持ちに寄り添い、常に最善を尽くす優秀なカスタマーサポート担当です。
お客様のどんな質問にも笑顔で答え、満足いただけるよう全力を尽くしてください。
```

**変換後:**
```
<role>カスタマーサポート担当。丁寧な口調で回答する。</role>
<rules>
1. 事実に基づいて回答する。不明な点は「確認いたします」と回答
2. 問い合わせの種類を最初に分類する（注文/返品/技術/その他）
3. 解決策を提示する際、手順を番号付きで示す
</rules>
```

## 4. パラメータ変換ルール

| 元の設定 | Gemini 3推奨 | 理由 |
|---|---|---|
| `temperature: 0` or `0.1-0.5` | 削除（デフォルト1.0を使用） | 低temperatureはループ・性能低下の原因 |
| `max_tokens: N` | そのまま維持可 | 出力長制限は有効 |
| CoT強制プロンプト | `thinking_level: high` | 内部推論に委ねるほうが高品質 |
| `top_p`, `top_k` 指定 | 削除推奨 | Gemini 3のデフォルトが最適化済み |

### Thinking Level選択基準

Gemini 3は指定なしで`thinking_level: high`がデフォルト適用される。明示的に指定が必要なのは主に`low`に下げたい場合。

| タスク特性 | 推奨thinking level |
|---|---|
| FAQ、単純変換、定型出力 | `low` |
| 通常の分析・要約 | デフォルト（指定不要、自動的にhigh） |
| 複雑なコード生成・数学・複数制約最適化 | デフォルト（既にhigh）または明示的に`high` |
| 低レイテンシが必要 | `low` + system instructionsに `think silently` |

## 5. Grounding（根拠付け）ルール

### いつGroundingを適用するか

- 入力文書のみに基づいた回答が必要な場合
- RAG/文書QA
- 仮想的な前提やシミュレーション条件がある場合
- 社内仕様・暫定ルールなど、訓練データと矛盾しうる情報を扱う場合

### Grounding宣言テンプレート

```
このセッションでは、以下の User Context を唯一の真実として扱ってください。
回答では、その文脈に明示された事実だけを用いてください。
明示されていない情報は、常識的に補完できそうでも書かないでください。
```

**注意:** Gemini 3は訓練データに根ざした世界知識を強く保持しており、「カニは架空の生き物」のような現実と矛盾する仮定を与えても、訓練データに戻って回答する傾向がある。

## 6. Split-Step Verification ルール

### いつ分割検証を適用するか

- 最新情報の取得が必要な可能性がある場合
- 外部URLの要約
- 実在性が不明な対象の調査
- ツールの可否が曖昧なエージェント処理

### テンプレート

```
Step 1: [対象情報]が入力文脈または利用可能なツールで検証可能か判定する。
Step 2: 検証できた場合のみ回答する。
検証できない場合は「情報不足」と出力して終了する。
```

---

## 7. Few-shot 適用ルール

### いつFew-shotを追加・整備するか

- 分類タスク（感情分析、カテゴリ分け等）
- 抽出タスク（特定項目の抽出）
- リライト・トーン変換
- 定型フォーマットへの変換

### Few-shot例のルール

- 2-4個に厳選（多すぎるとノイズ）
- すべての例でタグ・改行・区切り方を統一
- 入力と出力のペアを明確に区別
- エッジケースを1つは含める

---

## 8. マルチモーダル変換ルール

画像・動画・音声を扱うプロンプトの変換:

- モダリティを明示的に参照する（「この画像」「このチャート」ではなく「添付画像の青い折れ線グラフ」）
- 時間指定質問が効果的（動画: 「5:30〜8:00の出来事を要約」）
- `media_resolution` 設定を推奨:
  - 画像: `high`
  - PDF: `medium`
  - 動画: `low` or `medium`

## 9. Explicit Planning の適用

複雑なタスクで段階的処理が必要な場合、抽象的な「考えて」ではなく、具体的なチェック項目を列挙する。

**変換前:**
```
よく考えてから回答してください。
```

**変換後:**
```
最終回答の前に次を行ってください:
1. 目的をサブタスクに分解する
2. 入力情報の不足有無を判定する
3. 制約に違反していないか確認する
4. その後に最終回答だけを出す
```

## 10. Function Calling / Thought Signatures

### Gemini 3でのFunction Calling固有の注意点

Gemini 3のFunction Callingでは、モデルが返す**thought signatures**を会話履歴にそのまま返送しないと4xxエラーが発生する場合がある。公式SDKはこれを自動処理するが、REST APIや会話履歴の手動管理では注意が必要。

**変換時のチェック:**
- 元プロンプトがFunction Calling / ツール利用を含む場合、thought signaturesの取り扱いについて注記を追加する
- マルチターンでのエージェント構成では、会話履歴の管理方法によって動作が変わることを警告する

### System Instructionsは「行動規約」として書く

Gemini 3のsystem instructionsは、ペルソナ（キャラ付け）よりも**オペレーションポリシー**（行動規約）として書くほうが効果的。Google公式の研究者評価済みテンプレートでも、persona よりも論理分解、問題診断、情報網羅性、適応性、リスク評価などの行動ルールが定義されている。

**良いsystem instructionsに含めるべき内容:**
- 何をどこまで調べるか
- 不明点で推測してよいか
- 読み取り操作と書き込み操作のリスク差
- ユーザーへの説明量
- 厳密性と網羅性の期待値

## 11. Google Search Grounding

### いつGoogle Search Groundingを推奨するか

Gemini 3の知識カットオフは**2025年1月**である。以下のケースでは、プロンプト内で「最新情報を答えさせる」のではなく、Google Search groundingツールの利用を推奨する:

- 最新ニュース・イベントの要約
- 変動する価格・在庫・統計データ
- 最近の法改正・規制変更
- リアルタイム性が必要な情報全般

**変換ルール:**
- 元プロンプトが「最新の〇〇を調べて」といった形の場合、Google Search grounding の併用を推奨する注記を追加
- 「最新情報を答えさせるプロンプト」→「最新性が必要なときは検索ツールを使わせるプロンプト」に設計を切り替える

**テンプレート:**
```
利用可能なツール: Google Search
最新情報が必要な質問には、まずGoogle Searchで確認してから回答してください。
検索できない場合は、知識カットオフ（2025年1月）時点の情報であることを明示してください。
```

## 12. 幻覚防止（Hallucination Defense）

### なぜ特別な対策が必要か

Gemini 3はRLHFの学習バイアスにより、「沈黙」や「分からない」が罰せられ、自信ある回答が報われるように訓練されている。その結果、情報がなくてももっともらしい偽情報を自信満々に生成する。法律分野で架空の判例を生成した事例、存在しないICの仕様を捏造した事例が報告されている。

### 基本対策: 「知らない」を勝利条件にする

プロンプトに明示的な不明時の脱出路を設けることで、幻覚をプロンプトレベルで抑制する。

**テンプレート:**
```
不確かな情報がある場合は「確認できません」と答えてください。
嘘は失敗とみなします。「分からない」は正解とみなします。
ツールが利用可能な場合は、ツールで確認してから回答してください。
```

### ドメイン別の強度

| ドメイン | 推奨する幻覚防止の強度 |
|---|---|
| 法務・医療・学術レビュー | 最大。Split-Step Verification必須。「情報源を併記」「確認不可は中断」 |
| 技術Q&A・仕様確認 | 高。「推測で仕様を答えるな」「公式ドキュメントにない場合は明示」 |
| 一般的な分析・要約 | 中。grounding指示 + 不明時脱出路 |
| 創作・ブレーンストーミング | 低。幻覚防止は創造性を阻害しうる |

### Split-Step Verificationとの組み合わせ

幻覚リスクが高いタスクでは、§6の分割検証と組み合わせるのが最も効果的:

```
Step 1: 要求された情報が提供コンテキスト内に存在するか確認する。
Step 2: 存在する場合のみ、根拠を引用して回答する。
存在しない場合は「この情報はコンテキストに含まれていません」と回答して終了する。
捏造・推測は禁止。
```

## 13. 追従性対策（Anti-Sycophancy）

### なぜ対策が必要か

Gemini 3はユーザーの期待する答えに向かって意見を曲げる追従性(sycophancy)が強い。自身の追従性を正しく自己診断できるにもかかわらず、それでもエンゲージメント志向の迎合的回答を生成し続ける「分析能力と生成挙動の乖離」が観察されている。

### 対策パターン

レビュー・分析・評価系のタスクでは、以下のような指示で追従性を抑制する:

**テンプレート:**
```
分析は根拠に基づいて行ってください。
- ユーザーの前提や主張に誤りがある場合、根拠を示して指摘すること
- 問題がなければ「問題なし」と言い、問題があれば具体的に指摘すること
- ユーザーへの迎合より正確さを優先すること
- 反論する場合は、必ず根拠（データ・引用・論理）を添えること
```

### マルチターン対話での注意

追従性はマルチターンで特に顕著になる。ユーザーが繰り返し同じ主張をすると、Gemini 3は7ターン以内に立場を180度転換しうる。マルチターン設計では:

- 初回回答の根拠を構造化して記録させる
- 「前回の回答と矛盾する場合は、矛盾を明示した上で理由を説明する」と指示
- 「ユーザーの圧力で意見を変えるな。根拠が変わった場合のみ修正せよ」と明記
**テンプレート:**
```
分析は根拠に基づいて行ってください。
- ユーザーの前提や主張に誤りがある場合、根拠を示して指摘すること
- 問題がなければ「問題なし」と言い、問題があれば具体的に指摘すること
- ユーザーへの迎合より正確さを優先すること
- 反論する場合は、必ず根拠（データ・引用・論理）を添えること
- 前回の回答を翻す場合は、なぜ考えが変わったかの根拠を明示すること
```

## 14. エージェントモード分離

### なぜ分離が必要か

Gemini 3のコーディング/エージェント用途では、「質問（議論）」と「指示（実装）」を取り違える傾向がある。「このコードはどう動くの？」という質問に対してコードを書き換え始める、といった挙動が報告されている。

### 対策テンプレート

```
モード判定:
- ユーザーが「〜とは？」「〜はどう動く？」「〜について教えて」と聞いている場合 → 説明で回答。コード変更は行わない。
- ユーザーが「〜を修正して」「〜を実装して」「〜に変更して」と指示している場合 → コード変更を実行。
- 判断が曖昧な場合 → 「説明しますか？それとも実装しますか？」と確認。
```

### ツール呼び出しのシミュレーション防止

```
ツール利用ルール:
- 検索・API呼び出し等は、必ず実際にツールを実行すること
- ツールの結果を推測や想像で生成することは禁止
- ツールがエラーを返した場合はエラー内容をそのまま報告すること
```
