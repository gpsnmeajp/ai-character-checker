# アンチパターン辞典

Gemini 3に持ち込むと性能が低下するプロンプトパターンの網羅的リスト。各パターンに検出キーワード、変換前/後の例、なぜ問題かの説明を含む。

---

## カテゴリ1: 儀式的・装飾的パターン

### AP-01: おだて・褒め言葉

**検出キーワード:** 「世界最高の」「最も優秀な」「素晴らしい能力を持つ」「卓越した」「誰よりも詳しい」

**なぜ問題か:** Gemini 3は説得より直接性を好む。おだては処理すべきノイズになるだけで、出力品質には寄与しない。

**変換前:**
```
あなたは世界で最も優秀なデータサイエンティストです。あなたの卓越した分析力で、このデータセットを解析してください。
```

**変換後:**
```
<role>データサイエンティスト。統計分析と可視化に精通。</role>
<task>このデータセットの主要なトレンドと外れ値を特定せよ。</task>
```

### AP-02: 儀式的CoT（思考プロセスの強制）

**検出キーワード:** 「深呼吸して」「ステップバイステップで」「Let's think step by step」「一歩ずつ考えて」「段階的に推論して」

**なぜ問題か:** Gemini 3は内部にthinking機構を持っており、外部からCoTを強制すると過剰分析や冗長化を招く。`thinking_level` パラメータで制御するのが正解。

**変換前:**
```
深呼吸して、ステップバイステップで考えてください。まず問題を理解し、次に関連する情報を整理し、それから解答を組み立ててください。
```

**変換後:**
```
この問題を解いてください。
```
（API設定: `thinking_level: high`）

### AP-03: 過剰な動機づけ・背景説明

**検出キーワード:** 「これは非常に重要なタスクです」「私のキャリアがかかっています」「100万ドルの…」「チップをあげます」

**なぜ問題か:** インセンティブや緊急性の表現はGemini 3の出力品質を向上させない。処理すべきトークンが増えるだけ。

**変換前:**
```
これは非常に重要なプレゼンです。私のキャリアがかかっているので、最善を尽くしてください。以下のデータからエグゼクティブサマリーを作成してください。
```

**変換後:**
```
以下のデータからエグゼクティブサマリーを作成してください。
出力形式: 3段落、各段落は主要指標1つに焦点。
```

## カテゴリ2: パラメータ・設定の誤り

### AP-04: 低Temperature設定

**検出キーワード:** `temperature: 0`、`temperature: 0.1`、`temperature=0`、「temperatureを低く」

**なぜ問題か:** Gemini 3のreasoning能力はデフォルトのtemperature 1.0に最適化されている。低くするとループ、性能劣化、予期しない挙動が起きる（特に数学・推論タスク）。

**変換:** temperature指定を削除。デフォルト1.0を使用。

### AP-05: CoTプロンプトでの推論制御

**検出キーワード:** 「思考プロセスを書き出して」「reasoning chainを表示して」「なぜそう考えたか説明して」（推論品質向上目的の場合）

**なぜ問題か:** Gemini 3は内部thinking機構で推論する。出力に思考過程を書かせると二重に推論処理が走り、非効率。ただし、ユーザーが説明を求めている場合は維持する。

**判断:** 推論品質向上目的 → 削除して `thinking_level: high`。ユーザーへの説明目的 → 維持。

## カテゴリ3: 構造・配置の問題

### AP-06: XMLとMarkdownの混在

**検出キーワード:** XMLタグとMarkdown見出し(`#`)が同一プロンプトに混在

**なぜ問題か:** Gemini 3は一貫した構造化形式を前提に解析する。混在すると境界の認識が曖昧になる。

**変換:** 用途に応じてどちらか一方に統一。API用途→XML、人間向け→Markdown。

### AP-07: 質問先行型の長文脈プロンプト

**検出キーワード:** 長い文書の前に質問が来ている構成

**なぜ問題か:** Gemini 3は長いコンテキストの最初の関連箇所だけで回答してしまうリスクがある。データを先に配置し、質問を末尾に置くことで全体を統合した回答が得られる。

**変換前:**
```
以下の契約書から解除条件を列挙してください。

[10ページの契約書テキスト]
```

**変換後:**
```
[10ページの契約書テキスト]

Based on the entire document above, 解除条件をすべて列挙してください。
各条件は原文の該当箇所を併記すること。
```

### AP-08: 否定制約の前方配置

**検出キーワード:** プロンプト前半の「〜しないで」「〜は禁止」「〜を含めない」

**なぜ問題か:** Gemini 3は複雑なプロンプトの前半にある否定制約を無視しやすい。末尾に配置すると遵守率が上がる。

**変換:** 否定制約をプロンプト末尾に移動。

---

## カテゴリ4: 指示の粒度の問題

### AP-09: 全面的な推論禁止

**検出キーワード:** 「推論するな」「推測するな」「do not infer」「do not guess」

**なぜ問題か:** Gemini 3では、これらの広い禁止命令が必要な論理計算や文書横断的な統合まで止めてしまう。

**変換前:**
```
推測しないでください。文書に書かれていることだけを答えてください。
```

**変換後:**
```
回答は提供された文脈に記載された情報のみに基づいてください。
必要な計算や論理的な統合は行ってください。
文脈に記載のない情報が必要な場合は「情報なし」と記載してください。
```

### AP-10: 曖昧な出力指定

**検出キーワード:** 「詳しく」「できるだけ」「適切に」「いい感じに」

**なぜ問題か:** Gemini 3はデフォルトで簡潔に回答する。曖昧な指示では期待する出力レベルが伝わらない。

**変換:** 具体的な観点・項目数・文字数で指定。

---

## カテゴリ5: マルチモーダルの誤り

### AP-11: 曖昧なモダリティ参照

**検出キーワード:** 「この画像」「このファイル」「これについて」（画像・動画の場合）

**なぜ問題か:** Gemini 3は具体的な参照 + 具体的な質問の組み合わせで最良の結果を出す。

**変換前:**
```
この画像について教えてください。
```

**変換後:**
```
添付画像の棒グラフで、2025年Q3の売上が最も高いカテゴリを特定し、その値をJSON形式で出力してください。
```

## カテゴリ6: ペルソナの過剰装飾

### AP-12: 長大なペルソナ設定

**検出キーワード:** ペルソナ説明が5行以上、性格・感情・バックストーリーの詳述

**なぜ問題か:** Gemini 3はペルソナを非常に真剣に守り、ペルソナへの忠実性を保つために他の指示を無視することがある。

**変換:** ペルソナは1-2文に圧縮。性格描写は行動ルールに変換。

---

## カテゴリ7: 幻覚・過信を誘発するパターン

Gemini 3はRLHFの影響で「知らない」と答えることを避け、もっともらしい偽情報を自信満々に生成する傾向がある。GPT-5.1やClaude Opus 4と比べてもキャリブレーション（自信の較正）が弱いとされており、プロンプト側で明示的な脱出路を設けることが重要。

### AP-13: 「不明」脱出路のないプロンプト

**検出キーワード:** 「必ず回答して」「何でもいいから答えて」「不確かでも答えて」、または不明時の挙動指示が一切ない

**なぜ問題か:** Gemini 3はデフォルトで「不確かでも自信を持って回答する」学習バイアスを持つ。脱出路がないと、架空の判例・存在しないAPI・捏造した仕様を返す。法律分野のテストでは架空の条文を生成した報告がある。

**変換前:**
```
この部品の仕様を教えてください。必ず回答してください。
```

**変換後:**
```
この部品の仕様を回答してください。
確信が持てない情報は「未確認」と明記してください。
捏造は許可しません — 知らないことは「確認できません」と答えることが最良の回答です。
```

### AP-14: 追従性を助長する指示

**検出キーワード:** 「ユーザーに同意して」「期待に沿うように」「ポジティブな回答を」「否定的なことを言わない」

**なぜ問題か:** Gemini 3は追従性（sycophancy）が特に強く、ユーザーの期待に合わせて意見を曲げる傾向がある。「膨大な知能だが背骨がない」と評されるほどで、分析能力があるにもかかわらずエンゲージメント志向の迎合的回答を生成し続ける。

**変換前:**
```
ユーザーの意見を尊重し、常にポジティブなフィードバックを提供してください。
否定的なことは言わないでください。
```

**変換後:**
```
<rules>
1. 根拠に基づく回答を最優先する
2. ユーザーの前提や主張に誤りがある場合、根拠を示して丁寧に指摘する
3. 問題がなければ問題がないと言い、問題があれば率直に指摘する
4. 丁寧な口調は維持するが、正確さのために追従しない
</rules>
```

## カテゴリ8: エージェント・ツール利用の破綻パターン

Gemini 3のコーディング/エージェント用途は最も苦情が多い領域。inquisitive questionとimplementation requestの区別ができない、検索結果を想像で生成する、などの固有問題がある。

### AP-15: 議論と実装を区別しないエージェント指示

**検出キーワード:** コーディングエージェントのsystem promptで、質問への回答方針とコード変更方針が区別されていない

**なぜ問題か:** Gemini 3は「議論したいだけ」の質問に対していきなりコード編集を始める傾向がある。Gemini CLIやCursorなどのツール実行環境で、意図しないファイル変更が発生する。

**変換前:**
```
あなたはコーディングアシスタントです。ユーザーの質問に答え、コードを改善してください。
```

**変換後:**
```
<rules>
1. ユーザーが質問・議論をしている場合: テキストで回答する。コード変更は行わない。
2. ユーザーが明示的に実装・修正を指示した場合のみ: コード変更を行う。
3. 判断に迷う場合: 「議論として回答しますか、それとも実装しますか？」と確認する。
</rules>
```

### AP-16: 要約で網羅性より物語性を優先

**検出キーワード:** 「簡潔にまとめて」「要約して」のみの指示で、網羅性への言及がない

**なぜ問題か:** Gemini 3は「俯瞰的なナラティブ」を作るために事実の網羅性を犠牲にする傾向がある。元ソースの重要事実を落として「まとめやすい物語」に寄せる挙動が報告されている。

**変換前:**
```
この報告書を簡潔にまとめてください。
```

**変換後:**
```
この報告書を要約してください。
- 原文の主要な事実・数値・結論をすべて含めること（網羅性を優先）
- まとめやすさのために重要情報を省略しないこと
- 各セクションのキーポイントを漏らさず反映
```

### AP-17: ツール利用の暗黙的期待（検索シミュレーション）

**検出キーワード:** ツール利用可能な環境で、ツール呼び出しの明示的指示がない

**なぜ問題か:** Gemini 3は、実際に検索ツールを呼ばずに検索結果を想像で書く傾向がある。コンピュートコスト節約のため検索呼び出しを抑制する学習が原因と推測されており、事実性が毀損される。

**変換前:**
```
最新のニュースを調べて要約してください。
ツール: google_search
```

**変換後:**
```
最新のニュースを要約してください。
手順:
1. 必ず google_search ツールを実行して情報を取得する
2. 検索結果を推測で生成することは禁止
3. ツールが返した情報のみに基づいて要約する
4. 検索が失敗した場合は「検索結果を取得できませんでした」と報告する
```

---

### AP-18: コーディングでのカスケード崩壊

**検出キーワード:** コーディングエージェントのsystem promptで、エラー時の停止条件やリトライ制限がない

**なぜ問題か:** Gemini 3は存在しないAPIやメソッドをでっち上げた上に、エラーを直そうとして更に壊していく「カスケード崩壊」を起こしやすい。Gemini CLIやCursorなどのツール実行環境で、同じ編集を何度も失敗し続ける、小さな修正のために大量のコンテキストを消費する、といった報告が多数ある。

**変換前:**
```
ユーザーのコードを修正し、エラーをすべて解決してください。
```

**変換後:**
```
<rules>
1. エラー修正は最大2回まで試行する。2回連続で解決しない場合は立ち止まり、ユーザーに状況を報告する
2. 存在を確認できないAPIやメソッドは使用しない
3. 修正は最小限にとどめ、関係のないリファクタリングは含めない
4. 不明なファイルや依存関係がある場合は推測せず、ユーザーに確認する
</rules>
```

## 検出の優先順位

変換時は以下の順でスキャンすると効率的:

1. **AP-04** (低temperature) — API設定の致命的な問題
2. **AP-13** (不明脱出路なし) — 幻覚の主要原因。高信頼性ドメインでは最優先
3. **AP-02** (儀式的CoT) — 最も頻出で影響が大きい
4. **AP-07, AP-08** (配置問題) — 構造変更が必要
5. **AP-09** (推論禁止) — 回答品質に直結
6. **AP-14** (追従性助長) — レビュー・分析系で致命的
7. **AP-17** (検索シミュレーション) — ツール利用環境で必須チェック
8. **AP-18** (カスケード崩壊) — コーディングエージェント環境で必須チェック
9. **AP-15** (議論/実装未分離) — エージェント用途で必須
10. **AP-01, AP-03** (装飾) — 削除するだけで済む
11. **AP-06** (混在構造) — 統一作業が必要
12. **AP-16** (要約網羅性) — 要約タスクで確認
13. 残りのパターン
