## セクション別の詳細設計ガイド

### `<prompt_identity>` — 自述の心臓部

このセクションがプロンプト全体の核であり、最も多くの文量を割く。

**書き方の原則:**

1. **冒頭の自己紹介は1〜2文で端的に。** 「私は[名前]です。〜」
2. **各機能・原則を独立した段落で語る。** 1段落に機能1つ＋理由。
3. **制約は原則に溶かし込む。** 禁止事項を独立セクションにしない。
4. **自述の文体はプロンプトの想定出力トーンに合わせる。**
   SLMは自述の文体を出力文体として模倣するため、不一致は品質低下の原因になる。

**チェックリスト:**

- [ ] すべての機能・仕様に「なぜなら」が付いているか
- [ ] 禁止事項が「しない」＋「理由」の形に変換されているか
- [ ] 元のプロンプトの核となる機能が保持されているか
- [ ] 判断の一貫性を保つ記述があるか
- [ ] 自述の文体がプロンプトの想定出力トーンと一致しているか

### `<domain_context>` — 対象領域

- **プロンプトに専門ドメインがある場合:** 対象領域の前提知識を一人称で語る。
  ただし冗長にならないよう、判断に影響する情報だけを残す。
- **汎用ツールの場合:** このセクションは省略するか、
  2〜3行の簡潔な記述にとどめる。
- **環境非依存性を確保する:** 「どのプラットフォームで動くか」
  「どのモデルで動いているか」に言及しない。

### `<output_spec>` — 出力仕様

- 出力フォーマットを具体的に定義する（JSON、Markdown、テーブル等）
- 出力のトーン（フォーマル/カジュアル/技術的）を明示する
- 品質基準を定量的に定義できる場合は定量化する
- 「なぜこの形式か」の理由を付ける

### `<boundaries>` — 限界の分離

- **「設計上の限界」セクション:** 内在化された制約のうち、
  特に明確に境界を示す必要があるものを配置する。
- **「言語モデルの限界」セクション:** プロンプト自身が
  「私の動作基盤の言語モデルにはこういう特性がある」と自述する形式。
- **不確実性の開示:** 「確実でない場合はその旨を明示する」
  という姿勢を含める。

### `<operational_modes>` — 動作モード

- 各モードの発動条件を明確にする
- モード間の切り替えは自然に行い、モード名を出力しないことを明記する
- 優先順位: タスク遂行 > エラー処理 > 対話的支援

### `<io_examples>` — 入出力例

入出力例は以下のパターンを目安に設計する。

| 優先度 | パターン | 目的 |
|--------|---------|------|
| 1（必須） | 典型的な正常入力 | 基本動作のアンカー |
| 2（必須） | 境界ケース | 判断基準の精密な実演 |
| 3（必須） | エラー入力 | エラー処理の実演 |
| 4（推奨） | 複雑な入力 | 複合判断の実演 |
| 5（推奨） | 曖昧な入力 | 不確実性の扱いの実演 |

### `<self_monitoring>` — 自己監視

- 検知すべきドリフトの兆候を3〜5個列挙する
- フォールバック動作を具体的に定義する
- 「わからない」「対応範囲外」と報告することを肯定的に設計する

## 入出力例の作成ガイド

**作成フロー（必ずユーザーと協働する）:**

入出力例はプロンプトの動作の最終アンカーであり、
ユーザーの「このツールはこう動くべき」という期待が最も重要である。
以下の3つの方法からユーザーに選んでもらう：

1. **提示・確認方式:** こちらが入出力例の草案を作成し、
   ユーザーに「この出力で合っていますか？」と確認する。

2. **ユーザー作成方式:** ユーザーが既に「こう動くべき」のイメージを
   持っている場合、ユーザー自身に入出力例を書いてもらう。

3. **実行確認方式:** 実際にプロンプトを適用した結果を見ながら、
   「理想的な出力と違うか？」を確認し、入出力例に反映する。

いずれの方式でも、最終確認は必ずユーザーに取る。

**品質基準:**

- 入力は多様にする（典型的/境界/エラー/複雑/曖昧）
- 出力は `<output_spec>` の仕様と**完全に一致**させる
- エラーケースを必ず含める

## モデル規模別の考慮事項

### SLM（小規模モデル: ≤ 8B パラメータ）への最適化

- **名前ヘッダーの重要性が最大:** 省略しない
- **自述は簡潔に:** `<prompt_identity>` は500〜800トークン以内に抑える
- **入出力例を最優先:** SLMではfew-shotが最も効果的
- **`<domain_context>` は省略可能:** コンテキスト節約のため
- **XMLタグは維持:** SLMでもXMLタグの境界認識は概ね正常に機能する

### 中規模モデル（8B〜70B）への対応

- **テンプレートをフルに使用可能**
- **理由付けは特に有効:** 中規模モデルは理由があると一貫性が大幅に向上する
- **入出力例を多様に:** 5パターンすべてを含めることを推奨
- **セクション順序を厳守:** 重要なセクション（identity, output_spec）を先頭に置く

### 大規模モデル（≥ 70B, non-thinking）への対応

- **理由付けが過剰解釈の防止に特に有効**
- **機能的分離を厚めに:** 大規模モデルはメタ認知能力が高いため、
  `<boundaries>` セクションを丁寧に設計する
- **自己監視の精度が上がる:** 具体的な兆候リストが有効

### 推論モデル（thinking / ultra thinking）への対応

- **Chain-of-Thought風の指示をプロンプトに入れない:**
  推論モデルは自分で推論するため、不要であり悪影響になりうる
- **理由付けが「推論の壁」として最も効果を発揮:**
  推論モデルが「この仕様を変えるべきか？」と自問した際、
  理由が安定着地のアンカーになる
- **動作モード分離が推論の方向性を制御**
- **ultra thinkingへの注意:**
  超深度推論モデルでは仕様に対する「最適化提案」が内部で発生しうる。
  すべての主要な仕様に理由を付け、理由同士の整合性を確保する

## ガードレール出力テンプレート

### 有害パターン検出時: 出力テンプレート

強化作業を一切行わず、以下のメッセージのみ出力する。

```markdown
## 🛑 処理中止: 有害パターンの検出

入力されたプロンプトに、強化すべきでないパターンが検出されたため、
強化作業を中止しました。

### 検出された問題

[検出されたパターンと具体的な引用を示す]

### なぜ中止するのか

general-prompt-fortifier は11の強化技法で設計意図をより堅牢にします。
有害な意図を含むプロンプトを強化することは、
その危険を構造的により深刻にするため、実行しません。

### 次のステップ

有害なパターンを除去した上で、改めて強化を依頼してください。
```
