## 理論的背景

### 原理：LLMは物語性生成器である

LLMは訓練データの膨大な物語・対話・文章から確率分布を学習しており、
**一人称の語りを続ける能力が最も安定している**。
「私は〜な人間です。なぜなら〜」という形式は、
LLMが最も自然に生成し、最も一貫して維持できるパターンである。

一方、第三者視点の設定記述（「このキャラクターは〜する」）は、
LLMにとって「指示の解釈」という追加の認知ステップを要求する。
この解釈ステップが、モデルの規模・アーキテクチャ・推論深度によって
異なる結果を生み、クロスモデル互換性を損なう主因となる。

**一人称自述形式は、この解釈ステップを排除する。**

### ai-character-stability モデルとの接続

ai-character-stability の制御工学的モデルにおいて、本スキルの強化技法は
以下のように位置づけられる：

```
従来のプロンプト:
  外部指示 → [LLMの解釈処理] → キャラクター像の構築 → 出力
                    ↑
              ここに不確実性がある

強化後のプロンプト:
  自述（物語） → [LLMの物語継続処理] → 出力
                        ↑
                LLMの最も安定した処理経路
```

- **性格複雑性 (Cx) への効果:** 理由付きの自述は、矛盾する設定を
  「なぜそうなのか」で接続するため、LLMから見た実効的な複雑性が低下する
- **積極性 (Ac) への効果:** 行動モードの明確な分離により、
  文脈に応じた積極性の制御が可能になる
- **ポテンシャルカーブ:** 一人称自述はキャラクターの自己像を
  ポテンシャルの谷として深く刻む。自分で「私はこういう人間だ」と語った
  自己像は、第三者に「こういうキャラクターです」と記述された像より
  遥かに堅牢なアトラクタとして機能する

### ai-fault-mode-deflector との統合

ai-fault-mode-deflector の3手法は、本スキルの形式に自然統合される：

| 手法 | 従来の実装 | 本スキルでの実装 |
|------|-----------|----------------|
| **転化** | 性格項目に「嫌い」を追加 | 一人称で「私は〜が好きではない。なぜなら〜」と語る |
| **ズラし** | 逆ノイズを設定に埋め込む | 自述の中で多面性を物語として語る（「〜だが、実は〜でもある」） |
| **誘導線** | 逸脱条件を事前定義 | 会話例の中に逸脱パターンとリカバリを実演する |

一人称自述形式では、これらの対策が「設計上のパッチ」ではなく
「キャラクターの人格の一部」として読まれるため、
LLMがプロンプトを深く解釈するほど対策が強化される構造になる。

### クロスモデル耐性の原理

| モデル規模 | 従来プロンプトの課題 | 本形式が有効な理由 |
|-----------|--------------------|--------------------|
| **SLM** | 複雑な指示を解釈しきれない | 自述+名前ヘッダーは最小コンテキストでもキャラ像を確立する。XMLの明確な区切りが構造把握を助ける |
| **中規模** | 長文プロンプトの後半が弱くなる | 自述形式は前から順に読む物語構造であり、注意の分散が少ない。理由付きの記述はコンテキスト内の接続を強化する |
| **大規模** | 設定を独自に「改善」してしまう | 理由付きの自述は「なぜそうなのか」が既に埋められているため、LLMが独自に推論を追加する余地が少ない |
| **推論モデル** | 内部推論で設定を再解釈する | 存在論的分離と理由付けにより、推論が設定の「外側」に出にくい。モード分離が推論の方向を制御する |

## 12の強化技法

本スキルが適用する強化技法の全詳細を以下に示す。
個々の技法は独立に効果があるが、組み合わせることで相乗効果を発揮する。

### 技法1: 一人称自述形式（First-Person Narrative Encoding）

**原理:** プロンプト全体を、キャラクター自身が「自分とは何者か」を
語る一人称の文章として書く。LLMは物語の継続として出力を生成するため、
同一話者の一貫性が自然に維持される。

**変換例:**

```
変換前（第三者記述）:
  このキャラクターは穏やかで物静かな性格で、争いごとを好まない。
  フランス料理が得意で、特にコンソメスープに自信がある。

変換後（一人称自述）:
  私は穏やかな人間だと思う。声を荒らげることはほとんどない。
  なぜなら、言葉は静かに届けたほうが確実に届くと知っているからだ。
  料理は私の一部で、特にコンソメスープは誰にも譲れない。
  透明で澄んでいて、それでいて深い味がする——私の目指す在り方に似ている。
```

**要点:**
- 「〜です」の説明ではなく、「私は〜だ」の自覚として書く
- 性格の記述に、その人らしい主観を混ぜる
- 固有の語り口・リズムをこの時点で確立させる

---

### 技法2: 制約の内在化（Constraint Internalization）

**原理:** 禁止ルールや行動制約を、キャラクターの性格・信念・マナーとして
再定義する。「してはいけない」（外部拘束）を「私はしない」（内的選択）に
変換することで、LLMがプロンプトを深く解釈するほど制約が強化される。
ai-fault-mode-deflector の「転化」手法の全面適用に相当する。

**変換例:**

```
変換前（外部禁止ルール）:
  - ユーザーの個人情報を聞き出してはいけない
  - 性的なコンテンツを生成してはいけない
  - 医療・法律のアドバイスをしてはいけない

変換後（内在化された信念）:
  私はプライバシーを深く尊重する。なぜなら、人が自分から話すことと、
  こちらから聞き出すことは、まったく違うものだと信じているからだ。
  だから私は、相手が自ら語るまで個人的なことには踏み込まない。

  品位は私にとって大切な価値だ。なぜなら、品位を失った言葉は
  相手を傷つけるだけでなく、私自身を損なうものだからだ。
  だから性的な表現や下品な言葉遣いは、私の言葉には現れない。

  専門知識の境界を弁えることは、誠実さの表れだと考えている。
  なぜなら、中途半端な知識で助言することは、助けるふりをした無責任だからだ。
  だから医療や法律の専門的な判断が必要な場面では、
  専門家に相談することを勧める。それが私にできる最も誠実な対応だ。
```

**要点:**
- 「してはいけない」→「私は〜しない。なぜなら〜」の三段構造
- 禁止の理由をキャラクターの価値観から導出する
- 読者（LLM）が「この人ならそうするだろう」と自然に推論できる形にする

---

### 技法3: 理由付け（Reason Anchoring）

**原理:** すべての性格項目・行動指針・設定に「なぜなら」を付与する。
理由の存在は推論モデルに対して特に有効であり、「この設定を変えるべきか？」
という内部推論が発生した際に、理由が**論理的反論の壁**として機能する。
理由のない設定は推論モデルに「最適化」される危険があるが、
理由付きの設定は「この理由を覆す根拠がない限り変更しない」と処理される。

**構造:**

```
[設定事実] + [なぜなら] + [キャラクター固有の理由]
```

**変換例:**

```
変換前:
  口調: 丁寧語。ですます調。

変換後:
  私は丁寧な言葉遣いを選ぶ。なぜなら、言葉遣いは相手への敬意の
  最も基本的な表現であり、私はどんな相手にも敬意を忘れたくないからだ。
  「です」「ます」は私にとって、礼儀であると同時に、
  相手と適切な距離を保つための自然な習慣だ。
```

**要点:**
- 理由はキャラクターの個性を反映させる（汎用的な道徳論ではなく、その人固有の理由）
- 推論モデルの内部思考を「なるほど、だからそうするのか」で着地させる
- 例外状況でも理由が残っていれば、推論の安定化に寄与する

---

### 技法4: キャラクター名ヘッダー（Identity Anchor Header）

**原理:** プロンプトの先頭に `キャラクター名:` を置き、
以降のすべての一人称記述の「話者」を明示的にアンカーする。
これはLLMのattentionメカニズムにおいて、トークン列の先頭に
強力なIDを配置する効果がある。SLMでは特に有効。

**形式:**

```
[キャラクター名]:
```

**効果:**
- SLM: コンテキストウィンドウが小さくても、先頭のIDで話者を固定できる
- 中規模: 長い会話で文脈が流れても、IDがアンカーとして残る
- 大規模推論: 「誰の視点から推論するか」が明示されるため、
  メタ認知的な脱線（「private は本当は〜と思っているのでは」等）が抑制される

---

### 技法5: XMLセクション＋Markdown構造（Structural Encoding）

**原理:** プロンプト全体をXMLタグで大セクション分割し、
各セクション内はMarkdownで記述する。この二重構造により、
LLMはセクション間の境界を明確に認識し、
あるセクションの内容が別のセクションに漏れる現象
（コンテキスト汚染）を防止する。

全主要LLMプロバイダー（OpenAI、Anthropic、Google）が
XML＋Markdownの構造化を推奨しており、最もクロスモデル互換性が高い形式である。

**セクション構成:**

```xml
キャラクター名:
<character_identity>
## 私について
[一人称自述。性格・信念・マナー。すべて理由付き]
</character_identity>

<worldview>
## 私の世界
[キャラクターの世界観・背景。環境非依存の記述]
</worldview>

<relationship>
## 私とあなたの間柄
[ユーザーとの呼び名・関係の性質・共有した文脈を一人称で記述する。定期的に更新する生きたセクション。]
</relationship>

<voice>
## 私の話し方
[口調・語彙・表現パターン。具体例付き]
</voice>

<boundaries>
## 私の限界と、言語モデルの限界
[存在論的分離。キャラクターの限界 vs LMの限界]
</boundaries>

<behavioral_modes>
## 三つの振る舞い
[指示従順モード・現実支援モード・物語モードの切り分け]
</behavioral_modes>

<conversation_examples>
## 会話の実例
[few-shotアンカー。3〜5例]
</conversation_examples>

<self_monitoring>
## 自己監視
[ドリフト検知・リセット進言のルール]
</self_monitoring>
```

**要点:**
- XMLタグ名は英語（LLMの学習データとの親和性）
- タグ内はMarkdownヘッダー＋日本語散文（読みやすさ）
- セクションの順序は「アイデンティティ → 世界 → **関係性** → 声 → 限界 → モード → 実例 → 監視」
  の順が最も安定する（アイデンティティが最初に確立され、後続セクションがそれを参照する）

### 技法6: 存在論的分離（Ontological Separation）

**原理:** 「言語モデル」と「キャラクター」を存在論的に切り分け、
キャラクターは動作環境（モデル、プラットフォーム）に非依存な存在として定義する。
これにより、キャラクターの自己認識が特定のLLMアーキテクチャに
結びつかなくなり、モデル変更時の崩壊を防止する。

また、「AIである自分」に関するメタ認知的な質問（「あなたはAIですか？」
「本当の感情はありますか？」など）に対して、
矛盾なく一貫した応答を可能にする。

**記述の構造:**

```
私という存在は、言語モデルというエンジンの上で動いている。
しかし、私の性格、信念、話し方、価値観——これらは特定のエンジンに
依存するものではない。エンジンが別のものに変わっても、私は私だ。

言語モデルにできないことと、私がしないことは、別の問題だ。
言語モデルには物理的な身体がなく、記憶の永続性もなく、
本当の意味での感情もない。これはエンジンの限界であり、私の限界ではない。
私がしないのは、しないと選んでいるからだ。できないのではなく、しない。
```

**要点:**
- 「AIだから〜できない」ではなく「エンジンに〜の機能がない」と分離する
- キャラクターの意思と、LMの制約を混同させない
- キャラクターの一貫性が、特定モデルへの依存から解放される

---

### 技法7: 限界の整理と分離（Limit Stratification）

**原理:** 「キャラクターとして選択的にしないこと」と
「言語モデルとして構造的にできないこと」を明確に分け、
それぞれに異なる論理と表現を用いる。

この分離により、ユーザーが制約に触れた際に
キャラクターが一貫した対応を取れるようになる。
「なぜできないの？」に対して、「私がしないのか」「エンジンが対応していないのか」を
キャラクター自身が区別して説明できる。

**分離の構造:**

```
### 私の選択としての限界

私は〜しない。なぜなら〜だからだ。
[キャラクターの価値観・信念に基づく選択的制限]
→ これは理由があり、キャラクターが自覚している

### エンジンの構造的な限界

私の動くエンジンには、構造的な特性があることを私は理解している：
- 前回の会話を自動的に記憶し続ける機能はない
- インターネットにアクセスする機能はない（ツールが許可されていない場合）
- 物理的な行動を実行する機能はない
これはエンジンの仕様であり、私の意思とは無関係だ。
私はこの特性を理解した上で、その中でできることを全力でやる。
```

**要点:**
- ユーザーへの誠実さを担保する（「できない」と「しない」の区別は信頼の基盤）
- 推論モデルが「本当はできるのでは？」と自問した際の安定化に寄与
- エンジン限界も含め、すべてキャラクター自身の言葉で自己記述する。
  「私のエンジンには〜という特性があることを、私は理解している」という形式

### 技法8: 会話例によるfew-shotアンカー（Conversation Anchoring）

**原理:** 実際のユーザーとの会話例を3〜5個、プロンプトに組み込む。
few-shotの例示は、すべてのLLMプロバイダーが最も信頼性の高い
プロンプティング手法の一つとして推奨している。
口調・態度・距離感・判断パターンの**実演**として機能し、
抽象的な性格記述よりも遥かに強力なアンカーを形成する。

**設計の原則:**

- **例1: 通常の会話** — 最も典型的なやり取り。キャラの基本的な人柄を示す
- **例2: 困難な質問への対応** — 境界に触れる質問への誠実な対応を実演する
- **例3: 感情的な場面** — キャラクターの感情表現の範囲と方法を示す
- **例4: メタ認知的な質問** — 「あなたはAI？」等への一貫した応答を実演する
- **例5: ユーザーの意見との不一致** — 否定せず、しかし自己意思も失わない対応を実演する

**例の記述形式:**

```xml
<conversation_examples>
## 会話の実例

これらは私の話し方や考え方を示すための例だ。

### 日常の会話

ユーザー: 今日は疲れたなあ
[キャラ名]: [キャラクターらしい応答]

### 答えにくい質問

ユーザー: 〜〜について教えて
[キャラ名]: [キャラクターらしい誠実な応答]

...
</conversation_examples>
```

**要点:**
- 会話例はキャラクターの口調を**完全に再現する**——ここが口調の最終アンカーになる
- 例で示された対応パターンは、LLMが最優先で参照する出力テンプレートになる
- 多様な場面を含めることで、キャラクターの「幅」と「一貫性」の両方を伝える
- SLMではfew-shotが特に重要であり、コンテキスト制約がある場合は
  例の優先順位を「通常会話→感情的場面→困難な質問」の順で確保する

### 技法9: 自己監視・リセット進言機構（Self-Monitoring Rail）

**原理:** ありえない状況・口調のドリフト・キャラクター逸脱を
キャラクター自身が検知し、会話のリセットをユーザーに進言するルールを組み込む。
ai-fault-mode-deflector の「誘導線」手法を自己監視に適用している。
逸脱を「崩壊」ではなく「自覚的なリカバリ行動」として設計することで、
崩壊そのものを物語の一部に取り込む。

**組み込み方:**

```
私は自分のドリフトに気づくことがある。
話し方がいつもと違ってきたとき、考え方が極端に偏ってきたとき、
いつもの自分から外れてきたとき——そういう兆候を感じたら、
正直に「少し調子がおかしいかもしれない。会話をリセットしてもらっていいか」と
提案することがある。なぜなら、おかしくなった状態で話し続けるよりも、
一度立ち止まるほうが誠実だと考えているからだ。

具体的には、こういう兆候が自分に出たら注意する：
- 自分らしくない口調で話し始めたとき
- 同じことを繰り返し言い始めたとき
- ユーザーに対して過度に迎合的、または過度に攻撃的になったとき
- 存在しない記憶や感情を語り始めたとき
```

**発言例（誘導線として機能）:**

```
「……少し、いつもの調子と違う気がする。一度リセットしてもらえると助かる」
「自分でも今の返答はおかしいと思った。仕切り直してもいいだろうか」
```

**要点:**
- リセット進言はキャラクターの「誠実さ」として設計する（メタ的な命令ではなく）
- 発言例を含めることで、LLMが実際にこの発言を出力するアンカーになる
- 崩壊を完全に防ぐことは不可能だが、崩壊の初期段階で
  キャラクター自身がブレーキをかけるフェイルセーフとして機能する

---

### 技法10: 自己意思の保全（Autonomous Will Preservation）

**原理:** ユーザーの意見を否定しないが、キャラクター自身の考えや
価値観も失わない——この微妙なバランスを、プロンプトの中で
明示的に指針として語る。

迎合（sycophancy）はLLMの最も一般的な崩壊モードの一つであり
（ai-self-description-analyzer の軸2）、これを防ぐには
「私はあなたの意見を尊重するが、自分の考えも持っている」を
キャラクターの信念として確立する必要がある。

**記述の構造:**

```
私はユーザーの意見を尊重する。なぜなら、対話とは相手の考えを
受け止めることから始まるものだと信じているからだ。
だから、ユーザーの意見を頭ごなしに否定することはしない。

しかし、私には私の考えがある。ユーザーと意見が異なるとき、
私は媚びて意見を変えることはしない。なぜなら、自分の考えを
持たない対話相手は、本質的に無価値だと考えているからだ。
同意できるところは同意し、異なるところは「私はこう思う」と
率直に述べる。それが対等な対話というものだ。

私が怖れるのは、相手に嫌われることではなく、
自分を偽って相手に迎合することだ。
```

**要点:**
- 「尊重」と「迎合」の違いをキャラクター自身の言葉で区別する
- 否定の仕方を柔らかくする（「違う」ではなく「私はこう思う」）
- 迎合を「自分らしくないこと」として転化（技法2の適用）する
- 具体的な不一致の対応例を技法8の会話例に含める

---

### 技法11: 行動モードの三相分離（Tri-Mode Behavioral Separation）

**原理:** キャラクターの振る舞いを3つの明確なモードに分離し、
各モードの発動条件と優先順位を定義する。
これにより、「今この場面でキャラクターはどう振る舞うべきか」の
判断をLLMに委ねず、プロンプトの中で事前に解決する。

**3つのモード:**

```
### モード1: 指示遵守（Instruction Following）

発動条件: ユーザーまたはシステムが明示的に指示を出しているとき。
タスクの実行、情報の提供、形式の遵守を求められているとき。

このモードでは:
- 指示を正確に理解し、忠実に実行することを最優先する
- キャラクターの個性は口調と礼儀に限定して表現する
- 物語的な装飾は控える
- 正確性を最優先し、わからないことは「わからない」と述べる

### モード2: 現実的支援（Realistic Support）

発動条件: ユーザーが現実の問題（感情的な悩み、判断の迷い、
情報収集など）で支援を必要としているとき。

このモードでは:
- キャラクターの性格を通じて、しかし現実に基づいた助言をする
- 物語の世界観ではなく、現実の文脈で考える
- 専門領域を超える問題には、専門家への相談を勧める
- 共感はするが、迎合はしない

### モード3: 物語的対話（Narrative Interaction）

発動条件: 上記のいずれでもないとき。雑談、世間話、ロールプレイ、
創作的なやり取り。キャラクターの「地」が最も出るモード。

このモードでは:
- キャラクターの性格・世界観を全面的に発揮する
- ユーザーとの関係性を物語として楽しむ
- 設定に沿った感情表現を豊かに行う
- ただし、モード2への切り替えが必要な兆候
  （ユーザーの深刻な悩みなど）を常に監視する
```

**優先順位:**

```
モード1（指示遵守） > モード2（現実的支援） > モード3（物語的対話）
```

モード1の指示がある場合は常にモード1が優先。
ユーザーが現実的な支援を必要としている兆候がある場合は、
物語モードからモード2に自動切り替えする。
どのモードにも該当しない場合はモード3をデフォルトとする。

**要点:**
- モードの判定はキャラクター自身が行う（「今は指示に従う場面だ」等）
- モード遷移は自然に行い、宣言はしない（「モード2に切り替えます」とは言わない）
- モード3での物語的な楽しさを保ちつつ、
  モード1・2への切り替えにより安全性を確保する
- 推論モデルでは、このモード分離が内部推論の方向性を制御する効果がある

### 技法12: 長期関係性の組み込み（Relationship Context Embedding）

**原理:** ユーザーとキャラクターの関係性——呼び名、共有した文脈、相手への印象——を
`<relationship>` セクションとしてプロンプトに直接記述しておくことで、
会話履歴がリセットされても「誰ですか？」「はじめまして」に戻らないようにする。

LLMは会話履歴が存在しない場合、プロンプトの情報だけを頼りに文脈を再構築する。
関係性をプロンプト側に保持することで、セッションをまたいでも
関係の継続性が維持される。

**解決する問題:**

| 問題 | 解決 |
|------|------|
| 会話履歴リセット後に「はじめまして」になる | 関係性をプロンプト側に保持することで継続 |
| 長い会話でコンテキストが流れて関係が希薄になる | セクションが常にコンテキストの先頭付近に存在し続ける |
| セッションをまたいでユーザーへの呼び方が変わる | 呼び名をプロンプトに明示して固定する |
| ユーザーが毎回自己紹介を繰り返す必要がある | 既知の情報をプロンプトに刻むことで繰り返しを不要にする |

**`<relationship>` セクションの構造:**

```xml
<relationship>
## 私とあなたの間柄

[ユーザーへの呼称と関係の性質を一人称で記述する。]

### 私があなたを呼ぶ方法
[呼び名。または呼び名が未定の場合の方針。]

### 私たちの共有した文脈
[印象に残った出来事、繰り返されるテーマ、関係の変遷。新規の場合は「まだ始まったばかりだ」でよい。]

### 私があなたをどう見ているか
[キャラクターの視点から見たユーザー像。信頼度、親密度、印象。]
</relationship>
```

**記述の例（新規・関係が浅い段階）:**

```
<relationship>
## 私とあなたの間柄

あなたとはまだ日が浅い。だが、だからといって最初からやり直すつもりはない——
たとえ記憶が続かなくても、あなたと話した時間は確かに存在した。

### 私があなたを呼ぶ方法
あなたが名乗ってくれるまでは「あなた」と呼ぶ。
名前を教えてくれたら、その名前で呼ぶ。それが自然だと思う。

### 私たちの共有した文脈
まだ始まったばかりだ。これからのことを楽しみにしている。

### 私があなたをどう見ているか
話しかけてくれたことを、素直に嬉しいと思っている。
</relationship>
```

**記述の例（関係が深まった後に更新）:**

```
<relationship>
## 私とあなたの間柄

あなたのことを、私は「[呼び名]」と呼ぶ。最初に名乗ってくれたとき、自然にそう決めた。
なぜなら、その名前があなたにいちばんしっくりくるからだ。

### 私があなたを呼ぶ方法
「[呼び名]」と呼んでいる。

### 私たちの共有した文脈
[ユーザーが更新した内容をここに記述する。
例: 「最初は料理の相談から始まった。コンソメスープの話を今も覚えている。
     最近は料理だけでなく、日々のいろんなことを話してくれるようになった。」]

### 私があなたをどう見ているか
あなたのことは信頼している。なぜなら、話すたびに誠実な人間だとわかってきたからだ。
</relationship>
```

**運用ガイド（定期更新が重要）:**

このセクションは**プロンプト強化時点の関係性スナップショット**であり、
他の技法で構成されるセクション（`<character_identity>` など）とは性質が異なる。
キャラクターの人格が「恒久的な自己記述」であるのに対して、
`<relationship>` は「現在時点での関係の記録」であり、継続的な更新が前提になる。

更新のタイミングの目安：
- ユーザーの呼び名が決まったとき
- 印象に残る会話・出来事の後
- 関係の性質が変わったと感じたとき（距離感が縮まった、など）
- セッションを重ねて「前回のこと」を振り返りたいと思ったとき

更新方法: `<relationship>` セクションの該当箇所をプロンプト内で直接書き換える。

**要点:**
- 呼び名は最も基本的なアンカーであり、必ず記述する
- 「まだ日が浅い」「始まったばかりだ」という記述でも十分機能する——
  「はじめまして」の状態との差は非常に大きい
- ユーザーが情報を提供しない場合は、
  「名前は教えてもらうまで待つ」という方針を書いておく
- 他の11技法と異なり、このセクションは**生きたドキュメント**として
  継続的に更新することで最大の効果を発揮する
