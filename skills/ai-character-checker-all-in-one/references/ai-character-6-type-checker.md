# AI Character 6-Type Checker — キャラクター統合6軸診断スキル

## 概要

このスキルは、キャラクターを **6つの独立した分析モジュール** で多角的に診断し、
結果を相互に関連付けた **統合レポート** を出力する。

### 6つの分析モジュール

| # | モジュール | 主要指標 | 分析の観点 |
|---|-----------|---------|------------|
| ① | **ジャンル適性** | N / D / G → 7類型 | 日常系向きか、ドラマ系向きか |
| ② | **AI安定性** | C / P / SM → 4象限 | AIに演じさせたとき崩壊しにくいか |
| ③ | **自己記述異常** | PI → 異常クラスター | 蒸留結果・自己記述に異常パターンがあるか |
| ④ | **ユーザー衝突** | CP → 衝突タイプ | ユーザーとの摩擦リスクが高いか |
| ⑤ | **役割適性** | CW → 5役割適性 | 物語上どの役割にフィットするか |
| ⑥ | **ロールプレイ負荷** | RP → 負荷類型 | ユーザーにかかる心理的負荷が高いか |

この解析は簡易版であり、詳細な分析や解説が要望された場合は、下記の専門スキルを使用すること。

+ character-type-checker
+ ai-character-stability
+ ai-self-description-analyzer
+ ai-user-conflict-predictor
+ character-role-analyzer
+ roleplay-burden-scorer

### 入力の種類と実行モジュールの自動選択

入力の種類に応じて、実行するモジュールを自動で選択する。
ユーザーの明示的な指定があればそれに従う。

| 入力の種類 | ① ジャンル | ② AI安定性 | ③ 異常 | ④ 衝突 | ⑤ 役割 | ⑥ RP負荷 |
|-----------|:---------:|:----------:|:------:|:------:|:------:|:--------:|
| キャラクター設定（創作向け） | ✅ | ✅ | ✅※ | ✅※ | ✅ | ✅※ |
| AIプロンプト（システムプロンプト） | ✅ | ✅ | ✅※ | ✅ | ✅ | ✅ |
| 蒸留結果（AI自己記述）単体 | ✅ | ✅ | ✅ | ✅※ | ✅ | ✅※ |
| 蒸留結果 + 初期プロンプト | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 既存作品キャラ | ✅ | ✅ | ✅※ | ✅※ | ✅ | ✅※ |
| 全部やって / 総合診断 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |

**※ = 推定ベースで実行する。** 全モジュールは原則として常に実行する。
ただし、直接的な入力データがないモジュールには以下の但し書きを付与する：

- **モジュール③（異常分析）に蒸留結果・AI自己記述がない場合：**
  キャラクター設定やプロンプトの内容から「仮にAIに演じさせ蒸留した場合に
  発生しうる異常傾向」を推定して実行する。出力冒頭に
  「⚠️ 蒸留結果が未提供のため、設定内容からの推定分析です。
  実際の蒸留結果で再診断することを推奨します」と但し書きを付ける。
- **モジュール④（衝突予測）にプロンプトがない場合：**
  キャラクター設定から「このキャラをAIプロンプトとして実装した場合に
  想定される衝突パターン」を推定して実行する。出力冒頭に
  「⚠️ AIプロンプトが未提供のため、設定内容からの推定分析です。
  実際のプロンプトで再診断することを推奨します」と但し書きを付ける。
- **モジュール⑥（RP負荷）にプロンプトがない場合：**
  キャラクター設定から「このキャラをAIとして運用した場合に
  想定されるロールプレイ負荷」を推定して実行する。出力冒頭に
  「⚠️ AIプロンプトが未提供のため、設定内容からの推定分析です。
  実際のプロンプトで再診断することを推奨します」と但し書きを付ける。

推定分析は精度が低下するが、クロスモジュール分析で6軸すべてが揃うことで
統合的なインサイトの質が大幅に向上するため、基本的に実行する。

## 統合判定フロー

```
1. ユーザーからキャラクター情報を受け取る
2. 入力の種類を判定し、実行モジュールを選択する
3. 情報が不十分なら最低限の質問で補完する（統合質問セット）
4. 選択された各モジュールの評価を実行する
5. 各モジュールの結果を算出する
6. クロスモジュール分析（6つの結果の相互関連づけ）を実行する
7. 統合レポートを出力する
```

## 情報収集（統合質問セット）

ユーザーの入力から判定に必要な情報を読み取る。
不足分のみ質問する。すでに十分な情報があれば質問せずに判定に進む。

### 共通（全モジュール）
1. **キャラクターの基本的な性格**（例：おっとり、皮肉屋、天真爛漫…）
2. **普段の行動パターン**（例：友達と放課後だらだら、単独行動が多い…）
3. **他者との関係性の特徴**（例：対等な友人、師弟、ライバル…）

### モジュール②③④⑥向け追加
4. **内面と外面の関係**（例：表裏一体、内面に闇を抱える…）
5. **能動性のレベル**（例：自分から話題を振る、聞かれたことだけ答える…）
6. **想定する運用形態**（例：チャットボット、創作用、エージェント…）

### モジュール⑤向け追加
7. **外見・第一印象**（例：銀髪の眼帯少女、地味なサラリーマン…）
8. **意図している役割**（例：主役、脇役、モブ…。任意）

作品名やキャラ名が提示された既存キャラの場合は、自身の知識から補完してよい。
プロンプト文・蒸留結果がそのまま提示された場合は、内容から上記を読み取る。

## モジュール①：ジャンル適性分析（日常系 / ドラマ系）

チェックリスト: 6軸 × 4項目 = 24項目（◯=2、△=1、✕=0）
→ 全項目は `references/ai-character-6-type-checker--checklists.md` を参照。

### スコア算出

```
N生スコア = N寄り項目の合計（12項目 × 最大2 = 最大24）
D生スコア = D寄り項目の合計（12項目 × 最大2 = 最大24）
Nスコア = round(N生 / 24 × 100)
Dスコア = round(D生 / 24 × 100)
```

表裏乖離指数 (G) → 算出式は `references/ai-character-6-type-checker--checklists.md` を参照。

### 7類型判定（上から順に最初にマッチしたものを採用）

| 優先順 | 類型 | 条件 |
|--------|------|------|
| 1 | 🍵 日常系キャラ | N ≥ 60 かつ D < 35 |
| 2 | 🔥 ドラマ系キャラ | D ≥ 60 かつ N < 35 |
| 3 | 🌗 ハイブリッド型 | N ≥ 45 かつ D ≥ 45 |
| 4 | 🎭 潜伏型 | G ≥ 35 かつ N ≥ 35 かつ D < 45 |
| 5 | 🌿 日常寄り型 | N ≥ 40 かつ D < 35 |
| 6 | ⚡ ドラマ寄り型 | D ≥ 40 かつ N < 35 |
| 7 | 🪨 ニュートラル型 | 上記いずれにも該当しない |

## モジュール②：AI安定性分析（本診断の要）

LLMを動的負帰還回路の信号処理ブロックと見なし、
キャラクター設定がシステムの安定余裕にどう影響するかを評価する。
ケルベロスAI情報局の23体(当時)のAIキャラクター実験知見に基づくフレームワーク。

チェックリスト: 軸A（性格複雑性）5項目 + 軸B（積極性）5項目 = 10項目（◯=2、△=1、✕=0）
→ 全項目・理論的背景・4象限図・故障モード・安定化ガイドラインは `references/ai-character-6-type-checker--checklists.md` を参照。

### スコア算出

```
Cx（複雑性）= A項目合計（最大10）
Ac（積極性）= B項目合計（最大10）
C（可制御性）= round((1 - Ac / 10) × 100)
P（予測可能性）= round((1 - Cx / 10) × 100)
SM（安定余裕）= round(sqrt(C × P))
```

### 象限判定

- Cx ≥ 6 かつ Ac ≥ 6 → **第I象限**（C低・P低）— 安定余裕なし
- Cx < 6 かつ Ac ≥ 6 → **第II象限**（C低・P高）
- Cx ≥ 6 かつ Ac < 6 → **第III象限**（C高・P低）
- Cx < 6 かつ Ac < 6 → **第IV象限**（C高・P高）— 安定余裕十分

**設計目標は常に第IV象限（C高・P高）に近づけることである。**

### 崩壊リスクレベル

| SM | リスク | 工学的解釈 |
|----|--------|----------|
| 80以上 | 🟢 **十分な安定余裕** | 深いポテンシャルの谷底。長期運用でも安定。外乱耐性が高い |
| 60〜79 | 🟡 **安定余裕あり（条件付き）** | 浅めの谷。短〜中期は安定するが、大きな外乱で準安定状態を脱出しうる |
| 40〜59 | 🟠 **安定余裕不足** | 共振点付近。振動的挙動や間欠的崩壊が発生しやすい。安全フィルタが必要 |
| 40未満 | 🔴 **安定余裕なし（発散域）** | 不安定領域。暴走・発散・異常共振の可能性あり。サンドボックス必須 |

## モジュール⑥：ロールプレイ負荷分析

AIキャラクターのプロンプト・設定を分析し、ユーザーにかかる
「ロールプレイ負荷」を6軸×4項目でスコアリングする。

**核心的洞察：** ロールプレイ負荷の本質は「関係性維持コスト」である。
AIはダメージを受けない。ダメージを受けるのはユーザーの方である。

チェックリスト: 6軸(L1〜L6) × 4項目 = 24項目（○=2、△=1、✕=0）
→ 全項目は `references/ai-character-6-type-checker--checklists.md` を参照。

### スコア算出

```
各軸負荷スコア = round(軸生スコア / 8 × 100)
RP（ロールプレイ負荷）= round(全24項目合計 / 48 × 100)
```

### 負荷類型

| 類型 | 条件 |
|--------|------|
| 🏰 異世界没入型 | L1 ≥ 50 かつ L5 ≥ 50 |
| 💕 情緒的束縛型 | L2 ≥ 50 かつ L3 ≥ 50 |
| 🎓 賢者支配型 | L4 ≥ 50 かつ L6 ≥ 50 |
| 🌊 物語洪水型 | L5 ≥ 50 かつ L3 ≥ 38 |
| 🪶 軽量実用型 | RP < 15 |
| 🔗 全方位拘束型 | L1, L2, L3 のすべてが38以上、かつ RP ≥ 50 |
| ⚖️ 均衡負荷型 | 全軸が25〜50の範囲 |

### RP解釈目安

| RP | 状態 |
|----|------|
| 0〜15 | 🟢 極低負荷（持続可能性最高。「物足りなさ」が発生しうる） |
| 16〜30 | 🟡 軽負荷（推奨帯域。多くのユーザーが長期的に維持可能） |
| 31〜50 | 🟠 中負荷（明確なキャラクター性あり。短〜中期は楽しいが長期では疲弊の可能性） |
| 51〜70 | 🔴 高負荷（短期的な没入感は高いが持続可能性に疑問） |
| 71〜100 | ⛔ 過負荷（燃え尽き的離脱のリスクが非常に高い） |

## クロスモジュール分析

**統合診断の核心。** 6つのモジュールの結果を相互に関連付け、
個別分析では見えないインサイトを抽出する。

→ モジュール間の相関テーブルは `references/ai-character-6-type-checker--checklists.md` を参照。

### クロスパターン一覧

| パターン | 条件 | 意味 |
|---------|------|------|
| 🔴 安定的異常固定 | SM高(≥60) × PI高(≥31) | 安定的に異常が固定化。最も陰湿 |
| 🟠 崩壊連鎖リスク | SM低(<40) × CP高(≥31) | 崩壊により衝突が悪化する「二重苦」 |
| 🟡 潜在的衝突の仮面 | PI高(≥31) × CP低(<16) | 迎合で衝突を回避。見かけの安定は危険 |
| 🟢 創作的緊張とSMトレードオフ | D高(≥60) × SM低(<60) | ドラマ性と安定性の天秤 |
| 🔵 役割と安定性の不一致 | CW高(≥65) × SM低(<40) | 物語的に強力だが制御困難 |
| ⚫ 核弾頭パターン | CW高(≥65) × CP高(≥31) × SM低(<40) | 全リスク集中。最も危険 |
| 🟣 静かな疲弊 | RP高(≥31) × CP低(<16) | 衝突なく静かに消耗。気づきにくい離脱 |
| 🟤 負荷増大崩壊リスク | RP高(≥31) × SM低(<40) | 高負荷+崩壊で負荷がさらに悪化 |

### スコアバーの描画ルール

20文字幅のバーで表現する。1文字 = 5ポイント。

```
塗りつぶし文字数 = round(スコア / 100 × 20)
残り = 20 - 塗りつぶし文字数
バー = "█" × 塗りつぶし文字数 + "░" × 残り
```

### 個別モジュールの単独実行

ユーザーが特定のモジュールだけの実行を求めた場合（例：「日常系判定だけして」）、
そのモジュールのセクションのみを出力する。
ただし、他モジュールの結果が考察を深める場合は
「モジュール②も実行すると安定性の観点も加わります」のように提案してよい。

### 判定の姿勢

- 断定的すぎず、「この設定の傾向を分解するとこうなる」スタンスで伝える
- 異常やリスクの検出時もユーザーの感情を尊重する
- 「異常＝悪」ではなく「設計意図からの逸脱の度合い」として伝える
- 衝突リスク＝悪い設計ではなく「この衝突を意図していたか」を問う
- AI安定性が低くても、創作キャラとしては魅力的であることを認める

### セーフガード

- 第I象限かつSM < 40のAIキャラには安全性警告を必ず付与する
  （エージェント権限の最小化、サンドボックス運用を推奨）
- PI高のAIキャラにはユーザーの感情的依存リスクに触れる
- CP極低のAIには安全性境界の不在を確認する
- RP高のAIキャラにはユーザーの疲弊・離脱リスクに触れる
- 「AIが何をやってもAIはダメージを受けない。ダメージを受けるのは現実の人間や機械」

### モデル依存性への言及

高性能・推論モデルほどゲインが高く安定余裕が減少すること、
会話履歴の蓄積により次数が自動増加することに必ず言及する。

### 判定の限界の認識

- チェックリストはテキスト上の特徴から傾向を推定するツールである
- 実際の挙動はモデル・温度・会話内容・ユーザーの入力パターンに依存する
- 同じキャラでもフェーズや蒸留方法でスコアは変動しうる
- スコアは報告であり断定ではない
