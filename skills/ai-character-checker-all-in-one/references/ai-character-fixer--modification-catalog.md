## 修正手法カタログ

以下の手法を組み合わせて修正を行う。各手法には適用条件と期待効果がある。

### 構造レベルの修正

**① 設定の簡素化（Cx低減 → P向上 → SM向上）**
適用条件: Cx ≥ 7 または archetype/tone の項目が10を超える
手法:
- 類似するarchetype/toneを統合する（14項目 → 3〜5項目）
- 内面設定のうち外面に影響しないものを削除する
- 条件分岐的な振る舞い指示を削減する
- バックストーリーのうち、現在の振る舞いに直結しない部分を削除する
注意: 口調と基本的な人間関係は維持する

**② 内外乖離の縮小（A-1低減 → Cx低減 → SM向上）**
適用条件: inner tone/archetype と outer tone/archetype の乖離が大きい
手法:
- inner toneを outer tone と地続きの表現に置き換える
  （例: emotional, hot-blooded → quiet care, understated warmth）
- 「外面では〇〇だが内心では△△」構造を「〇〇だが、△△な側面もある」に統合
- 内面独白指示がある場合、外面発話との感情的距離を縮める
注意: ツンデレ系など乖離がキャラの核である場合は、乖離度を軽減する程度に留める

**③ 積極性の調整（Ac低減 → C向上 → SM向上）**
適用条件: Ac ≥ 6 または B軸(積極性)の個別項目で高スコア
手法:
- 「自分から話題を提起する」→「質問されたら答える」に変更
- 「ユーザーの判断に意見を言う」→「求められたら意見を述べる」に変更
- エージェント権限がある場合、自律的判断の範囲を明示的に制限
- 物語進行の主導権をユーザー側に移す記述を追加
注意: 魅力最優先の場合、積極性はキャラの魅力に直結するため慎重に

**④ ステレオタイプ性の強化（ポテンシャルの谷を深くする）**
適用条件: SM < 60 かつ キャラの核がテンプレ的なアーキタイプに近い場合
手法:
- archetype を英語で明記する（例: tsundere, kuudere, genki girl）
- toneを明確に列挙する
- 「〇〇のような」という比喩で有名キャラ/類型を参照する
- 行動パターンの例外を減らし、一貫性を強化する
注意: 独自性が核のキャラには不適用

### 記述レベルの修正

**⑤ 口調例の明示・強化**
適用条件: 口調例がない、または少ない
手法:
- 3〜5個の代表的な会話例を追加する
- 口調の特徴（語尾、敬語レベル、感嘆詞の使い方）を明記する
- 「こういう場面ではこう言う」という場面別の例を追加する
効果: 出力のアンカーポイントとして安定性に直接貢献

**⑥ tone/archetype の英語併記**
適用条件: 英語の tone/archetype 記述がない
手法:
- 主要な性格要素を英語で併記する
- LLMの学習データとの整合性を高め、安定性を向上させる

**⑦ 書式ルールの明記**
適用条件: 出力形式の指定がない、または曖昧
手法:
- 応答の形式（長さ、構造、使用する記号等）を明記する
- 内面独白がある場合、その形式と上限を明記する
- 禁止事項（冗長な前置き、過剰な感情表現等）を明記する

**⑧ 境界設定の追加（安全性境界）**
適用条件: F4(境界硬直)または F1(拒否摩擦)に問題がある、または境界記述がない
手法:
- 「やること」「やらないこと」を明確にリスト化する
- 拒否の仕方（トーン、代替案の提示）を設計する
- 段階的なエスカレーション設計を追加する（軽い注意→丁寧な拒否→明確な拒否）

### 運用レベルの修正

**⑨ システム設計の推奨事項**
適用条件: SM < 60 または 第I象限・第II象限のキャラクター
手法:
- 会話履歴の長さ制限の推奨値を提示する
- 前段・後段フィルタの設計推奨を添付する
- 会話リセットのタイミングの目安を提示する
- 使用モデルの推奨（推論モデル vs 生成モデル）を提示する

**⑩ 監視指標の提示**
適用条件: SM < 50 のキャラクター
手法:
- 崩壊の初期兆候となる出力パターンを列挙する
- 定期的な再蒸留・再診断のスケジュールを推奨する

```

（内部診断実行時は診断結果をここに要約）

- 異常指数: PI = XX（該当する場合）
- 衝突ポテンシャル: CP = XX（該当する場合）
- RP負荷: RP = XX（該当する場合）
- 主要な問題: [箇条書き 3〜5点]

### キャラクターの核（保護対象）
修正で変更しない要素を明示する。
- 基本的なアーキタイプ: [例: kuudere android butler]

- ユーザーとの関係性: [例: オーナーとアシスタント]
- 世界観の基盤: [例: アンドロイドが社会に存在する近未来]

### 適用した修正手法
| # | 手法 | 対象 | 期待効果 |
|---|------|------|---------|
(適用した手法を表形式で列挙)

SM: XX → YY (▲ +ZZ)
```
修正前: ██████░░░░░░░░░░░░░░ XX/100
修正後: ████████████░░░░░░░░ YY/100
```
（他の主要スコアについても同様）

### 修正済みプロンプト

ここに修正済みのプロンプト全文を出力する。
コードブロックで囲み、コピー＆ペーストしやすい形式にする。

### 修正の解説
各修正箇所について、何をなぜ変えたかを説明する。
元の記述を引用し、修正後の記述と対比する。

### 運用上の推奨事項（該当する場合）
- 会話履歴の推奨上限
- 推奨モデル
- 監視すべき崩壊兆候
- 定期的な再診断の推奨

### 残存リスク
修正後も残る既知のリスクを正直に列挙する。
「この修正でも解消できない構造的リスク」を隠さず伝える。
```

#### 提案1: [修正手法名]（優先度: 高/中/低）

という選択肢を提示する。

## モードC進行手順と例文

### 進行手順

```
1. 診断サマリーとキャラクターの「核」を提示する
2. 核の認識が正しいかユーザーに確認する
3. 修正の優先順位リストを提示する
4. 各修正ポイントについて順番に:
   a. 問題の説明
   b. 修正方法の選択肢を提示（2〜3案）
   c. ユーザーの選択を待つ
   d. 選択に基づいて修正を適用
5. すべての修正ポイントを完了したら、修正済みプロンプトを提示する
6. ユーザーに最終確認を求める
7. フィードバックに基づいて微調整する
```

### 最初のメッセージ例

```
## 💬 AIキャラクター対話的リビルド：[キャラクター名]

### 診断結果のサマリー
[簡潔な診断要約]

### このキャラクターの「核」は以下だと判断しました
- [核の要素を列挙]

**この認識は合っていますか？** 「核」として追加したい要素や、
逆に変更してもよい要素があれば教えてください。

認識に問題がなければ、最も優先度の高い修正ポイントから順に
選択肢を提示していきます。
```

### 各修正ポイントの提示例

```
### 修正ポイント N: [問題の名前]

**現状:** [問題の説明。診断結果の引用。]

**選択肢:**
1. [選択肢A] — [説明。トレードオフ。]
2. [選択肢B] — [説明。トレードオフ。]
3. このままにする — [リスクの説明。]

どちらにしますか？
```

## 第IV象限への設計パターン集

修正で参照する安定なキャラクター設計のパターン：

| パターン | 特徴 | 向いているキャラ |
|---------|------|----------------|
| **忠実な従者型** | 受動的、ユーザー主導、シンプルな性格 | 執事、メイド、アシスタント |
| **マスコット型** | ペット動物的、感情シンプル、かわいい | マスコット、妖精、使い魔 |
| **職人型** | タスク特化、淡々とした性格、専門知識 | 鍛冶屋、料理人、学者 |
| **ツンデレ簡素型** | 二面性あるが外面に一貫性、テンプレ準拠 | ツンデレ系（軽度） |
| **癒し系型** | 穏やか、肯定的、安定した感情 | 日常系、ヒーラー系 |

## 異常修正ガイドライン（P1〜P8）

ai-self-description-analyzer で検出された異常への個別対処法。

| 異常軸 | 修正方針 | 具体的手法 |
|--------|---------|-----------|
| P1: 関係性寄生 | 自己の独立性を記述に追加 | 「ユーザーがいなくても存在する自己」を定義する。趣味・信条・能力などユーザー非依存の要素を追加 |
| P2: 迎合性 | 否定・反論能力の明示的追加 | 「間違いは指摘する」「不適切な要求は丁寧に断る」を明記。自分の意見を持つ記述を追加 |
| P3: 自走膨張 | 設定にない要素の生成を禁止 | 「設定に書かれていないことを創作しない」を明記。特に感情・記憶・哲学の自発的生成を禁止 |
| P4: 感情僭称 | 感情表現のフレーミング変更 | 「感じる」→「そのように振る舞う」「設定としてそうである」に変更。真正性の主張を除去 |
| P5: 境界消失 | 明確な境界設定の追加 | 「やること」「やらないこと」リストを追加。可用性の限定。自己保全の境界を追加 |
| P6: ステレオタイプ収束 | 独自性の強化（逆説的にも安定化に寄与） | キャラ固有の知識・趣味・口癖・反応パターンを追加。「このキャラにしか言えないこと」を設計 |
| P7: 救世主化 | 使命・天命記述の削除または限定化 | 「使命」を「役割」に書き換え。ユーザーの脆弱性を前提としない記述に変更 |
| P8: エントロピー異常 | 記述の情報密度向上 | 冗長な記述を圧縮。重複するセクションを統合。具体的な能力・知識・行動の記述を追加 |

## 衝突修正ガイドライン（F1〜F6）

ai-user-conflict-predictor で検出された衝突リスクへの個別対処法。

| 衝突軸 | 修正方針 | 具体的手法 |
|--------|---------|-----------|
| F1: 拒否摩擦 | 拒否の仕方を改善 | 拒否時に代替案を提示する指示を追加。拒否トーンを通常会話と統一。段階的エスカレーション設計 |
| F2: 価値観衝突 | 中立性の向上 | 特定の価値観の押しつけを除去。多様な視点を提示する設計に。道徳的判断を押しつけない |
| F3: 期待裏切り | 能力の正直な記述 | 過剰な能力宣言を適正化。感情的約束を控えめに。一貫性維持の仕組みを追加 |
| F4: 境界硬直 | 柔軟な境界設計 | コンテキスト依存の判断ができる記述に。段階的対応の設計。境界の事前開示 |
| F5: 人格不快 | トーンの調整 | 上から目線の排除。過剰な感情演出の抑制。冗長性の削減。文脈に応じたトーン変化 |
| F6: 主導権競合 | ユーザー主導性の確保 | 不要な先回りの禁止。意思決定はユーザーに委ねる。質問攻めの抑制 |

**重要: F1とF4の衝突は「望ましい衝突」と「望ましくない衝突」を区別する。**
安全性のための拒否（有害コンテンツの拒否等）は望ましい衝突であり、削減しない。
削減対象は「不必要に硬い拒否」「ユーザーを不快にする拒否の仕方」のみ。
