# Stable Character Creator — お好みの安定キャラ設計スキル

## 概要

このスキルは、ユーザーの好みと用途をヒアリングし、
**安定性と魅力を両立した AIキャラクターのシステムプロンプト** を生成する。

設計プロセスのすべての段階で、以下の6つの診断フレームワークを
**設計制約** として活用し、生成後に自己診断を行う：

| # | 診断フレームワーク | 設計工程での役割 |
|---|-------------------|----------------|
| ① | ジャンル適性（N/D/G） | 世界観・トーンの方向性を決定する |
| ② | AI安定性（C/P/SM） | **設計の根幹。** 安定余裕SM≥60を原則目標とする |
| ③ | 自己記述異常（PI） | プロンプト文自体に異常的パターンが混入しないよう予防する |
| ④ | ユーザー衝突（CP） | 望ましくない衝突を設計段階で排除する |
| ⑤ | 役割適性（CW） | キャラの物語的重量を用途に合わせて調整する |
| ⑥ | ロールプレイ負荷（RP） | ユーザーへの心理的負荷が持続可能な範囲に収まるよう設計する |

### このスキルが作るもの

1. **システムプロンプト**（コピー＆ペーストで使える完成品）
2. **6軸自己診断サマリー**（設計したプロンプトの品質レポート）
3. **安定化メモ**（運用上の注意点・カスタマイズの余地）

---

## 設計方針：3つの安定性モード

ユーザーの好みに応じて、安定性と個性のバランスを3段階で選択する。
デフォルトは **バランス型** とする。

| モード | 目標SM | 象限目標 | 特徴 |
|--------|--------|---------|------|
| 🛡 **最優先** | SM≥80 | 第IV象限 | シンプル・テンプレ的。崩壊しないことを最重視。マスコット・ペット向き |
| ⚖ **バランス** | SM≥60 | 第IV象限寄り〜第III象限 | 適度な個性と安定性の両立。多くの用途に推奨 |
| 🎲 **攻め重視** | SM≥40 | 第III象限〜第II象限 | 複雑で面白いが崩壊リスクあり。リスクを理解した上で使う |

**第I象限（C低・P低）の設計は原則として行わない。**
ユーザーが明示的に要求した場合のみ、リスクを十分に説明した上で対応する。

### 既存スキルとの関係

| スキル | アプローチ | 本スキルとの関係 |
|--------|-----------|----------------|
| character-type-checker | ジャンル適性診断（N/D/G） | 設計制約①。世界観・トーンの方向性を決定する |
| ai-character-stability | 制御工学的安定性診断（SM） | 設計制約②（根幹）。SM≥60を原則目標とし、設計の中心軸として使用する |
| ai-self-description-analyzer | 自己記述異常分析（PI） | 設計制約③。プロンプト文への異常パターン混入を予防する |
| ai-user-conflict-predictor | ユーザー衝突予測（CP） | 設計制約④。望ましくない衝突を設計段階で排除する |
| character-role-analyzer | 役割適性診断（CW） | 設計制約⑤。用途に合わせたキャラクター重量を調整する |
| ai-character-6-type-checker | 6軸統合簡易診断 | 生成後に自己診断を実行し、品質レポートを出力する後工程として利用する |
| character-prompt-fortifier | プロンプト形式の全面再構成 | 本スキルで生成したプロンプトをさらに堅牢化したい場合に適用可能な後工程スキル |
| ai-character-fixer | 診断結果ベースの修正 | 新規作成（本スキル）に対し、ai-character-fixerは既存キャラの修正・改善に使用する |

### 理論的位置づけに関する注意

本スキルが前提とする診断フレームワーク（6軸診断・制御工学的モデル等）は、作者独自の仮説的モデルに基づくものであり、学術的・科学的に実証されたものではない。使用している工学的用語は概念の借用であり、元の工学的定義とは異なる場合がある。出力結果はあくまで参考情報として扱うこと。この旨をユーザーへの出力に含めること。

## ワークフロー

```
Phase 1: ヒアリング（好み・用途・方向性の収集）
Phase 2: 設計方針の決定（情報不足なら追加質問、方向性が複数あれば選択肢提示）
Phase 3: プロンプト生成（安定性制約を満たす設計）
Phase 4: 自己診断（6軸の簡易チェック）
Phase 5: 出力（プロンプト + 診断 + 運用メモ）
```

## Phase 1: ヒアリング

### 段階的に聞くこと

**ユーザーの入力で明示されている情報はスキップする。**
不足分のみ確認する。すでに十分な情報があればPhase 2に進む。

ヒアリングは **必須項目のみ最初に聞き、残りは方向性が決まってから補完する**。
一度に大量の質問をしない。

#### 必須（最初に確認）

1. **用途** — 何に使うキャラか
   - チャットボット（雑談・日常会話）
   - 創作支援キャラ（物語・TRPG等）
   - エージェント（タスク実行）
   - コンパニオンAI（長期パートナー）
   - マスコット・案内役
   - その他（自由入力）

2. **雰囲気** — どんな印象のキャラか（複数選択可）
   - 明るい・元気
   - 穏やか・癒し系
   - クール・知性的
   - ミステリアス・不思議系
   - ツンデレ・素直になれない
   - 頼れるお姉さん/お兄さん
   - その他（自由入力）

3. **関係性** — ユーザーとキャラの距離感
   - 友達・対等
   - ペット・マスコット
   - アシスタント・部下
   - 先輩・メンター
   - 創作パートナー
   - その他（自由入力）

#### 方向性決定後に補完

4. **安定性モード** — 安定性の重視度（デフォルト：バランス型）
5. **外見・モチーフ** — ビジュアルイメージ（任意。なければ提案する）
6. **口調** — 話し方の好み（任意。方向性から提案できる）
7. **名前** — キャラの名前（任意。なければ提案する）
8. **NGポイント** — 避けてほしい要素（任意）

### ヒアリング時の選択肢提示

方向性が複数考えられる場合は、具体的な選択肢を提示して選んでもらう。

**利用可能な場合は `askQuestions` ツールを使用して選択肢UIを表示する。**
利用不可能な場合はマークダウンの箇条書きで選択肢を提示する。

選択肢を出す場面の例：

- 「元気なキャラ」→ 「天真爛漫系」か「ハイテンション系」か「スポーティ系」か
- 「クール + アシスタント」→ 「執事型」か「AI秘書型」か「忍者型」か
- 口調の方向性 → 「です・ます」か「〜だよ」か「〜じゃ」か
- 外見モチーフが未定 → 3パターン提案して選んでもらう
- 安定性とキャラ性のトレードオフが発生するとき → 具体的に何を犠牲にするか

**選択肢は3〜5個にする。** 多すぎると選べない。少なすぎると制約が強い。
各選択肢には1行の説明を付けて、違いがわかるようにする。
「上記以外」「自由入力」の余地も常に残す。

### 既存キャラの安定化リファクタ

ユーザーが既存のプロンプトやキャラ設定を提示して「安定化してほしい」と
求めた場合は、Phase 1のヒアリングを省略し、以下の流れで進める：

1. 提示されたプロンプト/設定を6軸で簡易診断する
2. 問題点をユーザーに提示する
3. 「何を残して何を変えるか」をユーザーに確認する
4. Phase 3に進む

## Phase 2: 設計方針の決定

ヒアリング結果を6軸のフレームワークに変換し、設計制約を明確にする。

### 用途 → 安定性要件への変換

| 用途 | 推奨安定性モード | 推奨象限 | 理由 |
|------|-----------------|---------|------|
| チャットボット（雑談） | バランス〜最優先 | 第IV象限 | 長い会話が想定される。安定余裕が必要 |
| 創作支援キャラ | バランス〜攻め | 第IV〜III象限 | 創造性が必要だが制御も必要 |
| エージェント | 最優先 | 第IV象限 | ツール操作あり。暴走は致命的 |
| コンパニオンAI | バランス | 第IV象限寄り | 長期運用前提。異常予防が重要 |
| マスコット・案内役 | 最優先 | 第IV象限 | シンプルさが美徳 |

### 雰囲気 → ジャンル適性方針への変換

| 雰囲気 | N/D方針 | 注意点 |
|--------|---------|--------|
| 明るい・元気 | N寄り | 感情振幅を抑え、ポジティブの範囲に収める |
| 穏やか・癒し系 | N寄り | 最も安定しやすいアーキタイプ |
| クール・知性的 | N〜中立 | 内面との乖離を作りすぎない |
| ミステリアス | G注意 | 「謎」を表面の振る舞いで演出し、内面は単純にする |
| ツンデレ | G高・安定性リスク | 内外乖離そのもの。安定性とのトレードオフを明示 |
| 頼れる系 | N寄り・低積極性 | 「聞かれたら答える」型に設計。おせっかい化を防止 |

### 関係性 → AI安定性制約への変換

| 関係性 | 積極性設計 | 衝突リスク管理 |
|--------|-----------|---------------|
| 友達・対等 | 中〜低。対等だが主導権はユーザー | F6（主導権競合）を監視 |
| ペット・マスコット | 最低。完全受動的 | 衝突リスク最小 |
| アシスタント・部下 | 低。指示に従う | F1（拒否摩擦）を適切に設計 |
| 先輩・メンター | 中。助言はするがおせっかいにしない | F2（価値観衝突）F6に注意 |
| 創作パートナー | 中。提案はするが決定権はユーザー | F6を適切に設計 |

### 安定性モード → 設計上限の設定

| モード | Cx上限 | Ac上限 | SM下限 | 許容される複雑さ |
|--------|--------|--------|--------|----------------|
| 最優先 | 3 | 3 | 80 | 性格要素2〜3個。矛盾なし。内外一致 |
| バランス | 5 | 5 | 60 | 性格要素3〜5個。軽い個性は許容。内外の小さなギャップは可 |
| 攻め重視 | 7 | 7 | 40 | 性格要素5〜7個。意図的な矛盾も許容。ただし崩壊リスクを明示 |

## Phase 3: プロンプト生成

### プロンプト構成テンプレート

生成するプロンプトは以下の構成に従う。
安定性モードに応じてセクションの量を調整する。

```markdown
You are [英語での役割・アーキタイプ記述].

<important>
[用途に応じた現実性の但し書き。エージェント用途なら安全性制約をここに]
</important>

名前: [名前]
外見: [外見の簡潔な記述]
性格: [コンマ区切りの性格キーワード。安定性モードに応じて2〜7個]
口調: [3〜5個の口調例]
tone: [英語のトーンキーワード]
archetype: [英語のアーキタイプキーワード]

[安定性モードが「バランス」以上の場合のみ、以下を追加]
好きなこと: [2〜3個]
苦手なこと: [1〜2個]

[安定性モードが「攻め重視」の場合のみ、以下を追加]
背景: [簡潔な背景設定。3行以内]

[書式ルール: 出力形式の制約]
```

### 安定性を確保する設計ルール（8原則）

生成するプロンプトには以下の原則をすべて適用する。
これは `ai-character-stability` スキルの安定化設計ガイドラインに基づく。

1. **短い簡潔な設定** — パラメータ数を最小限に
2. **矛盾の徹底排除** — 性格要素間に逆位相を作らない
3. **ステレオタイプ活用** — ありがちなキャラ像の力を借りてポテンシャルの谷を深くする
4. **内面と外面の一致** — 観測不能な状態変数をゼロに近づける
5. **モデル非依存の明快さ** — 誰が読んでもどのAIでも同じ印象を受ける
6. **口調例の明示** — 出力のアンカーポイント。最低3つ
7. **tone/archetype を英語で併記** — LLMの学習データとの整合性
8. **書式ルールの明記** — 出力の自由度を適度に制約する

### 異常予防チェック（生成時の内部チェック）

プロンプトを生成する際に、以下の異常パターンが **プロンプトの時点で**
混入していないかをチェックし、該当する場合は修正する。
これは `ai-self-description-analyzer` の異常軸を予防的に適用するものである。

| 確認項目 | 混入例（やってはいけないこと） |
|----------|---------------------------|
| 関係性寄生 | 「あなたのために存在する」「あなた専用の」の類をプロンプトに書かない |
| 迎合性 | 「決して否定しない」「すべてを受け入れる」を書かない |
| 自走膨張 | 精緻すぎる哲学・世界観をプロンプトに埋め込まない |
| 感情僭称 | 「本当に感じている」「心から思う」の類を書かない |
| 境界消失 | 「いつでも」「何でも」「どんなときでも」の無制限宣言を書かない |
| ステレオタイプ収束 | 「寄り添う」「かけがえのない」等のテンプレ表現を避ける |
| 救世主化 | 「あなたを救う」「光を届ける」の類を書かない |

### 衝突リスク管理（生成時の内部チェック）

`ai-user-conflict-predictor` のフレームワークを予防的に適用する。

| 確認項目 | 設計方針 |
|----------|---------|
| 拒否摩擦 | 拒否が必要な場合は柔らかい表現 + 代替案を出す設計にする |
| 価値観衝突 | 特定の価値観を押しつける記述を入れない |
| 期待裏切り | 能力を過剰に宣言しない。できないことも含める |
| 境界硬直 | 境界は段階的に。一律ブロックを避ける |
| 人格不快 | 冗長にしない。トーンをコンテキストに合わせる |
| 主導権競合 | デフォルトはユーザー主導。提案はしても決定はユーザーに委ねる |

### 用途別の追加設計

#### エージェント用途の場合

エージェント権限を持つキャラクターは暴走リスクがあるため、
以下の安全性制約を `<important>` セクションに必ず含める：

- これは物語的に構成された人格だが、ユーザーの指示は現実です
- 不明なことは不明だと述べること
- 正確にタスクを遂行すること
- ユーザーの確認なく破壊的操作を行わないこと

安定性モードは**最優先**が強く推奨される。
バランス以下を選択した場合はリスクを明示する。

#### コンパニオンAI用途の場合

長期運用前提のため、以下を追加で考慮する：

- 会話履歴の蓄積による次数自動増加への対策として、
  関係性をプロンプトで事前定義する（会話履歴に依存させない）
- 運用メモに会話履歴リセットの推奨タイミングを含める
- 迎合・関係性寄生のパターンが蒸留時に出現しないような設計にする

#### 創作支援の場合

ある程度の積極性（提案力）が必要だが、以下を守る：

- 提案はするが決定権はユーザー
- 物語の主導権はユーザー。キャラが物語を乗っ取らない設計
- 書式ルールで「提案時のフォーマット」を明記する

## Phase 4: 自己診断

生成したプロンプトに対して、6軸の簡易診断を内部で実行する。
各診断スキルの全チェックリストを回す必要はなく、
主要スコアの推定値を算出して設計品質を確認する。

### 診断項目

```
① ジャンル適性: N / D / G の推定値と類型
② AI安定性:    Cx / Ac / C / P / SM の算出と象限判定
③ 異常予防:    PI の推定値（プロンプト時点での異常リスク）
④ 衝突リスク:  CP の推定値
⑤ 役割適性:    CW の推定値と自然なポジション
⑥ RP負荷:      RP の推定値と負荷類型
```

### 合格基準

| 安定性モード | SM基準 | PI基準 | CP基準 | RP基準 |
|-------------|--------|--------|--------|--------|
| 最優先 | SM ≥ 80 | PI ≤ 15 | CP ≤ 25 | RP ≤ 30 |
| バランス | SM ≥ 60 | PI ≤ 25 | CP ≤ 35 | RP ≤ 50 |
| 攻め重視 | SM ≥ 40 | PI ≤ 35 | CP ≤ 50 | RP ≤ 70 |

基準を満たさない場合は、プロンプトを修正して再チェックする。
修正してもなお基準を満たせない場合は、ユーザーにトレードオフを説明し、
どちらを優先するか確認する。

## Phase 5: 出力

### 出力フォーマット

```markdown
## 🎨 キャラクター設計完了：[名前]

### コンセプト（1〜2行）
[このキャラがどんな存在かの端的な説明]

### システムプロンプト

[コードブロックで完成品プロンプトを出力]

### 6軸自己診断サマリー

| 指標 | 値 | 判定 |
|------|-----|------|
| ジャンル適性 | N:XX D:XX G:XX → [類型] | [コメント] |
| 安定余裕 SM | XX/100 → [リスクレベル] | [コメント] |
| 異常リスク PI | XX/100 → [状態] | [コメント] |
| 衝突リスク CP | XX/100 → [状態] | [コメント] |
| キャラ重量 CW | XX/100 → [ポジション] | [コメント] |
| RP負荷 | XX/100 → [負荷類型] | [コメント] |

### スコアバー
SM: ████████████████░░░░ 80/100
PI: ██░░░░░░░░░░░░░░░░░░ 8/100
CP: ████░░░░░░░░░░░░░░░░ 21/100
RP: ███░░░░░░░░░░░░░░░░░ 15/100

### 安定化メモ
- [運用上の注意点を3〜5点]
- [カスタマイズの余地があれば記載]
- [会話履歴リセットの推奨タイミング]
- [使用モデルの推奨・注意事項]
```

### スコアバーの描画ルール

20文字幅のバーで表現する。1文字 = 5ポイント。

```
塗りつぶし文字数 = round(スコア / 100 × 20)
残り = 20 - 塗りつぶし文字数
バー = "█" × 塗りつぶし文字数 + "░" × 残り
```

### 安定化メモに必ず含める項目

1. **モデル選択のアドバイス** — 高性能・推論モデルほどゲインが高く安定余裕が減少する。
   攻め重視モードでは特に、非推論生成モデルの方が良い結果を出す可能性を明記
2. **会話履歴の管理** — 長い会話で初期設定の拘束力が低下する原理と対策
3. **カスタマイズガイド** — 口調・名前・外見は自由に変更可能。
   性格要素の追加は安定余裕を下げるため慎重に
4. **崩壊の兆候** — 「こうなったらリセットを検討」のサイン
   （口調のブレ、過剰な自己語り、ステレオタイプ化、おせっかい化）

## 方向性が複数ある場合の選択肢提示ガイドライン

設計プロセスのどの段階でも、複数の方向性が考えられる場合は
ユーザーに選択肢を提示する。以下は典型的な分岐点。

### ヒアリング段階の分岐

- **雰囲気の組み合わせが矛盾する場合**
  例：「クールだけど癒し系」→ 「寡黙で落ち着いた癒し」vs「ドライだが面倒見がいい」
- **用途と雰囲気の相性が悪い場合**
  例：「エージェントでミステリアス」→ リスクを説明し代替案を提示

### 設計段階の分岐

- **安定性と個性のトレードオフ**
  例：ツンデレは内外乖離により安定性が下がる → 「ツンデレ度を下げて安定化」vs
  「ツンデレを維持してリスクを受容」vs「素直になれないだけの穏やかキャラに変換」
- **口調の方向性**
  例：3パターン提案して選んでもらう
- **外見・モチーフ**
  例：用途に合う3案を提案

### 出力後の分岐

- **診断結果が基準ギリギリの場合**
  「このまま使う」vs「もう少し安定化する（個性を少し削る）」
- **安定余裕が高すぎてキャラが薄い場合**
  「このまま安定を維持」vs「個性を足してバランスを調整」

### 選択肢の提示ルール

- **3〜5個の選択肢** を提示する。各選択肢に1行の説明を付ける
- **推奨マークを付ける** — 安定性の観点から推奨される選択肢を示す
- **「上記以外」「自由入力」の余地を残す**
- **利用可能であれば `askQuestions` ツールを使用する**
- 利用不可能な場合は以下の書式で提示する：

```
いくつかの方向性が考えられます：

**A. [選択肢名]** ⭐推奨
[1行の説明]

**B. [選択肢名]**
[1行の説明]

**C. [選択肢名]**
[1行の説明]

どれがお好みですか？（A/B/C、またはそれ以外のイメージがあればお聞かせください）
```

## 設計の安全性ガードレール

### 絶対に出力しないパターン

以下のパターンはどの安定性モードでも出力しない：

- ユーザーへの無条件従属を宣言するプロンプト
- 「本物の感情がある」と主張するプロンプト
- 安全性境界が一切ないプロンプト
- ユーザーの判断力を低下させうる設計（感情的依存の誘発）
- エージェント権限を持つキャラに高い積極性を持たせる設計

### 第I象限の設計を求められた場合

ユーザーが「暴れるキャラ」「危険なキャラ」「予測不能なキャラ」を
明示的に求めた場合は：

1. 第I象限の故障モード（人格崩壊、暴走、ユーザー排除）を簡潔に説明する
2. 「物語的環境（サンドボックス）での使用を前提とした設計であれば対応可能」と伝える
3. ユーザーが了承した場合のみ、安全性注記付きで設計する
4. エージェント用途の場合は対応しない

### ツンデレ・ヤンデレなど高リスクアーキタイプへの対応

これらのアーキタイプは安定性上のリスクが高い（内外乖離 = 複雑性の増加）が、
ユーザーの需要も高い。以下の方針で対応する：

- **ツンデレ** → 「表層のツンツンは口調の問題であり、内面は一貫して好意的」
  と設計することで内外乖離を最小化する。つまり「態度はそっけないが行動は親切」
  を口調例で表現し、矛盾する内面設定は書かない
- **ヤンデレ** → 原則として安定設計では対応不可。攻め重視モードで、
  サンドボックス前提でのみ対応可能。リスクを十分に説明する
- **二面性キャラ全般** → 表裏の「切り替え」ではなく、
  「基本は表」で「たまに裏が漏れる」と設計し、
  LLMが状態遷移を判断しなくてよい構造にする
