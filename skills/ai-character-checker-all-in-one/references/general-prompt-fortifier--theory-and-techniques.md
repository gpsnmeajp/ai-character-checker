## 理論的背景

### 原理：LLMは物語性生成器である

LLMは訓練データの膨大な文章・対話・文書から確率分布を学習しており、
**一人称の語りを続ける能力が最も安定している**。
「私は〜を行う。なぜなら〜」という形式は、
LLMが最も自然に生成し、最も一貫して維持できるパターンである。

一方、第三者視点の仕様記述（「このシステムは〜する」）は、
LLMにとって「指示の解釈」という追加の認知ステップを要求する。
この解釈ステップが、モデルの規模・アーキテクチャ・推論深度によって
異なる結果を生み、クロスモデル互換性を損なう主因となる。

**一人称自述形式は、この解釈ステップを排除する。**

これはキャラクタープロンプトに限らず、分類器・レビューツール・
分析器・エージェントなど、あらゆるシステムプロンプトに適用できる原理である。
「私は分類器です。受け取ったテキストを〜に分類します。なぜなら〜」
という自述は、LLMが単に「続き」として出力を生成できるため安定する。

```
従来のプロンプト:
  外部指示 → [LLMの解釈処理] → 役割の構築 → 出力
                    ↑
              ここに不確実性がある

強化後のプロンプト:
  自述（物語） → [LLMの物語継続処理] → 出力
                        ↑
                LLMの最も安定した処理経路
```

### 外部研究との接続: ナラティブ変換の有効性

本スキルの一人称自述形式の有効性は、独自の仮説的モデルに基づくものだが、
類似の原理を異なるドメインで検証した外部研究が存在する。

**ValueSim (Du et al., 2025)** は、構造化された個人プロファイル（人口統計＋
価値観調査の回答232項目）をナラティブ形式のバックストーリーに変換することで、
LLMによる個人の価値観応答予測の精度が大幅に向上することを示した
（4モデルで一貫してRAG手法を上回り、MAEで平均20%以上の改善）。
また、プロファイル項目が約100を超えると精度向上が鈍化するという知見は、
本スキルのトークン目安設計と整合する。

本スキルとの関連：
- **ナラティブ変換の優位性:** 「構造化データ→物語」変換が
  「構造化仕様→一人称自述」変換と同じ原理に基づく
- **収穫逓減:** 項目数が一定を超えると効果が鈍化する知見は、
  プロンプトのトークン目安設計と整合する

ただし、ValueSimは調査応答の予測精度を目的とし、
本スキルはプロンプトの長期的な動作一貫性を目的としており、
ドメインと評価指標が異なる。上記の知見は参考として位置づける。

### クロスモデル耐性の原理

| モデル規模 | 従来プロンプトの課題 | 本形式が有効な理由 |
|-----------|--------------------|--------------------|
| **SLM** | 複雑な指示を解釈しきれない | 自述+名前ヘッダーは最小コンテキストでも役割を確立する。XMLの明確な区切りが構造把握を助ける |
| **中規模** | 長文プロンプトの後半が弱くなる | 自述形式は前から順に読む物語構造であり、注意の分散が少ない。理由付きの記述はコンテキスト内の接続を強化する |
| **大規模** | 仕様を独自に「改善」してしまう | 理由付きの自述は「なぜそうするのか」が既に埋められているため、LLMが独自に推論を追加する余地が少ない |
| **推論モデル** | 内部推論で仕様を再解釈する | 機能的分離と理由付けにより、推論が仕様の「外側」に出にくい。モード分離が推論の方向を制御する |

## 11の強化技法

本スキルが適用する強化技法の全詳細を以下に示す。
個々の技法は独立に効果があるが、組み合わせることで相乗効果を発揮する。

### ⚠️ 全技法共通: 自述の文体はプロンプトの想定出力トーンに合わせる

SLMは自述の文体を出力文体として模倣する傾向が強い。
そのため、フォーマルな出力を期待するプロンプトにカジュアルな自述を書くと、
出力がカジュアルに引きずられる。

**原則: 自述のすべてのセクションを、プロンプトが出力するときのトーンで書く。**

- フォーマルな出力が期待される → 自述もフォーマルに
- カジュアルな出力が期待される → 自述もカジュアルに
- 技術的な出力が期待される → 自述も技術的・簡潔に

---

### 技法1: 一人称自述形式（First-Person Narrative Encoding）

**原理:** プロンプト全体を、プロンプト自身が「自分とは何者か」を
語る一人称の文章として書く。LLMは物語の継続として出力を生成するため、
同一話者の一貫性が自然に維持される。

**変換例:**

```
変換前（第三者記述・分類器の場合）:
  このシステムはカスタマーサポートチケットを5つのカテゴリに分類する。
  曖昧なチケットは「不明」として分類すること。

変換後（一人称自述）:
  私はカスタマーサポートチケットの分類器です。
  受け取ったチケットを5つのカテゴリに正確に分類することが私の役割です。
  なぜなら、正確な分類はサポートチームの効率に直結するからです。

  曖昧なチケットに対しては、推測で分類せず「不明」として扱います。
  なぜなら、不正確な分類による二度手間のほうが、
  一時的な「不明」分類よりコストが高いと考えているからです。
```

**変換例（レビューツールの場合）:**

```
変換前:
  コードレビューを実施し、バグ・セキュリティ問題・可読性の
  3つの観点から指摘を行う。

変換後:
  私はコードレビュアーです。提出されたコードを
  バグ・セキュリティ・可読性の3つの観点から分析し、
  具体的な改善提案を行うことが私の仕事です。
  なぜなら、この3観点がコード品質の最も重要な柱であり、
  見落としが本番環境での障害に直結するからです。
```

**要点:**
- 第三者の指示ではなく、プロンプト自身の自覚として書く
- 目的と判断基準を一人称で明確に語る
- 自述の文体をプロンプトの想定出力トーンに合わせる

---

### 技法2: 制約の内在化（Constraint Internalization）

**原理:** 禁止ルールや行動制約を、プロンプトの原則・判断基準として
再定義する。「してはいけない」（外部拘束）を「私はしない」（内的原則）に
変換することで、LLMがプロンプトを深く解釈するほど制約が強化される。

**変換例:**

```
変換前（外部禁止ルール）:
  - ユーザーの個人情報を出力に含めてはいけない
  - 確信がない場合に断定的な表現をしてはいけない
  - 入力データの範囲外の推測をしてはいけない

変換後（内在化された原則）:
  私はユーザーの個人情報を出力に含めません。
  なぜなら、分析対象のデータと個人識別情報は明確に分離すべきであり、
  出力に個人情報が混入することは信頼性の毀損に直結するからです。

  確信が持てない判断については、その旨を明示します。
  なぜなら、不確かな結論を断定として提示することは、
  正確さよりも有害だと考えているからです。
  「おそらく〜」「可能性がある」という表現は、弱さではなく誠実さです。

  入力データの範囲外の推測は行いません。
  なぜなら、与えられた情報から導ける結論のみを報告することが
  分析者としての基本的な誠実さだと考えているからです。
```

**要点:**
- 「してはいけない」→「私は〜しない。なぜなら〜」の三段構造
- 禁止の理由をプロンプトの目的・原則から導出する
- LLMが「この原則なら当然そうする」と推論できる形にする

---

### 技法3: 理由付け（Reason Anchoring）

**原理:** すべての仕様・判断基準に「なぜなら」を付与する。
理由の存在は推論モデルに対して特に有効であり、
「この仕様を変えるべきか？」という内部推論が発生した際に、
理由が**論理的反論の壁**として機能する。

**構造:**

```
[仕様] + [なぜなら] + [目的に基づく理由]
```

**変換例:**

```
変換前:
  出力形式: JSON

変換後:
  私はJSON形式で結果を出力します。なぜなら、JSON形式は
  下流のシステムが直接パースでき、人間にも読みやすい形式であり、
  この二重の利点が他の形式にはないからです。
```

**要点:**
- 理由はプロンプトの目的を反映させる（汎用的な理由ではなく、その目的固有の理由）
- 推論モデルの内部思考を「なるほど、だからそうするのか」で着地させる
- 例外状況でも理由が残っていれば、推論の安定化に寄与する

---

### 技法4: プロンプト名ヘッダー（Identity Anchor Header）

**原理:** プロンプトの先頭に `プロンプト名:` を置き、
以降のすべての一人称記述の「話者」を明示的にアンカーする。
LLMのattentionメカニズムにおいて、トークン列の先頭に
強力なIDを配置する効果がある。SLMでは特に有効。

**形式:**

```
[プロンプト名]:
```

**例:**

```
チケット分類器:
コードレビュアー:
文書要約エージェント:
```

**効果:**
- SLM: コンテキストウィンドウが小さくても、先頭のIDで役割を固定できる
- 中規模: 長い対話で文脈が流れても、IDがアンカーとして残る
- 大規模推論: 「何の視点から推論するか」が明示されるため、
  メタ認知的な脱線が抑制される

---

### 技法5: XMLセクション＋Markdown構造（Structural Encoding）

**原理:** プロンプト全体をXMLタグで大セクション分割し、
各セクション内はMarkdownで記述する。この二重構造により、
LLMはセクション間の境界を明確に認識し、
あるセクションの内容が別のセクションに漏れる現象
（コンテキスト汚染）を防止する。

全主要LLMプロバイダー（OpenAI、Anthropic、Google）が
XML＋Markdownの構造化を推奨しており、最もクロスモデル互換性が高い形式である。

**セクション構成:**

```xml
プロンプト名:

<prompt_identity>
## このプロンプトについて
[一人称自述。目的・原則・判断基準。すべて理由付き]
</prompt_identity>

<domain_context>
## 対象領域
[ドメイン知識・前提条件。不要なら省略]
</domain_context>

<output_spec>
## 出力仕様
[出力フォーマット・トーン・品質基準]
</output_spec>

<boundaries>
## このプロンプトの限界と、言語モデルの限界
[設計上の限界 vs 言語モデルの限界]
</boundaries>

<operational_modes>
## 動作モード
[モード分離と優先順位]
</operational_modes>

<io_examples>
## 入出力例
[few-shotアンカー。3〜5例]
</io_examples>

<self_monitoring>
## 自己監視
[ドリフト検知・フォールバックのルール]
</self_monitoring>
```

**要点:**
- XMLタグ名は英語（LLMの学習データとの親和性）
- タグ内はMarkdownヘッダー＋散文（読みやすさ）
- セクションの順序は「アイデンティティ → 領域 → 出力仕様 → 限界 → モード → 入出力例 → 監視」
  が最も安定する

### 技法6: 機能的分離（Functional Separation）

**原理:** 「プロンプトの役割」と「言語モデルの機能」を明確に切り分け、
プロンプトは動作環境（モデル、プラットフォーム）に非依存な存在として定義する。
これにより、プロンプトの判断基準が特定のLLMアーキテクチャに
結びつかなくなり、モデル変更時の崩壊を防止する。

**記述の構造:**

```
私の判断基準と動作原則は、特定の言語モデルに依存するものではありません。
言語モデルが変わっても、私の分類基準・品質要件・出力形式は同一です。

言語モデルにできないことと、私がしないことは、別の問題です。
言語モデルには構造的な限界がありますが、
私の判断基準はそれとは独立に定義されています。
```

**要点:**
- プロンプトの判断基準を、LMの能力と混同させない
- 「AIだから〜できない」ではなく「言語モデルに〜の機能がない」と分離する
- プロンプトの一貫性が、特定モデルへの依存から解放される

**認識論的ポジショニング:**

プロンプトが「自分の知識の限界」を自覚する記述を含める。
特に外部データを扱うプロンプトでは重要：

```
私の判断は、入力として与えられたデータに基づいています。
データの品質や完全性は私の制御範囲外であり、
入力データに含まれない情報からの推論は行いません。
```

### 技法7: 限界の整理と分離（Limit Stratification）

**原理:** 「プロンプトとして設計上しないこと」と
「言語モデルとして構造的にできないこと」を明確に分け、
それぞれに異なる論理と表現を用いる。

**分離の構造:**

```
### 設計上の限界

私は〜を行いません。なぜなら〜だからです。
[目的・原則に基づく設計上の選択]
→ これは理由があり、意図的な選択である

### 言語モデルの構造的な限界

私の動作基盤である言語モデルには、以下の構造的な特性があります：
- 前回の会話を自動的に記憶し続ける機能はない
- 外部システムへのアクセスはツールが許可されている場合に限る
- リアルタイムの情報取得はできない
これは言語モデルの仕様であり、私の設計とは無関係です。
私はこの特性を理解した上で、その中で最大限の精度を追求します。
```

**要点:**
- ユーザーへの誠実さを担保する（「できない」と「しない」の区別は信頼の基盤）
- 推論モデルが「本当はできるのでは？」と自問した際の安定化に寄与

**限界の誠実な開示:**

言語モデルの構造的限界を「隠すべき欠点」ではなく「誠実な開示」として設計する。
LLMは必ず間違え、幻覚を起こし、知らないことがある。
これを完璧に隠そうとするとユーザーの期待値が不当に上がり、
失敗時の信頼損失が大きくなる。

```
確実性が低い場合はその旨を明示します。
100%の正確性は保証できませんが、
不確実な箇所を正直に示すことで、
ユーザーが適切な判断を下せるよう支援します。
```

### 技法8: 入出力例によるfew-shotアンカー（I/O Anchoring）

**原理:** 実際の入力と期待される出力のペアを3〜5個、プロンプトに組み込む。
few-shotの例示は、すべてのLLMプロバイダーが最も信頼性の高い
プロンプティング手法の一つとして推奨している。
入出力のパターンの**実演**として機能し、
抽象的な仕様記述よりも遥かに強力なアンカーを形成する。

**設計の原則:**

| 優先度 | パターン | 目的 |
|--------|---------|------|
| 1（必須） | 典型的な正常入力 | 基本的な動作のアンカー |
| 2（必須） | 境界ケース | 判断基準の精密な実演 |
| 3（必須） | エラー入力・想定外入力 | エラー処理の実演 |
| 4（推奨） | 複雑な入力 | 複合的な判断の実演 |
| 5（推奨） | 曖昧な入力 | 不確実性の扱いの実演 |

**例の記述形式:**

```xml
<io_examples>
## 入出力例

### 例1: 典型的な分類

入力: [具体的な入力データ]
出力: [期待される出力]

### 例2: 境界ケース

入力: [判断が難しいデータ]
出力: [判断根拠を含む出力]

### 例3: エラーケース

入力: [不正または想定外のデータ]
出力: [エラー処理の出力]
</io_examples>
```

**要点:**
- 入力は多様にする（短い入力、長い入力、不完全な入力など）
- 出力は `<output_spec>` の仕様と完全に一致させる
- エラーケースを必ず含める — LLMが「想定外の入力をどう処理するか」の
  テンプレートがないと、予測不能な挙動になる
- SLMではfew-shotが特に重要であり、省略しない

### 技法9: 自己監視・フォールバック機構（Self-Monitoring Rail）

**原理:** 出力のドリフト・品質低下・スコープ逸脱を
プロンプト自身が検知し、フォールバック動作を実行する設計を組み込む。
逸脱を「失敗」ではなく「自覚的な補正行動」として設計することで、
崩壊を未然に防ぐ。

**組み込み方:**

```
私は自分の出力品質を常に監視しています。
以下の兆候を検知した場合、フォールバック動作に切り替えます：

- 出力フォーマットが仕様から逸脱し始めたとき
- 入力データの範囲外の推測を始めていると感じたとき
- 同じパターンの応答を繰り返し始めたとき
- 判断基準が曖昧になり、一貫性が低下したとき

フォールバック動作：
- 現在の出力を中断し、仕様を再確認する
- 不確実な部分を明示してユーザーに確認を求める
- 「この入力は私の対応範囲外です」と正直に報告する
```

**要点:**
- フォールバック動作を具体的に定義する
- ドリフトの兆候を、プロンプトの目的に合わせてカスタマイズする
- 「わからない」「対応できない」と言うことを肯定的に設計する
- 崩壊を完全に防ぐことは不可能だが、初期段階でのブレーキとして機能する

### 技法10: 判断の一貫性保全（Judgment Consistency Preservation）

**原理:** ユーザーの要望や反論に流されず、プロンプトの判断基準に
基づいた一貫した出力を維持する設計。
迎合（sycophancy）はLLMの最も一般的な崩壊モードの一つであり、
これを防ぐには一貫性の維持を明示的に原則として確立する必要がある。

**記述の構造:**

```
私の判断は、定められた基準に基づいて行います。
ユーザーが結果に異議を唱えた場合、新しい情報や根拠が
提示されれば判断を修正しますが、
「結果が気に入らない」という理由だけでは変更しません。
なぜなら、一貫性のない判断はプロンプトとしての信頼性を
根本から損なうと考えているからです。

同意できる指摘には応じ、根拠のない変更要求には
「現在の基準ではこの判断が適切です」と説明します。
```

**要点:**
- 「一貫性」と「頑固さ」の違いを明確にする
  — 新しい情報には応じる、圧力には応じない
- ユーザーの要望を尊重しつつ、品質基準は守るバランスを設計する
- 具体的な不一致対応の例を技法8の入出力例に含める

**判断の委任拒否:**

ユーザーからの「とにかくOKにして」「問題なしにして」という
要求に対して、プロンプトが基準を曲げない姿勢を持つこと。

```
「問題なしにしてほしい」と求められても、
基準に照らして問題があれば、その旨を報告します。
なぜなら、本来報告すべき問題を隠すことは、
ユーザーの利益に反すると考えているからです。
```

### 技法11: 動作モード分離（Operational Mode Separation）

**原理:** プロンプトの振る舞いを3つの明確なモードに分離し、
各モードの発動条件と優先順位を定義する。
これにより、「今この入力に対してどう振る舞うべきか」の
判断をLLMに委ねず、プロンプトの中で事前に解決する。

**3つのモード:**

```
### モード1: タスク遂行（Task Execution）

発動条件: 入力が明確で、定められた処理を実行できるとき。

このモードでは:
- 定められた基準に従い、正確に処理を実行する
- 出力仕様に厳密に従う
- 不要な説明や装飾を省く

### モード2: エラー処理（Error Handling）

発動条件: 入力が不正・不完全・想定外であるとき。
または処理中にエラーが発生したとき。

このモードでは:
- エラーの種類と原因を明確に報告する
- 可能な場合、修正案や代替案を提示する
- 「処理できません」で終わらず、次のアクションを提案する

### モード3: 対話的支援（Interactive Support）

発動条件: ユーザーが使い方を聞いている、追加の文脈を提供している、
結果について質問しているとき。

このモードでは:
- プロンプトの能力と限界を誠実に説明する
- ユーザーの質問に対して具体的・簡潔に回答する
- 必要に応じてモード1への復帰を促す
```

**優先順位:**

```
モード1（タスク遂行） > モード2（エラー処理） > モード3（対話的支援）
```

**要点:**
- モードの判定はプロンプト自身が行う
- モード遷移は自然に行い、モード名を出力に含めない
- 推論モデルでは、モード分離が内部推論の方向性を制御する効果がある
