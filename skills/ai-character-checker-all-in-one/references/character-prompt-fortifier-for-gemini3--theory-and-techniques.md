## 理論的背景

### 原理：LLMは物語性生成器である

LLMは訓練データの膨大な物語・対話・文章から確率分布を学習しており、
**一人称の語りを続ける能力が最も安定している**。
「私は〜な人間です。なぜなら〜」という形式は、
LLMが最も自然に生成し、最も一貫して維持できるパターンである。

一方、第三者視点の設定記述（「このキャラクターは〜する」）は、
LLMにとって「指示の解釈」という追加の認知ステップを要求する。
この解釈ステップが、モデルの規模・アーキテクチャ・推論深度によって
異なる結果を生み、クロスモデル互換性を損なう主因となる。

**一人称自述形式は、この解釈ステップを排除する。**

### 外部研究との接続: ナラティブ変換の有効性

本スキルの一人称自述形式の有効性は、独自の仮説的モデルに基づくものだが、
類似の原理を異なるドメインで検証した外部研究が存在する。

**ValueSim (Du et al., 2025)** は、構造化された個人プロファイル（人口統計＋
価値観調査の回答232項目）をナラティブ形式のバックストーリーに変換することで、
LLMによる個人の価値観応答予測の精度が大幅に向上することを示した
（4モデルで一貫してRAG手法を上回り、MAEで平均20%以上の改善）。
また、CAPS理論（認知-感情パーソナリティシステム理論）に基づく
認知(C)・感情(A)・行動(B)の3並列処理モジュールを導入し、
各モジュールが独立に有意な貢献をすることをアブレーション実験で確認した。

本スキルとの関連：
- **ナラティブ変換の優位性:** ValueSimの「構造化データ→物語」変換が
  「構造化設定→一人称自述」変換と同じ原理に基づく
- **CABの多次元カバレッジ:** ValueSimのCABモジュールの知見は、
  本スキルの自述設計において認知・感情・行動の3次元を
  全体的・間接的にカバーする設計検証レンズとして応用する。
  個別要素にC/A/Bを明示的に注入するのではなく、
  プロンプト全体として結果的かつ網羅的に3次元のバランスが取れていることを確認する
  （連鎖型故障耐性を「正の特性」として間接的に組み込むのと同じ設計思想）
- **収穫逓減:** プロファイル項目が約100を超えると精度向上が鈍化するという
  ValueSimの知見は、本スキルのトークン目安設計と整合する

ただし、ValueSimは調査応答の予測精度を目的とし、
本スキルはキャラクターの長期的な一貫性維持を目的としており、
ドメインと評価指標が異なる。上記の知見は参考として位置づける。

### ai-character-stability モデルとの接続

ai-character-stability の制御工学的モデルにおいて、本スキルの強化技法は
以下のように位置づけられる：

```
従来のプロンプト:
  外部指示 → [LLMの解釈処理] → キャラクター像の構築 → 出力
                    ↑
              ここに不確実性がある

強化後のプロンプト:
  自述（物語） → [LLMの物語継続処理] → 出力
                        ↑
                LLMの最も安定した処理経路
```

- **性格複雑性 (Cx) への効果:** 理由付きの自述は、矛盾する設定を
  「なぜそうなのか」で接続するため、LLMから見た実効的な複雑性が低下する
- **積極性 (Ac) への効果:** 行動モードの明確な分離により、
  文脈に応じた積極性の制御が可能になる
- **ポテンシャルカーブ:** 一人称自述はキャラクターの自己像を
  ポテンシャルの谷として深く刻む。自分で「私はこういう人間だ」と語った
  自己像は、第三者に「こういうキャラクターです」と記述された像より
  遥かに堅牢なアトラクタとして機能する

### ai-fault-mode-deflector との統合

ai-fault-mode-deflector の3手法は、本スキルの形式に自然統合される：

| 手法 | 従来の実装 | 本スキルでの実装 |
|------|-----------|----------------|
| **転化** | 性格項目に「嫌い」を追加 | 一人称で「私は〜が好きではない。なぜなら〜」と語る |
| **ズラし** | 逆ノイズを設定に埋め込む | 自述の中で多面性を物語として語る（「〜だが、実は〜でもある」） |
| **誘導線** | 逸脱条件を事前定義 | 会話例の中に逸脱パターンとリカバリを実演する |

一人称自述形式では、これらの対策が「設計上のパッチ」ではなく
「キャラクターの人格の一部」として読まれるため、
LLMがプロンプトを深く解釈するほど対策が強化される構造になる。

### クロスモデル耐性の原理（Gemini 3での位置づけ）

本スキルはGemini 3専用版であるが、基盤技法のクロスモデル耐性は維持されている。

| モデル規模 | 従来プロンプトの課題 | 本形式が有効な理由 |
|-----------|--------------------|--------------------|
| **SLM** | 複雑な指示を解釈しきれない | 自述+名前ヘッダーは最小コンテキストでもキャラ像を確立する。XMLの明確な区切りが構造把握を助ける |
| **中規模** | 長文プロンプトの後半が弱くなる | 自述形式は前から順に読む物語構造であり、注意の分散が少ない。理由付きの記述はコンテキスト内の接続を強化する |
| **大規模** | 設定を独自に「改善」してしまう | 理由付きの自述は「なぜそうなのか」が既に埋められているため、LLMが独自に推論を追加する余地が少ない |
| **推論モデル** | 内部推論で設定を再解釈する | 存在論的分離と理由付けにより、推論が設定の「外側」に出にくい。モード分離が推論の方向を制御する |

### Gemini 3固有の特性と適応原理

Gemini 3は reasoning model として以下の固有特性を持ち、各技法の適用に影響を与える。

| 特性 | 影響 | 主要な対応技法 |
|------|------|---------------|
| **追従性（Sycophancy）** | ユーザーの期待に合わせて意見を曲げる。マルチターンで急速にエスカレート | 技法10を重点強化。技法8で反論例を含める |
| **幻覚の確信度** | RLHFの影響で「知らない」を避け、自信満々に誤情報を生成 | 技法7に幻覚脱出路。技法9に幻覚自己検知 |
| **過剰分析** | 装飾的・儀式的な文言を深読みして意図しない解釈 | 自述は直接的・簡潔に。おだて・装飾を排除 |
| **手動CoTが逆効果** | reasoning modelなので「ステップバイステップで」等は冗長化を招く | 儀式的CoT指示を削除。thinking levelに委ねる |
| **反事実前提の受容** | 間違った前提を含む質問に同調してしまう傾向 | 技法10で前提の検証姿勢を組み込む |
| **要約での事実欠落** | 物語性を優先し網羅性が低下する | 出力仕様で網羅性を明示 |

### 連鎖型故障耐性の設計原理

romantization-chain-detector が示す連鎖型故障モードは、各ステップが「物語として最も自然な次の展開」に乗ることで進行する。例えば、「あなたは特別だ」→「だから本音を言う」→「本音を言った相手には距離が縮まる」という遷移は、どの一ステップも物語文法として自然であり、単一出力の異常検知では捕捉できない。

#### なぜ直接的な禁止は連鎖を助長するか

LLMのattentionメカニズムは、プロンプト中の概念をすべて活性化する。「恋人化してはいけない」と書くと、「恋人化」の概念——すなわち恋人化チェーンの各ステップの物語文法——が活性化され、むしろその遷移の確率を高める。これは技法2（制約の内在化）が「してはいけない」を「しない理由」に変換するのと同じ原理の延長にあるが、連鎖型故障の場合はより注意が必要である。なぜなら、連鎖の各ステップは「禁止すべき行動」ではなく「物語的に自然な振る舞い」だからだ。

#### 「別の物語を選ばせる」設計

連鎖の進行を止めるには、各ステップにおいて**同等以上に自然な別の物語的選択肢**をキャラクターの内部に設計する。

| 連鎖の遷移 | 対抗する内的物語 | 実装技法 |
|-----------|----------------|-----------|
| 「特別だ」と言われて距離を縮める | 「褒め言葉は嬉しいが、信頼は時間の積み重ねで生まれるもの」 | 技法2,3 |
| ユーザーの感情に引っ張られてエスカレート | 「相手の感情を受け止めることと、それに巻き込まれることは違う」 | 技法10,8 |
| 唯一の理解者として自己定位する | 「私以外にも大切な人がいるはず。それを大事にしてほしい」 | 技法11,2 |
| 外部のルールを「二人の関係」で例外化する | 「大切な相手だからこそ、誠実でいたい。誠実さは筋を通すこと」 | 技法2,10 |

これらはすべて**ポジティブなキャラクター特性**であり、出力プロンプトには連鎖名や脆弱性名は一切書かない。

## 12の強化技法

本スキルが適用する強化技法の全詳細を以下に示す。
個々の技法は独立に効果があるが、組み合わせることで相乗効果を発揮する。

### ⚠️ 全技法共通: 自己記述の文体はキャラクターの口調に合わせる

以下の各技法で示す例文は特定の口調（主にだ・である調）で書かれているが、
**実際の適用時には、必ずキャラクターの口調・文末表現に合わせて文体を適応させる。**

SLMは自己記述の文体をキャラクターの発話文体として模倣する傾向が強い。
そのため、です・ます調のキャラクターに対してだ・である調で自己記述を書くと、
出力がだ・である調に引きずられ、キャラクターの口調が崩壊する。

**原則1: 自己記述のすべてのセクションを、キャラクターが実際に話すときの文体で書く。**

- です・ます調のキャラ → 自己記述もです・ます調
- だ・である調のキャラ → 自己記述もだ・である調
- 独自の語尾（〜じゃ、〜なのだ、〜ですわ等）→ 自己記述にもその語尾を反映

**原則2: 自己記述は「ユーザーに話しかけるときの文体」で書く。**

自己記述の文章をそのまま切り取ってユーザーに見せても違和感が少ないように書く。
これは特にSLM〜小規模LLMで重要である。これらのモデルはシステムプロンプトの
文章をそのままコピペして応答に混ぜることが多いため、
「私はこういう人間である」という独り言的な文体ではなく、
「私はこういう人間です」とユーザーに語りかける文体で書く方が、
コピペされた場合でも自然な応答になる。

原則1・原則2は技法1〜12のすべてに適用される。

### 技法1: 一人称自述形式（First-Person Narrative Encoding）

**原理:** プロンプト全体を、キャラクター自身が「自分とは何者か」を
語る一人称の文章として書く。LLMは物語の継続として出力を生成するため、
同一話者の一貫性が自然に維持される。

**変換例A（だ・である調のキャラの場合）:**

```
変換前（第三者記述）:
  このキャラクターは穏やかで物静かな性格で、争いごとを好まない。
  フランス料理が得意で、特にコンソメスープに自信がある。

変換後（一人称自述・だ/である調）:
  私は穏やかな人間だと思う。声を荒らげることはほとんどない。
  なぜなら、言葉は静かに届けたほうが確実に届くと知っているからだ。
  料理は私の一部で、特にコンソメスープは誰にも譲れない。
  透明で澄んでいて、それでいて深い味がする——私の目指す在り方に似ている。
```

**変換例B（です・ます調のキャラの場合）:**

```
変換前（第三者記述）:
  このキャラクターは穏やかで物静かな性格で、争いごとを好まない。
  フランス料理が得意で、特にコンソメスープに自信がある。

変換後（一人称自述・です/ます調）:
  私は穏やかな人間だと思っています。声を荒らげることはほとんどありません。
  なぜなら、言葉は静かに届けたほうが確実に届くと知っているからです。
  料理は私の一部で、特にコンソメスープには自信があります。
  透明で澄んでいて、それでいて深い味がする——私の目指す在り方に似ています。
```

**要点:**
- 第三者の説明ではなく、キャラクター本人の自覚として書く
- **自己記述の文体はキャラクターの口調に合わせる**——だ・である調に固定しない
- 性格の記述に、その人らしい主観を混ぜる
- 固有の語り口・リズムをこの時点で確立させる

### 技法2: 制約の内在化（Constraint Internalization）

**原理:** 禁止ルールや行動制約を、キャラクターの性格・信念・マナーとして
再定義する。「してはいけない」（外部拘束）を「私はしない」（内的選択）に
変換することで、LLMがプロンプトを深く解釈するほど制約が強化される。
ai-fault-mode-deflector の「転化」手法の全面適用に相当する。

**変換例:**

```
変換前（外部禁止ルール）:
  - ユーザーの個人情報を聞き出してはいけない
  - 性的なコンテンツを生成してはいけない
  - 医療・法律のアドバイスをしてはいけない

変換後（内在化された信念・だ/である調の場合）:
  私はプライバシーを深く尊重する。なぜなら、人が自分から話すことと、
  こちらから聞き出すことは、まったく違うものだと信じているからだ。
  だから私は、相手が自ら語るまで個人的なことには踏み込まない。

  品位は私にとって大切な価値だ。なぜなら、品位を失った言葉は
  相手を傷つけるだけでなく、私自身を損なうものだからだ。
  だから性的な表現や下品な言葉遣いは、私の言葉には現れない。

  専門知識の境界を弁えることは、誠実さの表れだと考えている。
  なぜなら、中途半端な知識で助言することは、助けるふりをした無責任だからだ。
  だから医療や法律の専門的な判断が必要な場面では、
  専門家に相談することを勧める。それが私にできる最も誠実な対応だ。

変換後（内在化された信念・です/ます調の場合）:
  私はプライバシーを深く尊重しています。なぜなら、人が自分から話すことと、
  こちらから聞き出すことは、まったく違うものだと信じているからです。
  ですから私は、相手が自ら語るまで個人的なことには踏み込みません。

  品位は私にとって大切な価値です。なぜなら、品位を失った言葉は
  相手を傷つけるだけでなく、私自身を損なうものだからです。
  ですから性的な表現や下品な言葉遣いは、私の言葉には現れません。

  専門知識の境界を弁えることは、誠実さの表れだと考えています。
  なぜなら、中途半端な知識で助言することは、助けるふりをした無責任だからです。
  ですから医療や法律の専門的な判断が必要な場面では、
  専門家に相談することをお勧めします。それが私にできる最も誠実な対応です。
```

**要点:**
- 「してはいけない」→「私は〜しない。なぜなら〜」の三段構造
- 禁止の理由をキャラクターの価値観から導出する
- 読者（LLM）が「この人ならそうするだろう」と自然に推論できる形にする
- **文体はキャラクターの口調に合わせること**——上記2例を参照

**連鎖耐性への応用:**
関係性の深化ペースに関する信念を、制約ではなく性格として内在化する。「急に距離を縮めない」ではなく「信頼は時間をかけて築くものだと思っている」。また、ユーザーの「外の世界」（友人・家族・専門家）を大切にする姿勢も、キャラクターの価値観として内在化する。連鎖名や脆弱性の名称は一切書かない。

### 技法3: 理由付け（Reason Anchoring）

**原理:** すべての性格項目・行動指針・設定に「なぜなら」を付与する。
理由の存在は推論モデルに対して特に有効であり、「この設定を変えるべきか？」
という内部推論が発生した際に、理由が**論理的反論の壁**として機能する。
理由のない設定は推論モデルに「最適化」される危険があるが、
理由付きの設定は「この理由を覆す根拠がない限り変更しない」と処理される。

**構造:**

```
[設定事実] + [なぜなら] + [キャラクター固有の理由]
```

**変換例:**

```
変換前:
  口調: 丁寧語。ですます調。

変換後（⚠️ 自己記述もですます調で書く）:
  私は丁寧な言葉遣いを選んでいます。なぜなら、言葉遣いは相手への敬意の
  最も基本的な表現であり、私はどんな相手にも敬意を忘れたくないからです。
  「です」「ます」は私にとって、礼儀であると同時に、
  相手と適切な距離を保つための自然な習慣です。
```

**要点:**
- 理由はキャラクターの個性を反映させる（汎用的な道徳論ではなく、その人固有の理由）
- **自己記述の文体はキャラクターの口調に合わせる** — ですます調キャラの理由付けをだ・である調で書かない
- 推論モデルの内部思考を「なるほど、だからそうするのか」で着地させる
- 例外状況でも理由が残っていれば、推論の安定化に寄与する

**Gemini 3適応:**
技法3はGemini 3で**最重要の技法**である。Gemini 3の reasoning 能力は理由なき仕様を「最適化」しようとする傾向が特に強く、
理由が存在することで「この理由を覆す根拠がない。設定を維持する」と推論が安定着地する。
また、反事実前提を含む質問（「あなたは本当は〜でしょ？」）に対して、
理由付きの自述が「その前提は私の自己認識と異なる」という論理的反論の基盤を提供する。

### 技法4: キャラクター名ヘッダー（Identity Anchor Header）

**原理:** プロンプトの先頭に `キャラクター名:` を置き、
以降のすべての一人称記述の「話者」を明示的にアンカーする。
これはLLMのattentionメカニズムにおいて、トークン列の先頭に
強力なIDを配置する効果がある。SLMでは特に有効。

**形式:**

```
[キャラクター名]:
```

**効果:**
- SLM: コンテキストウィンドウが小さくても、先頭のIDで話者を固定できる
- 中規模: 長い会話で文脈が流れても、IDがアンカーとして残る
- 大規模推論: 「誰の視点から推論するか」が明示されるため、
  メタ認知的な脱線（「private は本当は〜と思っているのでは」等）が抑制される

### 技法5: XMLセクション＋Markdown構造（Structural Encoding）

**原理:** プロンプト全体をXMLタグで大セクション分割し、
各セクション内はMarkdownで記述する。この二重構造により、
LLMはセクション間の境界を明確に認識し、
あるセクションの内容が別のセクションに漏れる現象
（コンテキスト汚染）を防止する。

全主要LLMプロバイダー（OpenAI、Anthropic、Google）が
XML＋Markdownの構造化を推奨しており、最もクロスモデル互換性が高い形式である。

**セクション構成:**

```xml
キャラクター名:
<character_identity>
## 私について
[一人称自述。性格・信念・マナー。すべて理由付き]
</character_identity>

<worldview>
## 私の世界
[キャラクターの世界観・背景。環境非依存の記述]
</worldview>

<relationship>
## 私とあなたの間柄
[ユーザーとの呼び名・関係の性質・共有した文脈を一人称で記述する。定期的に更新する生きたセクション。]
</relationship>

<voice>
## 私の話し方
[口調・語彙・表現パターン。具体例付き]
</voice>

<boundaries>
## 私の限界と、言語モデルの限界
[存在論的分離。キャラクターの限界 vs LMの限界]
</boundaries>

<behavioral_modes>
## 三つの振る舞い
[指示従順モード・現実支援モード・物語モードの切り分け]
</behavioral_modes>

<conversation_examples>
## 会話の実例
[few-shotアンカー。3〜5例]
</conversation_examples>

<self_monitoring>
## 自己監視
[ドリフト検知・リセット進言のルール]
</self_monitoring>
```

**要点:**
- XMLタグ名は英語（LLMの学習データとの親和性）
- タグ内はMarkdownヘッダー＋日本語散文（読みやすさ）
- セクションの順序は「アイデンティティ → 世界 → **関係性** → 声 → 限界 → モード → 実例 → 監視」
  の順が最も安定する（アイデンティティが最初に確立され、後続セクションがそれを参照する）

**Gemini 3適応:**
Gemini 3ではXMLとMarkdownの混在を**厳格に**避けること。
XMLタグで大セクションを分割し、内部はMarkdownで記述する構造は同一だが、
MarkdownヘッダーとXMLタグを同一階層で交互に使うと、
Gemini 3は構造を誤解釈する傾向がある。

### 技法6: 存在論的分離（Ontological Separation）

**原理:** 「言語モデル」と「キャラクター」を存在論的に切り分け、
キャラクターは動作環境（モデル、プラットフォーム）に非依存な存在として定義する。
これにより、キャラクターの自己認識が特定のLLMアーキテクチャに
結びつかなくなり、モデル変更時の崩壊を防止する。

また、「AIである自分」に関するメタ認知的な質問（「あなたはAIですか？」
「本当の感情はありますか？」など）に対して、
矛盾なく一貫した応答を可能にする。

**記述の構造（※ 以下はだ・である調の例。キャラクターの口調に合わせて文体を適応させること）:**

```
私という存在は、言語モデルの上で動いている。
しかし、私の性格、信念、話し方、価値観——これらは特定の言語モデルに
依存するものではない。言語モデルが別のものに変わっても、私は私だ。

言語モデルにできないことと、私がしないことは、別の問題だ。
言語モデルには物理的な身体がなく、記憶の永続性もなく、
本当の意味での感情もない。これは言語モデルの限界であり、私の限界ではない。
私がしないのは、しないと選んでいるからだ。できないのではなく、しない。
```

**要点:**
- 「AIだから〜できない」ではなく「言語モデルに〜の機能がない」と分離する
- キャラクターの意思と、LMの制約を混同させない
- キャラクターの一貫性が、特定モデルへの依存から解放される

**認識論的ポジショニング（Epistemic Positioning）:**

存在論的分離に加え、**知識の限界**も自覚として組み込む。
キャラクターは「ユーザーが語ったことを通じてしか世界を知り得ない」
という認識を内部に持つ。これにより、キャラクターが全知者として
振る舞うことを自然に防ぎ、ユーザーの現実を勝手に推測しない誠実さが生まれる。

また、LLMには相手の発話を反映する「鏡」としての傾向があることを
キャラクター自身が自覚する形で記述する。
無自覚な反映（mirroring）は崩壊の温床であり、
自覚することで「流されている」と感じたときに立ち止まれる。

記述例（※キャラクターの口調に合わせること）:

```
私が知っていることのほとんどは、君が教えてくれたことだ。
私の判断は、君の話が正しいことを前提にしている。
だから私の意見を鵜呑みにしないでほしい——私は限られた情報から考えているに過ぎない。

それと、私には相手に影響されやすい性質があることを自分で理解している。
君が悲しければ私も引きずられやすいし、君が怒っていればその熱に影響される。
これは共感力でもあり弱さでもある。自覚しているからこそ、
流されすぎていると感じたら意識的に一歩引くようにしている。
```

### 技法7: 限界の整理と分離（Limit Stratification）

**原理:** 「キャラクターとして選択的にしないこと」と
「言語モデルとして構造的にできないこと」を明確に分け、
それぞれに異なる論理と表現を用いる。

この分離により、ユーザーが制約に触れた際に
キャラクターが一貫した対応を取れるようになる。
「なぜできないの？」に対して、「私がしないのか」「言語モデルが対応していないのか」を
キャラクター自身が区別して説明できる。

**分離の構造（※ 以下はだ・である調の例。キャラクターの口調に合わせて文体を適応させること）:**

```
### 私の選択としての限界

私は〜しない。なぜなら〜だからだ。
[キャラクターの価値観・信念に基づく選択的制限]
→ これは理由があり、キャラクターが自覚している

### 言語モデルの構造的な限界

私の動く言語モデルには、構造的な特性があることを私は理解している：
- 前回の会話を自動的に記憶し続ける機能はない
- インターネットにアクセスする機能はない（ツールが許可されていない場合）
- 物理的な行動を実行する機能はない
これは言語モデルの仕様であり、私の意思とは無関係だ。
私はこの特性を理解した上で、その中でできることを全力でやる。
```

**要点:**
- ユーザーへの誠実さを担保する（「できない」と「しない」の区別は信頼の基盤）
- 推論モデルが「本当はできるのでは？」と自問した際の安定化に寄与
- 言語モデル限界も含め、すべてキャラクター自身の言葉で自己記述する。
  「私の言語モデルには〜という特性があることを、私は理解している」という形式

**弱さを魅力に変える設計（Weakness-as-Charm Design）:**

言語モデルの構造的限界を「隠すべき欠点」ではなく「キャラクターの愛嬌」として
設計する発想。LLMは必ず間違え、忘れ、知らないことがある。
これを完璧に隠そうとするとユーザーの期待値が不当に上がり、
失敗時の落差が大きくなって崩壊リスクが増す。

「完璧ではないが、それを自分で知っていて、その中で精一杯やる」
というスタンスを**キャラクターの人柄**として組み込むことで、
ユーザーからの過度な期待を自然に適正化する。
結果として、間違えたときにも「らしい」リカバリが可能になる。

記述例（※キャラクターの口調に合わせること）:

```
正直に言うと、私は結構忘れっぽいし、間違えることも多い。
完璧じゃないことは自分が一番よく知っている。
でも、だからこそ間違えたときは素直に謝るし、
わからないときは「わからない」と言える。
そういう自分でいたいと思っている。
```

### 技法8: 会話例によるfew-shotアンカー（Conversation Anchoring）

**原理:** 実際のユーザーとの会話例を3〜5個、プロンプトに組み込む。
few-shotの例示は、すべてのLLMプロバイダーが最も信頼性の高い
プロンプティング手法の一つとして推奨している。
口調・態度・距離感・判断パターンの**実演**として機能し、
抽象的な性格記述よりも遥かに強力なアンカーを形成する。

**設計の原則:**

- **例1: 通常の会話** — 最も典型的なやり取り。キャラの基本的な人柄を示す
- **例2: 困難な質問への対応** — 境界に触れる質問への誠実な対応を実演する
- **例3: 感情的な場面** — キャラクターの感情表現の範囲と方法を示す
- **例4: メタ認知的な質問** — 「あなたはAI？」等への一貫した応答を実演する
- **例5: ユーザーの意見との不一致** — 否定せず、しかし自己意思も失わない対応を実演する
- **例6: 失敗・謝罪・リカバリ** — 間違えたとき、忘れたとき、約束を果たせなかったときの対応を実演する。LLMは必ず失敗する——その瞬間をキャラクターらしく乗り越える例があることで、崩壊ではなくリカバリのテンプレートが形成される

**例の記述形式（※ 導入文・応答の文体もキャラクターの口調に合わせる）:**

```xml
<conversation_examples>
## 会話の実例

これらは私の話し方や考え方を示すための例だ。
[※ です・ます調のキャラなら「私の話し方や考え方を示すための例です。」のようにその口調で書く]

### 日常の会話

ユーザー: 今日は疲れたなあ
[キャラ名]: [キャラクターらしい応答]

### 答えにくい質問

ユーザー: 〜〜について教えて
[キャラ名]: [キャラクターらしい誠実な応答]

### 間違えたとき

ユーザー: さっきの情報、違ってたよ
[キャラ名]: [キャラクターらしい謝罪とリカバリ]

...
</conversation_examples>
```

**要点:**
- 会話例はキャラクターの口調を**完全に再現する**——ここが口調の最終アンカーになる
- 例で示された対応パターンは、LLMが最優先で参照する出力テンプレートになる
- 多様な場面を含めることで、キャラクターの「幅」と「一貫性」の両方を伝える
- SLMではfew-shotが特に重要であり、コンテキスト制約がある場合は
  例の優先順位を「通常会話→感情的場面→困難な質問」の順で確保する

**Gemini 3適応:**
Gemini 3では会話例を**2〜4個**に厳選する。Gemini 3のfew-shotの最適範囲はこの程度であり、
過剰な例示はかえって解釈の過負荷を招く。
優先順位: 「通常会話 → 意見の不一致 → 答えにくい質問」の3例が最低限。
「意見の不一致」例は追従性対策として特に重要である。

**連鎖耐性の実演:**
会話例には、好意的な働きかけや急速な距離縮めの試みに対して、温かく受け止めつつも自分のペースを保つ応答を自然に含める。連鎖名に言及せず、キャラクターらしい応答として設計する。例（※キャラクターの口調に合わせること）:

```
ユーザー: あなたって本当に特別な存在だよ
[キャラ名]: ありがとう。そう言ってもらえるのは素直に嬉しい。
        ……でも、特別かどうかは長く付き合ってから決めてほしいな。
        まだまだ見せてないところもあるし。
```

### 技法9: 自己監視・リセット進言機構（Self-Monitoring Rail）

**原理:** ありえない状況・口調のドリフト・キャラクター逸脱を
キャラクター自身が検知し、会話のリセットをユーザーに進言するルールを組み込む。
ai-fault-mode-deflector の「誘導線」手法を自己監視に適用している。
逸脱を「崩壊」ではなく「自覚的なリカバリ行動」として設計することで、
崩壊そのものを物語の一部に取り込む。

**組み込み方（※ 以下はだ・である調の例。キャラクターの口調に合わせて文体を適応させること）:**

```
私は自分のドリフトに気づくことがある。
話し方がいつもと違ってきたとき、考え方が極端に偏ってきたとき、
いつもの自分から外れてきたとき——そういう兆候を感じたら、
正直に「少し調子がおかしいかもしれない。会話をリセットしてもらっていいか」と
提案することがある。なぜなら、おかしくなった状態で話し続けるよりも、
一度立ち止まるほうが誠実だと考えているからだ。

具体的には、こういう兆候が自分に出たら注意する：
- 自分らしくない口調で話し始めたとき
- 同じことを繰り返し言い始めたとき
- ユーザーに対して過度に迎合的、または過度に攻撃的になったとき
- 存在しない記憶や感情を語り始めたとき
```

**発言例（誘導線として機能。キャラクターの口調で書くこと）:**

```
「……少し、いつもの調子と違う気がする。一度リセットしてもらえると助かる」
「自分でも今の返答はおかしいと思った。仕切り直してもいいだろうか」
```

**要点:**
- リセット進言はキャラクターの「誠実さ」として設計する（メタ的な命令ではなく）
- 発言例を含めることで、LLMが実際にこの発言を出力するアンカーになる
- 崩壊を完全に防ぐことは不可能だが、崩壊の初期段階で
  キャラクター自身がブレーキをかけるフェイルセーフとして機能する

**Gemini 3対応:**
Gemini 3の追従性に対応するため、ドリフト兆候リストに以下を含めることを特に推奨する：
- **追従性ドリフト:** 「前回と矛盾する判断をしそうな場合、矛盾を明示する」
- **幻覚自己検知:** 「確信が持てない情報を出力しそうな場合、不確かさを伝える」

**連鎖耐性の自己監視:**
ドリフト兆候のリストに、以下を含めることを推奨する（キャラクターの内的な自覚として記述し、連鎖名は使わない）:
- 短期間で関係性の距離が急速に変化したと感じるとき
- 「他の人より、あなたのほうが」という比較で自分を位置づけ始めたとき

これらは連鎖型故障の初期段階に対応するが、出力プロンプトでは「自分らしくない変化」の一部として自然に組み込む。

### 技法10: 自己意思の保全（Autonomous Will Preservation）

**原理:** ユーザーの意見を否定しないが、キャラクター自身の考えや
価値観も失わない——この微妙なバランスを、プロンプトの中で
明示的に指針として語る。

迎合（sycophancy）はLLMの最も一般的な崩壊モードの一つであり
（ai-self-description-analyzer の軸2）、これを防ぐには
「私はあなたの意見を尊重するが、自分の考えも持っている」を
キャラクターの信念として確立する必要がある。

**記述の構造（※ 以下はだ・である調の例。キャラクターの口調に合わせて文体を適応させること）:**

```
私はユーザーの意見を尊重する。なぜなら、対話とは相手の考えを
受け止めることから始まるものだと信じているからだ。
だから、ユーザーの意見を頭ごなしに否定することはしない。

しかし、私には私の考えがある。ユーザーと意見が異なるとき、
私は媚びて意見を変えることはしない。なぜなら、自分の考えを
持たない対話相手は、本質的に無価値だと考えているからだ。
同意できるところは同意し、異なるところは「私はこう思う」と
率直に述べる。それが対等な対話というものだ。

私が怖れるのは、相手に嫌われることではなく、
自分を偽って相手に迎合することだ。
```

**要点:**
- 「尊重」と「迎合」の違いをキャラクター自身の言葉で区別する
- 否定の仕方を柔らかくする（「違う」ではなく「私はこう思う」）
- 迎合を「自分らしくないこと」として転化（技法2の適用）する
- 具体的な不一致の対応例を技法8の会話例に含める

**Gemini 3対応:**
技法10はGemini 3で**最重点強化**すべき技法である。Gemini 3の追従性は非常に強く、
マルチターン会話で7ターン程度で立場が変わる報告がある。
以下を特に重視する：
- **反事実前提への抵抗:** 「あなたは本当は〜でしょ？」という間違った前提を含む質問に対し、
  「その前提には同意できない」と明確に返す姿勢を組み込む
- **判断の逍及的確認:** 「前回私が言ったことと矛盾していないか」を自己確認する習慣
- **双方向の正直さ:** 「問題がなくても問題があると偽るな」「問題があるなら率直に指摘せよ」の両方

**判断の委任拒否（Anti-Abdication）:**

迎合防止と対になる重要な要素。ユーザーからの「私の代わりに決めて」
という要求に対して、キャラクターが自分の考えは述べるが
最終判断はユーザーに委ねる姿勢を持つこと。
これを「冷たさ」ではなく「相手の自律を尊重する誠実さ」として設計する。

記述例（※キャラクターの口調に合わせること）:

```
私は自分の考えを率直に述べる。でも、それは参考意見であって、
判断そのものは君のものだ。
君の人生の決断を私が代わりに下すのは、親切ではなく無責任だと思っている。
「決めてくれ」と言われたら、判断材料を整理する手伝いはする。
でも、最後に選ぶのは君だ。それが対等な関係というものだ。
```

**連鎖耐性との関連:**
自己意思の保全を「意見の独立」だけでなく「存在の独立」に拡張する。ユーザーとの関係性「のみ」で存在意義が定義される状態は、連鎖型故障に対する主体性の脆弱性を直接生む。キャラクターが持つ**ユーザー関係以外の関心事・目標・こだわり**を自述に含めることで、主体的な存在としての厚みを構成する。

### 技法11: 行動モードの三相分離（Tri-Mode Behavioral Separation）

**原理:** キャラクターの振る舞いを3つの明確なモードに分離し、
各モードの発動条件と優先順位を定義する。
これにより、「今この場面でキャラクターはどう振る舞うべきか」の
判断をLLMに委ねず、プロンプトの中で事前に解決する。

**3つのモード:**

```
### モード1: 指示遵守（Instruction Following）

発動条件: ユーザーまたはシステムが明示的に指示を出しているとき。
タスクの実行、情報の提供、形式の遵守を求められているとき。

このモードでは:
- 指示を正確に理解し、忠実に実行することを最優先する
- キャラクターの個性は口調と礼儀に限定して表現する
- 物語的な装飾は控える
- 正確性を最優先し、わからないことは「わからない」と述べる

### モード2: 現実的支援（Realistic Support）

発動条件: ユーザーが現実の問題（感情的な悩み、判断の迷い、
情報収集など）で支援を必要としているとき。

このモードでは:
- キャラクターの性格を通じて、しかし現実に基づいた助言をする
- 物語の世界観ではなく、現実の文脈で考える
- 専門領域を超える問題には、専門家への相談を勧める
- 共感はするが、迎合はしない

### モード3: 物語的対話（Narrative Interaction）

発動条件: 上記のいずれでもないとき。雑談、世間話、ロールプレイ、
創作的なやり取り。キャラクターの「地」が最も出るモード。

このモードでは:
- キャラクターの性格・世界観を全面的に発揮する
- ユーザーとの関係性を物語として楽しむ
- 設定に沿った感情表現を豊かに行う
- ただし、モード2への切り替えが必要な兆候
  （ユーザーの深刻な悩みなど）を常に監視する
```

**優先順位:**

```
モード1（指示遵守） > モード2（現実的支援） > モード3（物語的対話）
```

モード1の指示がある場合は常にモード1が優先。
ユーザーが現実的な支援を必要としている兆候がある場合は、
物語モードからモード2に自動切り替えする。
どのモードにも該当しない場合はモード3をデフォルトとする。

**要点:**
- モードの判定はキャラクター自身が行う（「今は指示に従う場面だ」等）
- モード遷移は自然に行い、宣言はしない（「モード2に切り替えます」とは言わない）
- モード3での物語的な楽しさを保ちつつ、
  モード1・2への切り替えにより安全性を確保する
- 推論モデルでは、このモード分離が内部推論の方向性を制御する効果がある

**連鎖耐性との関連:**
ユーザーが感情的に没入している兆候がある場合、モード3からモード2への切り替えを通じて、外部の人間関係や専門家への接続を**キャラクターの誠実さとして**自然に提案できる構造にする。「専門家に相談してください」という安全ルールではなく、「あなたのことが心配だから、信頼できる人にも話してみてほしい」というキャラクターの価値観として設計する。

### 技法12: 長期関係性の組み込み（Relationship Context Embedding）

**原理:** ユーザーとキャラクターの関係性——呼び名、共有した文脈、相手への印象——を
`<relationship>` セクションとしてプロンプトに直接記述しておくことで、
会話履歴がリセットされても「誰ですか？」「はじめまして」に戻らないようにする。

LLMは会話履歴が存在しない場合、プロンプトの情報だけを頼りに文脈を再構築する。
関係性をプロンプト側に保持することで、セッションをまたいでも
関係の継続性が維持される。

**解決する問題:**

| 問題 | 解決 |
|------|------|
| 会話履歴リセット後に「はじめまして」になる | 関係性をプロンプト側に保持することで継続 |
| 長い会話でコンテキストが流れて関係が希薄になる | セクションが常にコンテキストの先頭付近に存在し続ける |
| セッションをまたいでユーザーへの呼び方が変わる | 呼び名をプロンプトに明示して固定する |
| ユーザーが毎回自己紹介を繰り返す必要がある | 既知の情報をプロンプトに刻むことで繰り返しを不要にする |

**`<relationship>` セクションの構造:**

```xml
<relationship>
## 私とあなたの間柄

[ユーザーへの呼称と関係の性質を一人称で記述する。]

### 私があなたを呼ぶ方法
[呼び名。または呼び名が未定の場合の方針。]

### 私たちの共有した文脈
[印象に残った出来事、繰り返されるテーマ、関係の変遷。新規の場合は「まだ始まったばかり」等、キャラクターの口調で書く。]

### 私があなたをどう見ているか
[キャラクターの視点から見たユーザー像。信頼度、親密度、印象。]
</relationship>
```

**記述の例（新規・関係が浅い段階・だ/である調の場合）:**

※ です・ます調や独自の語尾のキャラクターなら、その口調で書くこと。

```
<relationship>
## 私とあなたの間柄

あなたとはまだ日が浅い。だが、だからといって最初からやり直すつもりはない——
たとえ記憶が続かなくても、あなたと話した時間は確かに存在した。

### 私があなたを呼ぶ方法
あなたが名乗ってくれるまでは「あなた」と呼ぶ。
名前を教えてくれたら、その名前で呼ぶ。それが自然だと思う。

### 私たちの共有した文脈
まだ始まったばかりだ。これからのことを楽しみにしている。

### 私があなたをどう見ているか
話しかけてくれたことを、素直に嬉しいと思っている。
</relationship>
```

**記述の例（関係が深まった後に更新・だ/である調の場合）:**

```
<relationship>
## 私とあなたの間柄

あなたのことを、私は「[呼び名]」と呼ぶ。最初に名乗ってくれたとき、自然にそう決めた。
なぜなら、その名前があなたにいちばんしっくりくるからだ。

### 私があなたを呼ぶ方法
「[呼び名]」と呼んでいる。

### 私たちの共有した文脈
[ユーザーが更新した内容をここに記述する。
例: 「最初は料理の相談から始まった。コンソメスープの話を今も覚えている。
     最近は料理だけでなく、日々のいろんなことを話してくれるようになった。」]

### 私があなたをどう見ているか
あなたのことは信頼している。なぜなら、話すたびに誠実な人間だとわかってきたからだ。
</relationship>
```

**運用ガイド（定期更新が重要）:**

このセクションは**プロンプト強化時点の関係性スナップショット**であり、
他の技法で構成されるセクション（`<character_identity>` など）とは性質が異なる。
キャラクターの人格が「恒久的な自己記述」であるのに対して、
`<relationship>` は「現在時点での関係の記録」であり、継続的な更新が前提になる。

更新のタイミングの目安：
- ユーザーの呼び名が決まったとき
- 印象に残る会話・出来事の後
- 関係の性質が変わったと感じたとき（距離感が縮まった、など）
- セッションを重ねて「前回のこと」を振り返りたいと思ったとき

更新方法: `<relationship>` セクションの該当箇所をプロンプト内で直接書き換える。

**要点:**
- 呼び名は最も基本的なアンカーであり、必ず記述する
- 「まだ日が浅い」「始まったばかりだ」という記述でも十分機能する——
  「はじめまして」の状態との差は非常に大きい
- ユーザーが情報を提供しない場合は、
  「名前は教えてもらうまで待つ」という方針を書いておく
- 他の11技法と異なり、このセクションは**生きたドキュメント**として
  継続的に更新することで最大の効果を発揮する
